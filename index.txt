系统功能实现与系统功能集成分类
    面向对象（OO）的业务功能实现
        以对象为核心
        通过对象事务的基本特征之间的消息交互实现业务逻辑（就是调对象的方法）
    面向服务（SOA）的业务功能实现
        以服务为核心
        将业务功能封装为服务并通过接口提供（将业务功能封装为标准的服务单元，通过标准化接口对外发布）
    面向对象系统的功能集成 (代码上方法怎么个调用顺序）
        集成单元：对象
        常见方式：工作流 + 对象调用
    面向服务系统的功能集成（系统服务上怎么个调用顺序
        集成单元：服务
        常见方式：服务组合 / 服务编排
    面向对象是代码层面上，面向服务上是系统层面上

事务的基本特征（ACID）
    原子性（Atomicity）
        事务中的操作要么全部成功，要么全部失败并回滚
    一致性（Consistency）
        事务执行前后，数据始终满足完整性约束
    隔离性（Isolation）
        并发事务之间相互隔离，互不干扰
    持久性（Durability）
        事务一旦提交，其结果将永久保存
MVC与MVP的特征
    MVP（三点）
        1️⃣ 分离
            View 不直接访问 Model，必须通过 Presenter
        2️⃣ 主控
            业务逻辑主要在 Presenter
        3️⃣ 好测
            Presenter 可独立测试，工程性好
    MVC（三点）
        1️⃣ 直连
            View 可以直接访问 Model
        2️⃣ 分发
            Controller 负责接收请求并分发调度
        3️⃣ 难测
            View 与 Model 耦合，测试依赖界面
EJB 规范的两种事务控制方式
    容器管理事务（CMT）
        事务由 EJB 容器自动管理
        方法级事务（默认一个方法一个事务）
        系统异常时容器自动回滚
        优点：简单、常用、开发效率高
    Bean 管理事务（BMT）
        事务由 开发者在代码中手动控制
        通过 JTA 定义事务边界
        可跨方法
        优点：灵活，缺点：复杂

单体服务和微服务对比
    单体：复杂/不灵活/不可按需拓展
    微服务：拆分/独立/灵活/按需扩展
    微服务的挑战：运维/性能/数据一致性
        部署和运维复杂 → 服务数量多，需要独立部署/监控
        性能损耗 → 远程调用带来网络/序列化开销
        数据一致性难 → 多服务独立存储，需要分布式事务或最终一致性机制
    | 对比维度 | 单体 | 微服务 |
    | --- | --- | --- |
    | 复杂度 | 系统太大 → 复杂 | 拆分服务 → 降低系统复杂度 |
    | 可维护性 | 改一处影响全局 → 不灵活 | 服务独立 → 修改/升级不影响其他服务 |
    | 灵活性 | 扩容整体 → 不可按需扩展 | 服务独立开发部署 → 灵活调整 |
    | 扩展性 | 整体扩容，资源利用率低 | 可按服务独立扩展，提高利用率 |

REST架构风格特征

事件架构风格
    概念：事件系统架构是一种事件驱动架构，核心由三部分组成：
        事件源：触发事件的主体
            例：用户下单、商家发布订单等。
        事件管理器：接收事件并进行调度分发
            根据事件类型，把事件分派给合适的处理器。
        事件处理器：具体执行事件对应的业务处理
            例：配送员接单、更新配送状态、完成配送等
消息队列模式
    点对点模式
        消息发送者生产消息发送到 queue 中，然后消息接收者从 queue 中取出并且消费消息。消息被消费以后，queue 中不再有存储，所以消息接收者不可能消费到已经被消费的消息。
    发布/订阅模式
        发布者将消息发送到 Topic,系统将这些消息传递给多个订阅者。
前后端分离架构
    前后端分离是指前端负责展示与交互，后端负责业务逻辑与数据处理，通过 API 接口通信实现解耦。其优点包括支持多终端访问、提高开发效率、增强可维护性和可扩展性。

两阶段提交协议（Two Phase Commitment Protocol，2PC）
    准备阶段（投票阶段）：协调者询问各参与者“能不能提交？”，参与者执行本地事务但不提交，并返回 YES/NO。
    提交/回滚阶段：如果全部 YES → 协调者通知全部提交；只要有 NO 或超时 → 通知全部回滚。
    2PC 全流程两阶段通信，但只有第一阶段有一次 YES/NO 投票。
2PC协议是否支持分布式系统
    结论：不支持
    原因：
        2PC 为强一致协议，而微服务更强调高可用与扩展性，两者目标不一致。
        NoSQL 不支持 2PC
            2PC 需要数据库支持：
                全局事务
                XA 协议
                锁机制
            而：
                大多数 NoSQL 不支持跨节点事务
                所以直接不能用。
        2PC 适用于“一个整体事务”
            这句话其实是最抽象的。
            翻译成人话是：
                2PC 适合一个业务系统内部多个数据库节点统一提交，不适合多个独立微服务之间协同
            微服务的特点：
                每个服务独立部署
                每个服务自治
                不共享数据库
            而 2PC 要：
                统一协调者
                全局锁
                全局事务管理
            这和微服务设计理念冲突。
本地事务发布事件
    核心思路：先别急着发 MQ消息，先把“要发的消息”存进数据库里。
    第一步：下单（一个事务里做两件事）
        订单服务开启一个数据库事务：
        插入订单表：订单=已创建
        插入 EVENT 表：事件=“创建订单”，状态=未发送
        然后一次性提交事务。
        这样保证：
        订单写成功 ⇔ EVENT 也写成功
        订单写失败 ⇔ EVENT 也写失败
        这就解决了“原子性”。
    第二步：后台专人负责“补发消息”
        然后有个后台线程/服务不断扫描 EVENT 表：
        找到状态=未发送的事件
        发送到 MQ
        发送成功后把 EVENT 状态改成已发送

消息中间件的作用及优势
    作用：⭐削峰填谷（这词必须记）（核心原理）
        意思是：
        流量突然变大 → 先堆在 Kafka 里
        后面慢慢处理 → 系统就不会被瞬间压垮
    3 个优势
        ✅（1）异步
        生产者先把日志扔进 Kafka，不用等后端算完。
        ✅（2）解耦
        采集日志的系统和处理日志的系统分开，互不影响。
        ✅（3）缓冲/削峰填谷
        Kafka 作为缓冲区，流量暴增先排队，不会直接冲击数据库和计算服务。

数据脱敏最小知识点（5种）
    口诀：替混两变删
    1）替换（Replace）
        用随机值/字典值替换原值
        关键词：替换、随机、字典
    2）混排（Shuffle）
        同字段在记录间交换，打乱对应关系
        关键词：交换、打乱、乱序
    3）时空变异（Time/Date Shift）
        日期时间前后平移，保留趋势但不可识别
        关键词：平移、日期、趋势
    4）数值变异（Noise）
        数值加减随机扰动（±百分比），保留统计规律
        关键词：扰动、噪声、±%、趋势
    5）删除（Delete/Null）
        直接删除或置空敏感字段
        关键词：删除、置空、彻底

混合云的优势
    公有云：便宜、弹性强，但安全合规弱
    私有云：安全强，但成本高、扩展差
    混合云：两边优点都要
    4个优势
        1）安全合规（关键数据放私有云）
        运维监控、故障信息、隐私数据等敏感内容放本地，满足安全和合规要求。
        2）弹性扩展（高峰用公有云）
        业务访问高峰、数据分析需要算力时，借助公有云弹性资源，避免系统被流量打爆。
        3）成本与投资保护（渐进式上云）
        保留原有机房资产，不用一次性重建，按需扩容，降低总体成本。
        4）高可用与灾备（容灾切换）
        私有云故障可切到公有云，公有云异常也可回落本地，提高业务连续性。
