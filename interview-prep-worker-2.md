## 1. React 渲染机制是什么？组件为什么会重新渲染？

**一行总结：**
React 重新渲染由 props、state、父级或 context 变化触发，优化重点是减少无效计算和高成本更新。

**面试回答版：**
React 的核心是“状态变化驱动 UI 重新计算”。组件重新渲染通常有 4 类原因：`props` 变化、`state` 变化、父组件重新渲染、`context` 变化。重新渲染不等于真实 DOM 一定重建，React 会先生成新的虚拟 DOM，再通过 Diff 找出最小变更提交到真实 DOM。实际优化重点不是“避免所有 render”，而是减少无效计算、无效 Diff 和高成本子树更新。

**追问点：**

- `React.memo`、`useMemo`、`useCallback` 分别解决什么问题？
- 父组件 render 了，子组件为什么不一定真的有性能问题？
- key 为什么会影响 Diff 结果？

## 2. useEffect 和 useLayoutEffect 的区别是什么？

**一行总结：**
默认优先 useEffect，只有依赖同步布局结果时才使用 useLayoutEffect。

**面试回答版：**
`useEffect` 在浏览器绘制后异步执行，适合请求、订阅、日志上报这类不阻塞页面展示的副作用；`useLayoutEffect` 在 DOM 更新后、浏览器绘制前同步执行，适合读写布局、计算尺寸、修正闪动。如果在大组件里滥用 `useLayoutEffect`，会阻塞渲染，影响首屏性能。我的原则是默认优先 `useEffect`，只有确实依赖同步布局结果时才用 `useLayoutEffect`。

**追问点：**

- 为什么 `useLayoutEffect` 会更容易造成卡顿？
- effect 清理函数的执行时机是什么？
- React StrictMode 下 effect 为什么会执行两次？

## 3. React Hooks 常见闭包陷阱怎么处理？

**一行总结：**
Hooks 闭包问题本质是拿到旧值，常用解法是补全依赖、useRef 保存最新值和函数式更新。

**面试回答版：**
闭包问题本质是回调拿到的是创建当时的旧变量。常见场景有 `setInterval`、事件监听、异步回调、节流防抖函数。处理方式一般有 3 种：一是把依赖写全，让 effect/回调重新创建；二是用 `useRef` 保存最新值；三是更新 state 时优先用函数式写法，避免依赖旧值。我在线上更关注“高频事件 + 旧闭包”导致的数据不同步问题。

**追问点：**

- 为什么 ESLint Hooks 规则建议补全依赖？
- 什么场景更适合 `useRef`，什么场景应该重建回调？
- 防抖/节流函数为什么更容易踩闭包坑？

## 4. React 中 key 的作用是什么？为什么不能随便用 index？

**一行总结：**
key 决定同层节点的身份识别和状态复用，动态列表里不要随便用 index。

**面试回答版：**
key 是 React 在同层级节点中识别“谁是谁”的稳定标识，直接影响 Diff 时节点复用和状态保留。使用稳定唯一 key，React 才能正确判断节点是新增、删除还是移动。用 index 作为 key 在纯静态列表问题不大，但在插入、删除、排序场景下会导致状态错位、重复渲染，表单输入和动画列表里尤其明显。

**追问点：**

- 什么叫状态错位？
- 为什么列表 reorder 时 index key 问题最明显？
- 有哪些场景可以接受 index key？

## 5. TypeScript 里 any、unknown、never 的区别是什么？

**一行总结：**
any 放弃检查、unknown 先收窄再使用、never 用于不可能分支和穷尽校验。

**面试回答版：**
`any` 是放弃类型检查，能赋值给任何类型，也能接收任何类型，风险最高；`unknown` 表示“我现在不知道类型”，使用前必须先做类型收窄，更安全；`never` 表示“不可能出现的值”，常用于兜底分支校验、抛异常函数、或联合类型穷尽检查。我的习惯是：边界输入先用 `unknown`，不要轻易用 `any`，复杂业务分支用 `never` 做穷尽保护。

**追问点：**

- 联合类型怎么做穷尽检查？
- 为什么说 `unknown` 比 `any` 更适合接口入参？
- `never` 在条件类型里有什么作用？

**一行总结：**

**一行总结：**

## 12. HTTP 缓存和浏览器缓存策略怎么理解？

**一行总结：**
缓存策略本质是用强缓存提速、用协商缓存保鲜，并靠 hash 文件名控制静态资源更新。

**面试回答版：**
我一般分强缓存和协商缓存。强缓存靠 `Cache-Control` / `Expires`，命中后浏览器直接用本地资源；协商缓存靠 `ETag` / `Last-Modified`，浏览器会带条件请求问服务端资源有没有变化。工程上常见策略是：静态资源文件名加 hash，配合长期强缓存；HTML 入口文件短缓存或不缓存，确保能拉到最新资源。这样能兼顾加载速度和版本更新。

**追问点：**

- `no-cache` 和 `no-store` 的区别？
- 为什么 HTML 不适合长缓存？
- ETag 和 Last-Modified 各自局限是什么？

## 13. 说一下浏览器事件循环（Event Loop）

**一行总结：**
Event Loop 的关键是同步任务先执行，调用栈清空后优先跑微任务，再进入下一轮宏任务。

**面试回答版：**
JavaScript 主线程是单线程执行模型。同步任务先进入调用栈执行；异步任务完成后，其回调会进入任务队列等待。事件循环会在调用栈清空后，先取微任务，再进入下一轮宏任务。常见微任务有 `Promise.then`、`queueMicrotask`，宏任务有 `setTimeout`、I/O、UI 事件。理解事件循环对处理高频更新、避免阻塞主线程、安排批量更新很重要。

**追问点：**

- 为什么微任务会先于下一个宏任务执行？
- `requestAnimationFrame` 适合解决什么问题？
- 高频更新场景里怎么利用事件循环思想降压？

## 16. JWT 和 Session 的区别是什么？

**一行总结：**
Session 把状态放服务端，JWT 把声明放 token 里，选型关键看架构约束和失效管理。

**面试回答版：**
Session 方案是服务端保存会话状态，客户端只保存 session id；JWT 方案是服务端签发 token，用户信息和声明放在 token 里，由客户端携带。JWT 更适合无状态、分布式接口鉴权，但也更依赖过期时间控制、签名校验和刷新机制。我的理解是 JWT 不是更安全，只是更适合某些架构，关键还是怎么管理失效、续期和权限边界。

**追问点：**

- JWT 为什么不好主动失效？

- 什么场景更适合 Session？

## 18. TCP 粘包 / 拆包是什么？怎么处理？

**一行总结：**
TCP 没有消息边界，解决粘包拆包要靠应用层协议定义和缓冲区切帧。

**面试回答版：**
TCP 是字节流协议，没有消息边界，所以发送端两次写入的数据，接收端可能一次收到，也可能拆成多次收到，这就是粘包和拆包。处理关键不是“让 TCP 不粘包”，而是在应用层定义消息边界。常见做法有固定长度、分隔符、消息头 + 消息体长度。我在设备接入场景里会先做缓冲区累积，再按协议规则切帧解析，保证报文完整性。

**追问点：**

- 为什么 UDP 没有这个典型问题？
- 半包场景如何缓存剩余数据？
- 如果协议里既有长度又有校验位，解析顺序怎么设计？

**一行总结：**
LangChain 更适合做大模型应用层编排，真正决定效果的是领域约束、数据质量和结果校验。

**面试回答版：**
LangChain 更适合做大模型应用层编排，比如 prompt 模板、上下文组装、工具调用、简单工作流串联，不适合把它当成“万能 AI 框架”。结合我的简历，比较贴切的场景是自然语言转报表配置：先把用户问题结构化，再做字段映射、图表推荐和参数补全。我的理解是它更像应用胶水层，真正决定效果的还是领域约束、数据质量和结果校验。

**追问点：**

- 为什么自然语言生成报表不能只靠 prompt？
- 怎么降低 AI 输出不稳定的问题？
- 什么场景下你不会选 LangChain？

## 常见 JS / 算法题速记（高频手写 + 讲法）

### 1. debounce

- **一句话思路：** 每次触发先清旧定时器，再重开新定时器，只让最后一次执行。
- **面试表达：** debounce 适合输入框搜索、窗口 resize 这类“连续触发但只关心最后结果”的场景，关键实现点是保存 timer、透传 `this` 和参数。
- **易错点：** timer 写到函数内部、丢失 `this`、不支持立即执行版。

### 2. throttle

- **一句话思路：** 在固定时间窗口内只允许执行一次，常见做法是时间戳版或定时器版。
- **面试表达：** throttle 更适合滚动、拖拽、鼠标移动这类高频事件，目的是限制执行频率而不是只保留最后一次。
- **易错点：** 不清楚首执行/尾执行策略、遗漏最后一次补执行。

### 3. arrayToTree

- **一句话思路：** 先用 Map 按 id 建索引，再二次遍历把节点挂到父节点 children 上。
- **面试表达：** 这题重点不是背代码，而是说明为什么要把 O(n^2) 的父子查找降成 O(n)。
- **易错点：** 根节点判定不清、children 未初始化、原数据被意外污染。

### 4. 深拷贝

- **一句话思路：** 基础版递归处理对象和数组，进阶版要考虑循环引用、Date、RegExp、Map、Set。
- **面试表达：** 如果面试官只问常规实现，我会先给可运行版本，再主动补一句“生产里更建议明确数据边界，避免无脑深拷贝”。
- **易错点：** `null` 判错、循环引用爆栈、特殊对象类型丢失。

### 5. Promise.all

- **一句话思路：** 并发收集多个 Promise 结果，全部成功才 resolve，有一个失败就 reject。
- **面试表达：** 这题关键是结果顺序要和输入顺序一致，而且要正确处理非 Promise 值和空数组。
- **易错点：** 用完成顺序覆盖输入顺序、计数结束条件错误、漏掉 reject 分支。

### 6. LRU Cache

- **一句话思路：** 用 Map 的插入顺序模拟“最近使用”，访问后删掉再重插，超容量时删除最老 key。
- **面试表达：** 前端面试通常 Map 版就够了；如果追问更高性能实现，再补“哈希表 + 双向链表”是标准解。
- **易错点：** `get` 时没有刷新顺序、容量满时删错 key。

### 7. 事件循环题

- **一句话思路：** 先执行同步代码，再清空微任务队列，最后进入下一轮宏任务。
- **面试表达：** 做这类题我会先把 `console.log` 执行时机分成同步、微任务、宏任务三层，再给最终输出顺序。
- **易错点：** 混淆 `Promise.then`、`setTimeout`、`async/await` 的执行时机。

### 8. 继承 / 原型链

- **一句话思路：** 先讲原型链查找规则，再说组合继承或 ES6 `class extends` 本质还是基于原型机制。
- **面试表达：** 这类题不要只背定义，最好顺手画出“实例 -> 原型对象 -> 构造函数原型 -> Object.prototype”的链路。
- **易错点：** `__proto__`、`prototype`、`constructor` 三者关系讲混。

## 偏考试 / 面试高分表达（直接套用）

### 1. 回答原理题的高分模板

- **起手句：** “这个问题我一般会先讲定义，再讲触发条件/执行时机，最后补项目里的落地点。”
- **展开句：** “如果只讲概念会比较空，所以我会结合我在 BI / 大屏项目里的实际处理方式一起回答。”
- **收束句：** “所以我真正关注的不是背术语，而是这个原理会怎么影响线上性能、稳定性和代码边界。”

### 2. 回答性能题的高分模板

- **起手句：** “性能题我不会一上来堆优化手段，我一般先定位瓶颈，再决定改哪一层。”
- **展开句：** “通常我会先区分是数据获取、数据处理、React 渲染还是图表实例更新更重，然后只改最主要的那段。”
- **收束句：** “这样回答的重点是体现方法论，而不是背一串零散优化点。”

### 3. 回答项目题的高分模板

- **起手句：** “这个项目里我尽量区分团队参与和我主导的模块，重点讲我真正做决策的部分。”
- **展开句：** “我会按背景、问题、我的方案、取舍和结果这五步讲，避免只报功能列表。”
- **收束句：** “如果量化数据不够完整，我会明确说‘明显改善’而不是编百分比。”

### 4. 回答全栈 / AI 题的高分模板

- **起手句：** “这块我会如实定义能力边界，不会把工程实践包装成算法研究。”
- **展开句：** “比如全栈我强调的是业务链路打通能力，AI 我强调的是应用落地和可控性，而不是模型训练本身。”
- **收束句：** “这样更容易建立可信度，面试官通常也更愿意继续深挖。”

### 5. 高频加分表达替换

- 不说：**“我做了很多优化。”**
- 改说：**“我先定位瓶颈，再分别从数据处理、渲染边界和更新时机三层做优化。”**
- 不说：**“这个项目我都参与了。”**
- 改说：**“项目是团队协作完成的，但图表渲染体系 / TCP 解析链路这两块我是主导设计和落地的。”**
- 不说：**“这个方案比较好。”**
- 改说：**“我当时选这个方案，是因为它在当前数据规模、交付时间和维护成本下更平衡。”**
- 不说：**“我也懂一点 AI。”**
- 改说：**“我现在的 AI 经验主要是 LangChain.js 这一类应用层编排和业务接入，定位是工程探索，不会夸大成算法能力。”**
