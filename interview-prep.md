## 三、最可能被深挖的项目题

> 原则：项目题优先级高于八股文。先把这部分答顺，再补基础题。

### 1. BI 报表分析平台（最高概率）

#### 题 1：你在 BI 报表平台里做的最核心工作是什么？

**60 秒回答**

我在这个项目里主要负责复杂查看链路的需求迭代、线上问题处理，以及图表渲染体系的建设。相对更核心的一块，是基于 Storybook 搭建图表渲染器组件体系，并把数据处理到图表渲染这条链路做成管道式解耦，最后抽离成 npm 包提升复用性。除此之外，我也做了复杂报表场景下的加载和渲染性能优化、移动端适配、埋点建设、国际化，以及基于 LangChain.js 的自然语言生成报表探索。

**高频追问点**

- 为什么要做图表渲染器组件体系？
- 管道模式具体怎么拆？
- npm 包抽离后怎么管理版本和兼容性？
- 复杂报表性能问题主要卡在哪？
- LangChain 方案最后做到什么程度？

**追问参考回答**

- 图表类型和配置组合太多，如果每个页面都单独拼装，维护成本会很高，所以先统一渲染入口，再拆数据处理和渲染职责。
- 管道模式可以按“原始数据 -> 结构化数据 -> 图表配置 -> 渲染输出”分层，每层只关心输入输出。
- 抽 npm 包是为了统一升级和复用，尤其适合 Monorepo 下的多业务场景。
- 性能优化重点通常在首屏加载、复杂联动和图表重复计算，不会一上来盲目优化，而是先定位瓶颈。
- LangChain 相关工作属于探索型验证，主要做自然语言到图表配置的基础映射，不会包装成成熟 AI 平台。

#### 题 2：你说做了报表性能优化，具体怎么定位和解决？

**60 秒回答**

我一般先区分是数据获取慢、数据处理慢，还是渲染慢。对 BI 报表这类系统，常见瓶颈是字段多、图表多、联动复杂，导致首屏加载慢和渲染卡顿。我会通过性能面板、埋点和实际场景复现去定位瓶颈，再分别处理，比如减少不必要渲染、避免重复数据转换、拆分重计算链路、按需渲染非首屏内容，以及优化图表组件的更新时机。核心思路不是堆优化手段，而是先定位最重的那一段。

**追问参考回答**

- 如果 React 层更新频繁，就优先看状态拆分、依赖控制和组件边界。
- 如果图表实例更新特别重，就看 setOption 时机、数据量和增量更新方式。
- 如果没有严格沉淀过量化数据，回答里就说“明显改善响应和流畅度”，不要编数字。

#### 题 3：你在报表平台里做的 AI 能力探索是什么？

**60 秒回答**

我做的是基于 LangChain.js 的自然语言生成报表能力探索，目标是把用户的自然语言问题映射成字段、筛选条件和图表类型，再生成基础配置。这个能力我会定义成探索型方案，不会夸大成完整 AI 产品。它更像是在验证“自然语言 -> 报表配置”这条链路是不是能跑通，以及在哪些场景下有实际价值。

**追问参考回答**

- LangChain 更适合做多步骤编排，比如提示词组织、字段映射、上下文拼接和结构化输出。
- 这个场景真正难的不是生成一句答案，而是稳定、可控地映射到受约束的字段和图表配置。
- 如果没有正式上线，就如实说是探索和验证。

## 六、高频技术八股文（按命中概率排序）

### 1. React 渲染机制是什么？组件为什么会重新渲染？

**标准答案**

React 的核心是状态变化驱动 UI 重新计算。组件重新渲染通常有 4 类原因：props 变化、state 变化、父组件重新渲染、context 变化。重新渲染不等于真实 DOM 一定重建，React 会先生成新的虚拟 DOM，再通过 Diff 找出最小变更提交到真实 DOM。实际优化重点不是避免所有 render，而是减少无效计算、无效 Diff 和高成本子树更新。

**追问点**

- React.memo、useMemo、useCallback 分别解决什么问题？
- 父组件 render 了，子组件为什么不一定真的有性能问题？
- key 为什么会影响 Diff 结果？

### 2. useEffect 和 useLayoutEffect 的区别是什么？

**标准答案**

useEffect 在浏览器绘制后异步执行，适合请求、订阅、日志上报这类不阻塞页面展示的副作用；useLayoutEffect 在 DOM 更新后、浏览器绘制前同步执行，适合读写布局、计算尺寸、修正闪动。我的原则是默认优先 useEffect，只有确实依赖同步布局结果时才用 useLayoutEffect。

### 3. React Hooks 常见闭包陷阱怎么处理？

**标准答案**

闭包问题本质是回调拿到的是创建当时的旧变量。常见场景有 setInterval、事件监听、异步回调、节流防抖函数。处理方式一般有 3 种：补全依赖、用 useRef 保存最新值、更新 state 时优先用函数式写法。

### 4. React 中 key 的作用是什么？为什么不能随便用 index？

**标准答案**

key 是 React 在同层级节点中识别“谁是谁”的稳定标识，直接影响 Diff 时节点复用和状态保留。使用 index 作为 key 在插入、删除、排序场景下容易导致状态错位、重复渲染，表单输入和动画列表里尤其明显。

### 5. TypeScript 里 any、unknown、never 的区别是什么？

**标准答案**

any 是放弃类型检查；unknown 表示暂时不知道类型，使用前必须先收窄；never 表示不可能出现的值，常用于穷尽检查和兜底分支。我习惯边界输入优先用 unknown，少用 any，用 never 做复杂业务分支保护。

### 6. TypeScript 泛型的价值是什么？你在项目里怎么用？

**标准答案**

泛型的价值是把类型也当参数传入，让同一套逻辑适配不同数据结构，同时保留类型推导能力。实际项目里适合用在接口响应封装、表格列配置、表单 Schema、通用 hooks、组件 props 约束上。

### 7. 说一下你理解的前端性能优化，怎么结合项目落地？

**标准答案**

我一般把性能拆成加载性能、渲染性能、交互性能、长时间运行稳定性四块。加载看资源体积、懒加载、缓存、代码拆分；渲染看减少无效 render、虚拟列表、缓存和批量更新；交互看防抖节流和主线程长任务；稳定性看内存泄漏、订阅清理和异常监控。

### 8. 高频图表 / 大屏场景的性能优化思路有哪些？

**标准答案**

重点看 5 件事：控制数据量、增量更新、降低 React 层参与度、拆分静态区和动态区、清理定时器 / WebSocket / 图表实例，避免长时间运行后内存上涨。对图表库更倾向复用实例和局部更新，而不是频繁销毁重建。

### 9. WebSocket 和 HTTP 轮询相比有什么区别？

**标准答案**

WebSocket 更适合服务端主动推送、低延迟、高频更新场景；HTTP 轮询实现简单，但高频场景下请求开销更大、时效性更差。实际项目里要重点处理连接状态、断线重连、消息节流 / 批处理和资源释放。

### 10. 什么是前端内存泄漏？常见场景有哪些？

**标准答案**

内存泄漏是对象本来应该被回收，但由于还存在引用而长期驻留，导致内存持续上涨。常见场景包括未清理的定时器、事件监听器、WebSocket/订阅、闭包持有大对象、图表/地图实例未销毁、缓存无限增长。

### 11. 说一下浏览器从输入 URL 到页面渲染的过程

**标准答案**

先做 URL 解析、缓存检查和 DNS 解析，然后建立 TCP 连接，HTTPS 还会走 TLS 握手；拿到 HTML 后解析生成 DOM，解析 CSS 生成 CSSOM，再合并成 Render Tree，之后布局和绘制。同步脚本可能阻塞解析，所以会用 defer、async、预加载和代码拆分降低阻塞。

### 12. HTTP 缓存和浏览器缓存策略怎么理解？

**标准答案**

我一般分强缓存和协商缓存。强缓存靠 Cache-Control / Expires；协商缓存靠 ETag / Last-Modified。工程上常见策略是静态资源加 hash 并长缓存，HTML 入口文件短缓存或不缓存。

### 13. 说一下浏览器事件循环（Event Loop）

**标准答案**

同步任务先进调用栈执行；异步任务完成后回调进入任务队列。调用栈清空后，事件循环先取微任务，再进入下一轮宏任务。Promise.then 属于微任务，setTimeout 属于宏任务。理解这一点对高频更新和主线程调度很重要。

### 14. 你做过哪些工程化工作？Monorepo 有什么优缺点？

**标准答案**

我做过 Monorepo、Webpack、npm 包抽离发布、Storybook 组件体系建设。Monorepo 的优点是依赖统一、版本协同、跨包改动方便、公共能力复用成本低；缺点是仓库体积大、构建链路复杂，如果没有缓存和边界治理，反而会拖慢效率。

### 15. Webpack 打包优化常见手段有哪些？

**标准答案**

常见思路包括缩小打包范围、合理代码拆分、开启缓存、提取公共依赖、按需处理资源、开启 Tree Shaking、优化 Source Map 和压缩策略。线上更关注首屏加载和拆包合理性，开发时更关注缓存和增量编译效率。

### 16. JWT 和 Session 的区别是什么？

**标准答案**

Session 是服务端保存会话状态，客户端只存 session id；JWT 是服务端签发 token，由客户端携带。JWT 更适合无状态接口鉴权，但失效控制和续期机制要设计好。JWT 不是更安全，只是更适合某些架构。

### 17. RBAC 权限模型怎么设计？

**标准答案**

RBAC 是用户-角色-权限模型。后台系统里通常会拆成接口权限、菜单 / 路由权限、按钮 / 操作权限。前端负责展示和交互层控制，后端必须做最终鉴权，复杂场景下还要补数据权限。

### 18. TCP 粘包 / 拆包是什么？怎么处理？

**标准答案**

TCP 是字节流协议，没有消息边界，所以发送端多次写入的数据在接收端可能一次收到，也可能分多次收到。处理关键不是让 TCP 不粘包，而是在应用层定义消息边界，比如固定长度、分隔符、消息头 + 长度字段，然后做缓冲区累积和切帧解析。

### 19. Redis 常见使用场景有哪些？

**标准答案**

Redis 常见场景包括缓存热点数据、存登录态/验证码、做计数器、排行榜、轻量队列和限流。在设备平台这类系统里，适合用 Redis 做高频读缓存和临时状态，但要注意缓存一致性和大 key 问题。

### 20. 慢 SQL 通常怎么排查和优化？

**标准答案**

先确认是不是 SQL 本身慢，再看执行计划、索引命中、回表、排序、分页和 join 方式。常见优化方式是补索引、减少 select 字段、避免函数操作索引列、优化分页方案、拆复杂查询、必要时做冗余或预聚合。

### 21. Taro / 多端开发的核心难点是什么？

**标准答案**

核心难点在端差异治理，包括生命周期、样式兼容、宿主能力差异、组件能力不一致和调试链路复杂。我的做法一般是把业务逻辑抽到稳定层，把端差异收敛到适配层。

### 22. LangChain 适合做什么？你在项目里是怎么用的？

**标准答案**

LangChain 更适合做大模型应用层编排，比如 prompt 模板、上下文组装、工具调用和简单工作流串联。结合我的经历，比较贴切的场景是自然语言转报表配置：先把用户问题结构化，再做字段映射、图表推荐和参数补全。

## 七、手写代码题（高概率）

> 目标不是写出最炫实现，而是 10~15 分钟内写出稳定、可解释、边界清晰的版本。

### 1. 手写防抖 debounce

**题目**

实现一个 debounce(fn, delay)，在高频触发时只执行最后一次。

**标准写法**

```js
function debounce(fn, delay) {
  let timer = null;

  return function (...args) {
    const context = this;
    clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(context, args);
    }, delay);
  };
}
```

**面试讲解模板**

核心思路就是维护一个定时器，每次触发都先清掉上一次，再重新计时。这样连续触发时，只有最后一次会真正执行。这里我用 apply 保留 this 和参数，避免在输入框、事件回调场景里丢上下文。

**易错点**

- 忘了保留 this
- 忘了透传参数
- 把 timer 写到函数内部，导致无法复用

### 2. 手写节流 throttle

**标准写法**

```js
function throttle(fn, delay) {
  let lastTime = 0;

  return function (...args) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}
```

**讲解要点**

节流是让函数在一段时间内最多执行一次，适合滚动、拖拽、resize 这类持续触发场景。它和防抖的核心区别，是防抖保最后一次，节流保固定频率。

### 3. 数组转树 arrayToTree

**题目**

把带有 id、parentId 的平铺数组转成树结构。

**标准写法**

```js
function arrayToTree(list) {
  const map = new Map();
  const result = [];

  for (const item of list) {
    map.set(item.id, { ...item, children: [] });
  }

  for (const item of list) {
    const node = map.get(item.id);
    if (item.parentId == null) {
      result.push(node);
    } else {
      const parent = map.get(item.parentId);
      if (parent) {
        parent.children.push(node);
      }
    }
  }

  return result;
}
```

**讲解要点**

这题重点不是递归，而是时间复杂度。先建 map，再二次遍历挂载，复杂度能做到 O(n)。如果用递归每次找 parent，容易退化。

### 4. 实现深拷贝 deepClone（简化版）

**标准写法**

```js
function deepClone(value, cache = new WeakMap()) {
  if (value === null || typeof value !== 'object') return value;
  if (cache.has(value)) return cache.get(value);

  const result = Array.isArray(value) ? [] : {};
  cache.set(value, result);

  for (const key in value) {
    if (Object.prototype.hasOwnProperty.call(value, key)) {
      result[key] = deepClone(value[key], cache);
    }
  }

  return result;
}
```

**讲解要点**

要主动说明：这是简化版，主要处理对象、数组和循环引用；如果要完整支持 Date、RegExp、Map、Set、函数拷贝等，需要继续扩展。面试时能主动讲边界，比硬装“全支持”更稳。

### 5. LRU 缓存

**题目**

实现一个支持 get / put 的 LRU 缓存。

**可背诵版本**

```js
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();
  }

  get(key) {
    if (!this.map.has(key)) return -1;
    const value = this.map.get(key);
    this.map.delete(key);
    this.map.set(key, value);
    return value;
  }

  put(key, value) {
    if (this.map.has(key)) {
      this.map.delete(key);
    }
    this.map.set(key, value);
    if (this.map.size > this.capacity) {
      const oldestKey = this.map.keys().next().value;
      this.map.delete(oldestKey);
    }
  }
}
```

**讲解要点**

如果面试不是算法岗，这个 Map 版就够了。重点讲清楚：Map 保持插入顺序，所以最近访问的 key 删除后重插即可放到最后；超容量时删除最老 key。

### 6. Promise.all

**标准写法**

```js
function promiseAll(promises) {
  return new Promise((resolve, reject) => {
    const result = [];
    let count = 0;

    if (promises.length === 0) {
      resolve([]);
      return;
    }

    promises.forEach((p, index) => {
      Promise.resolve(p)
        .then((value) => {
          result[index] = value;
          count += 1;
          if (count === promises.length) {
            resolve(result);
          }
        })
        .catch(reject);
    });
  });
}
```

**易错点**

- 忘了保持结果顺序
- 忘了处理非 Promise 值
- 忘了空数组边界

### 7. React 手写题：封装 useDebounce

**标准写法**

```jsx
import { useEffect, useState } from 'react';

function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => clearTimeout(timer);
  }, [value, delay]);

  return debouncedValue;
}
```

**讲解要点**

这题面试官通常不是只看你会不会写，而是看你有没有副作用清理意识。一定要主动讲：依赖变化时会清理旧 timer，避免内存泄漏和旧值覆盖。

### 8. React 场景题：如何避免列表页重复请求？

**推荐回答结构**

- 先区分是 StrictMode 导致的开发态重复 effect，还是业务逻辑真的触发了两次。
- 请求层要有幂等或去重能力。
- 组件层要收敛依赖，避免 effect 依赖写错。
- 对于快速切换筛选条件的场景，可以加取消请求或只保留最后一次结果。
