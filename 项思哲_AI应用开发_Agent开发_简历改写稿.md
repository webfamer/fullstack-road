### 企业知识库管理平台 | 全栈开发 / AI 应用开发

2026.04 - 2026.06  
企业内部知识碎片化严重，跨部门资料缺少统一归集渠道，传统文档检索效率低、知识复用成本高。为盘活企业文档、制度、方案、项目资料等知识资产，主导开发企业级知识库管理平台，提供文档管理、AI 问答、全文/向量检索、权限控制、数据统计等能力，并为后续知识图谱与多模态解析扩展预留架构边界。  
技术栈：Python / FastAPI / PostgreSQL / pgvector / Redis / ARQ / MinIO / SQLAlchemy / Alembic / OpenAI-compatible API / React / SSE

- 负责知识库后端核心链路设计与开发，完成知识库、文档、文档分块、会话消息、用户权限等数据模型设计，并通过 Alembic 管理数据库版本演进。
- 设计文档入库流水线：上传文件先进入 MinIO 对象存储，再通过 Redis + ARQ 异步任务完成 PDF / DOCX / TXT 解析、文本清洗、分块、Embedding 生成和向量入库，避免大文件处理阻塞主接口。
- 基于 pgvector 实现知识库内向量检索，检索时强制按知识库 ID、文档状态和删除状态过滤，避免跨知识库召回和已删除文档误命中，保证企业知识隔离。
- 实现 RAG 问答链路：用户提问后先召回相关 chunk，再组装带来源信息的上下文 Prompt，通过 OpenAI-compatible 模型流式生成答案，并返回引用文档、页码、chunk 等溯源信息。
- 设计 SSE 流式问答接口，支持前端实时展示 token、处理状态、引用来源和消息结束事件，提升 AI 问答过程的可感知性与可调试性。
- 落地 JWT 认证与知识库级 RBAC 权限控制，区分查看、提问、文档写入等权限，保障不同角色在知识库、文档、问答链路中的访问边界。
- 针对文档入库失败、重试、软删除等场景补齐状态流转：失败文档可记录错误原因并重试，删除文档从列表和检索中立即不可见，降低知识库维护成本。
- 为后续优化预留混合检索、Rerank、知识图谱、多模态 OCR / VLM 解析等扩展点，可进一步提升复杂企业文档的召回质量和知识组织能力。

### BI 智能问数 Agent | 全栈开发 / AI 应用开发

2025.11 - 2026.04  
面向 BI 自助分析场景的自然语言问数工具，支持业务人员通过中文问题查询销售额、订单数、商品品类、区域等业务指标，降低非技术用户使用 BI 系统时对 SQL 和报表配置的依赖。  
技术栈：NestJS / LangGraph / LangChain / MySQL / Qdrant / Elasticsearch / React / SSE

- 参与自然语言问数 Agent 的核心链路设计，将一次问数拆分为问题理解、关键词扩展、指标/字段识别、元数据检索、SQL 生成、SQL 校验、查询执行和图表展示等步骤。
- 使用 LangChain 封装 LLM 调用、Prompt 模板、元数据检索和 SQL 生成能力，沉淀可复用的问数节点，降低后续新增业务数据集的接入成本。
- 基于 LangGraph 构建多节点状态图，维护用户问题、候选字段、指标口径、召回片段、生成 SQL、执行结果和错误信息等中间状态，并通过 checkpoint 支持流程恢复与问题排查。
- 构建 BI 元数据知识库，将表结构、字段别名、指标口径、枚举取值和业务说明接入 MySQL / Qdrant / Elasticsearch，辅助 Agent 在生成 SQL 前完成字段和指标对齐。
- 设计 SQL 生成前后的安全约束：通过 Prompt 约束、只读查询限制、表字段白名单、结果行数限制和异常兜底，降低模型生成错误 SQL 或高风险 SQL 的概率。
- 实现 SSE 问数接口和前端聊天式交互，实时展示 Agent 当前处理阶段、生成过程和最终查询结果，并将结果转成表格 / 图表，提升业务人员对 AI 问数过程的信任感。
- 结合 BI 报表项目经验，对图表展示、指标口径、筛选条件和下钻路径进行适配，使问数结果能够承接到现有 BI 分析场景中。
