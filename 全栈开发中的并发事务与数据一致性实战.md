# 全栈开发中的并发、事务与数据一致性实战

> 面向 Python FastAPI / Node.js NestJS / AI 应用开发的学习与面试手册

## 0. 这份文档解决什么问题

这不是一本操作系统教材。它的目标是让有前端经验、正在转向全栈与 AI 应用开发的工程师，能够处理下面这些真实问题：

- 两个请求同时扣库存，如何避免超卖？
- 同一个报表生成接口被重复调用，如何只创建一个任务？
- 两个审核人同时修改状态，如何避免互相覆盖？
- RAG 文档因为重试被导入两次，如何保证幂等？
- 多个服务实例同时执行定时任务，如何保证只有一个实例工作？
- MQ 消息重复投递，为什么不能假设“消息只会消费一次”？
- 数据库已经更新，缓存删除失败，如何处理短暂不一致？

读完后，你应该能够完成三个动作：

1. 判断共享资源位于进程内、数据库、缓存还是分布式系统。
2. 从事务、原子更新、乐观锁、悲观锁、唯一约束、幂等、分布式锁和 MQ 中选择合适方案。
3. 在面试中讲清楚方案解决了什么、没有解决什么，以及失败时怎么恢复。

本文以 MySQL InnoDB 为主要数据库，以 Python/FastAPI 为主要代码语言，关键位置补充 Node.js/NestJS 对照。

## 1. 先建立并发问题的整体地图

### 1.1 并发、并行与异步

- **并发（Concurrency）**：多个任务在重叠的时间段内推进，不要求同一时刻真的同时执行。
- **并行（Parallelism）**：多个任务在同一时刻由不同 CPU 核心执行。
- **异步（Asynchronous）**：任务遇到等待时主动让出执行权，完成后再继续。

异步代码仍然可能产生并发问题：

```python
balance = 100

async def withdraw(amount: int):
    global balance
    old_balance = balance
    await asyncio.sleep(0.01)  # 模拟数据库或网络等待，执行权在这里让出
    balance = old_balance - amount
```

如果两个协程同时执行 `withdraw(60)`，它们都可能读到 `100`，最后都写入 `40`。系统实际执行了两次扣款，但余额只减少了一次。这就是竞态条件。

### 1.2 六个必须掌握的词

#### 共享资源

同时被多个执行单元访问或修改的数据，例如：

- 进程内变量
- 文件
- 数据库中的库存、余额、任务状态
- Redis Key
- MQ 中的任务
- 第三方支付订单

#### 竞态条件（Race Condition）

结果取决于多个操作不可预测的执行顺序。

#### 临界区（Critical Section）

访问共享资源、不能被多个执行单元随意交错执行的代码区域。

#### 原子性（Atomicity）

一组操作要么全部成功，要么全部失败；其他执行者不能看到中间状态。

#### 幂等性（Idempotency）

同一个业务操作执行一次和执行多次，对最终业务状态的影响相同。

#### 一致性（Consistency）

系统状态始终满足业务不变量，例如：

```text
库存不能小于 0
余额不能凭空增加或减少
同一份文件不能重复入库
已发布内容不能回到草稿状态
同一个业务订单只能有一条支付记录
```

### 1.3 先问“共享资源在哪一层”

```text
共享资源
├── 单进程内存：asyncio.Lock、Mutex、Semaphore、线程安全队列
├── 数据库：事务、原子 UPDATE、行锁、乐观锁、唯一约束
├── 多实例系统：数据库约束、幂等键、Redis 分布式锁
├── 异步任务：MQ、ACK、重试、幂等消费、补偿
└── 数据库 + 缓存：Cache Aside、失效策略、最终一致性
```

一个非常重要的原则：**尽量把最终正确性建立在数据所在的系统上。**

库存位于 MySQL，就优先依靠 MySQL 的事务、条件更新或约束；不要仅靠某个 API 进程里的内存锁。

## 2. HTTP 并发：Node.js 单线程为什么仍然会出错

### 2.1 一个请求不是执行完才轮到下一个

无论 FastAPI 还是 NestJS，服务器都需要同时处理大量连接。请求 A 等待数据库时，请求 B 可以继续执行。

```text
请求 A：读取库存 1 ─────────────── 写入库存 0
请求 B：      读取库存 1 ─────────────── 写入库存 0
```

两个请求都认为扣减成功，库存却只减少了一件。

### 2.2 Node.js 单线程不等于没有竞态

JavaScript 代码通常在一个事件循环线程中执行，但一次业务操作会跨越多个事件循环阶段：

```ts
let stock = 1;

async function buy() {
  const current = stock;
  await saveOrder(); // 等待期间，另一个请求可以进入 buy
  stock = current - 1;
}
```

只要中间存在 `await`，多个逻辑事务就可能交错。更何况生产环境通常还会启动多个 Node 进程、容器或服务实例。

### 2.3 FastAPI 的 async 也不是“自动线程安全”

`async def` 适合 I/O 密集型工作，但共享可变状态仍然需要保护。`asyncio.Lock` 只能协调当前事件循环中的协程，不能协调：

- 多个 Uvicorn worker
- 多个容器
- 多台服务器
- 其他语言编写的服务

### 2.4 第一条工程结论

```text
前端禁用按钮：改善体验
接口幂等：防止重复业务操作
数据库事务/约束：守住最终正确性
```

这三层经常需要同时存在，不能互相替代。

## 3. 数据库事务：后台开发最重要的并发基础

### 3.1 ACID 用业务语言理解

- **Atomicity 原子性**：转账扣款与入账必须一起成功或一起失败。
- **Consistency 一致性**：事务前后业务不变量成立。
- **Isolation 隔离性**：并发事务不会随意看见或覆盖对方的中间状态。
- **Durability 持久性**：提交后的数据在系统恢复后仍然存在。

### 3.2 没有事务会发生什么

```python
async def transfer(from_id: int, to_id: int, amount: int):
    await db.execute(
        "UPDATE account SET balance = balance - :amount WHERE id = :id",
        {"amount": amount, "id": from_id},
    )
    # 如果这里超时、进程崩溃或第二条 SQL 失败，钱已经扣了
    await db.execute(
        "UPDATE account SET balance = balance + :amount WHERE id = :id",
        {"amount": amount, "id": to_id},
    )
```

### 3.3 FastAPI + SQLAlchemy 事务示例

```python
from fastapi import HTTPException
from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession

async def transfer(
    session: AsyncSession,
    from_id: int,
    to_id: int,
    amount: int,
) -> None:
    if amount <= 0:
        raise HTTPException(400, "amount must be positive")

    async with session.begin():
        result = await session.execute(
            text("""
                UPDATE account
                SET balance = balance - :amount
                WHERE id = :id AND balance >= :amount
            """),
            {"id": from_id, "amount": amount},
        )

        if result.rowcount != 1:
            raise HTTPException(409, "insufficient balance")

        result = await session.execute(
            text("""
                UPDATE account
                SET balance = balance + :amount
                WHERE id = :id
            """),
            {"id": to_id, "amount": amount},
        )

        if result.rowcount != 1:
            raise HTTPException(404, "target account not found")
```

任何异常都会使上下文退出并回滚事务。

### 3.4 NestJS + Prisma 事务对照

```ts
await prisma.$transaction(async (tx) => {
  const debit = await tx.account.updateMany({
    where: { id: fromId, balance: { gte: amount } },
    data: { balance: { decrement: amount } },
  });

  if (debit.count !== 1) {
    throw new ConflictException('insufficient balance');
  }

  await tx.account.update({
    where: { id: toId },
    data: { balance: { increment: amount } },
  });
});
```

### 3.5 事务边界应该放在哪里

事务应该覆盖一个需要共同成功或失败的业务动作，而不是覆盖整个 HTTP 请求。

不要在事务中进行耗时的外部操作：

```text
开启事务
锁住订单
调用大模型 30 秒      ← 锁被长时间持有
上传对象存储
更新订单
提交事务
```

更好的方式通常是：

1. 在短事务中抢占或创建任务。
2. 提交事务，释放锁。
3. 执行耗时任务。
4. 在另一个短事务中写回结果。
5. 用状态机、版本号和幂等保证结果不会错写。

## 4. 隔离级别与常见并发现象

### 4.1 四种现象

#### 脏读

事务 B 读到事务 A 尚未提交的数据，A 随后回滚。

#### 不可重复读

同一事务内，两次读取同一行得到不同结果，因为其他事务提交了修改。

#### 幻读

同一事务内，两次按范围查询得到不同的行集合，因为其他事务插入或删除了满足条件的行。

#### 丢失更新（Lost Update）

两个事务都基于旧值计算新值，后写入者覆盖先写入者。

### 4.2 MySQL 常见隔离级别

| 隔离级别 | 特点 | 常见使用考虑 |
|---|---|---|
| READ UNCOMMITTED | 可读取未提交数据 | 极少使用 |
| READ COMMITTED | 每次普通读取使用新的已提交快照 | 并发较好，需注意重复读取变化 |
| REPEATABLE READ | 同一事务普通读取保持一致快照，MySQL InnoDB 默认 | 需理解快照读与当前读差异 |
| SERIALIZABLE | 最强隔离，普通读取也可能产生更多阻塞 | 正确性强，吞吐与等待成本高 |

隔离级别不是越高越好。隔离越强，通常阻塞、死锁或重试成本也越高。

### 4.3 快照读与当前读

普通 `SELECT` 通常读取一致性快照；以下操作需要读取当前版本并参与加锁：

```sql
SELECT ... FOR UPDATE;
SELECT ... FOR SHARE;
UPDATE ...;
DELETE ...;
```

面试时不要只背“REPEATABLE READ 解决幻读”。真正需要说明的是：

- 普通快照读依靠 MVCC。
- 锁定读和写操作读取当前数据。
- 范围条件可能涉及记录锁、间隙锁或 next-key lock。
- 最终锁了多少数据，和索引及扫描范围有关。

## 5. 数据库并发控制：优先掌握这五种方案

### 5.1 方案一：数据库原子更新

### 错误方式：先查再改

```python
row = await session.execute(text("SELECT stock FROM product WHERE id=:id"), {"id": product_id})
stock = row.scalar_one()

if stock <= 0:
    raise HTTPException(409, "sold out")

await session.execute(
    text("UPDATE product SET stock=:stock WHERE id=:id"),
    {"stock": stock - 1, "id": product_id},
)
```

检查与修改是两个步骤，其他事务可以插入中间。

### 正确方式：把条件和修改合并为一条 SQL

```sql
UPDATE product
SET stock = stock - 1
WHERE id = :id AND stock > 0;
```

```python
result = await session.execute(
    text("""
        UPDATE product
        SET stock = stock - 1
        WHERE id = :id AND stock > 0
    """),
    {"id": product_id},
)

if result.rowcount != 1:
    raise HTTPException(409, "sold out")
```

适合：

- 库存扣减
- 计数器
- 限额扣减
- 简单状态迁移

优点：SQL 少、锁持有时间短、并发性能通常优于“查询后在应用层计算”。

### 面试回答

> 简单库存扣减我会优先使用带条件的原子 UPDATE，把库存校验和扣减放在一条 SQL 中，并检查 affected rows。这样可以避免先查再改产生的竞态。只有业务计算较复杂、必须先读取多项数据时，才考虑显式行锁或乐观锁。

### 5.2 方案二：悲观锁 `SELECT ... FOR UPDATE`

悲观锁假设冲突可能频繁发生，因此在读取时先锁定目标记录。

```python
async with session.begin():
    row = await session.execute(
        text("""
            SELECT id, stock
            FROM product
            WHERE id = :id
            FOR UPDATE
        """),
        {"id": product_id},
    )
    product = row.mappings().one_or_none()

    if not product or product["stock"] <= 0:
        raise HTTPException(409, "sold out")

    await session.execute(
        text("UPDATE product SET stock = stock - 1 WHERE id = :id"),
        {"id": product_id},
    )
```

必须注意：

1. `FOR UPDATE` 必须处于显式事务中，否则语句结束后锁可能立即释放。
2. 锁会一直持有到 commit 或 rollback。
3. 查询条件缺少合适索引时，可能扫描并锁住远超预期的数据。
4. 多个事务以不同顺序锁多行，可能死锁。
5. 不要拿着行锁调用大模型或第三方接口。

适合：

- 需要读取多列后进行复杂判断
- 冲突频率高
- 操作必须串行化
- 任务队列表中的任务领取

数据库任务领取示例：

```sql
SELECT id
FROM job
WHERE status = 'PENDING'
ORDER BY id
LIMIT 10
FOR UPDATE SKIP LOCKED;
```

`SKIP LOCKED` 适合多个 worker 从任务表并行领取任务，但它返回的是跳过被锁行后的不一致视图，不适合一般业务查询。

### 5.3 方案三：乐观锁

乐观锁不在读取时阻塞其他事务，而是在更新时检查数据是否仍是读到的版本。

```sql
CREATE TABLE article (
    id BIGINT PRIMARY KEY,
    content TEXT NOT NULL,
    version INT NOT NULL DEFAULT 0
);
```

```sql
UPDATE article
SET content = :content,
    version = version + 1
WHERE id = :id
  AND version = :old_version;
```

如果 affected rows 为 0，表示数据已经被别人修改。

```python
result = await session.execute(
    text("""
        UPDATE article
        SET content=:content, version=version+1
        WHERE id=:id AND version=:version
    """),
    {"id": article_id, "content": content, "version": version},
)

if result.rowcount != 1:
    raise HTTPException(409, "article has been modified; reload and retry")
```

适合：

- 编辑内容
- 审核与发布
- 冲突不频繁
- 不希望长时间持有数据库锁

不适合：冲突极其频繁且每次失败都自动重试，因为大量重试会放大数据库压力。

### 5.4 方案四：状态机 + 条件更新

审核、发布、Agent 任务都不应该允许任意修改状态。

```text
PENDING → RUNNING → SUCCEEDED
                  ↘ FAILED
```

抢占任务：

```sql
UPDATE agent_task
SET status = 'RUNNING',
    worker_id = :worker_id,
    started_at = NOW()
WHERE id = :id
  AND status = 'PENDING';
```

只有一个 worker 能把 `PENDING` 改为 `RUNNING`。其他 worker 得到 affected rows = 0，必须停止执行。

写回结果时继续校验状态和 worker：

```sql
UPDATE agent_task
SET status = 'SUCCEEDED', result = :result
WHERE id = :id
  AND status = 'RUNNING'
  AND worker_id = :worker_id;
```

这比“先查状态是不是 PENDING，再 UPDATE”更安全。

### 5.5 方案五：唯一约束

当业务规则是“只能存在一条”时，唯一约束是最后防线。

```sql
CREATE TABLE document_import (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    knowledge_base_id BIGINT NOT NULL,
    file_hash CHAR(64) NOT NULL,
    status VARCHAR(32) NOT NULL,
    UNIQUE KEY uk_kb_file (knowledge_base_id, file_hash)
);
```

即使两个服务实例同时检查“文件不存在”，最终也只有一个 INSERT 能成功。

适合：

- 同一知识库中的文件指纹
- 业务订单号
- 第三方支付流水号
- 用户与角色的关联关系
- 幂等键

不要用“先 SELECT 判断不存在，再 INSERT”代替唯一约束，因为检查与插入之间仍然存在竞态窗口。

## 6. 死锁：有锁就必须接受重试

### 6.1 典型死锁

```text
事务 A：锁账户 1 → 等待账户 2
事务 B：锁账户 2 → 等待账户 1
```

数据库会检测死锁并回滚其中一个事务。

### 6.2 降低死锁概率

1. 所有代码按相同顺序加锁，例如总是先锁较小 ID。
2. 事务尽可能短。
3. 为查询条件建立合适索引，减少扫描与锁定范围。
4. 不在事务中调用外部服务。
5. 对死锁错误做有限次数、带随机退避的重试。

```python
import asyncio
import random

async def with_deadlock_retry(operation, max_attempts: int = 3):
    for attempt in range(max_attempts):
        try:
            return await operation()
        except DatabaseDeadlockError:
            if attempt == max_attempts - 1:
                raise
            await asyncio.sleep((2**attempt) * 0.05 + random.random() * 0.05)
```

只有确认错误是可重试的瞬时冲突时才能自动重试。参数校验失败、唯一约束冲突等业务错误不能盲目重试。

## 7. 接口幂等：重复请求不能产生重复业务结果

### 7.1 重复请求从哪里来

- 用户连续点击
- 前端请求库自动重试
- Nginx 或网关重试
- 网络超时，但服务端其实已经成功
- MQ 重复投递
- 定时任务重复触发
- 服务重启后恢复未确认任务

前端防抖只能减少重复请求，不能提供业务正确性保证。

### 7.2 幂等键方案

客户端为一次业务操作生成稳定的 `Idempotency-Key`：

```http
POST /reports
Idempotency-Key: 7c7b26b8-...
```

表结构：

```sql
CREATE TABLE idempotency_record (
    idempotency_key VARCHAR(128) PRIMARY KEY,
    request_hash CHAR(64) NOT NULL,
    status VARCHAR(16) NOT NULL,
    response_code INT NULL,
    response_body JSON NULL,
    created_at DATETIME NOT NULL,
    expires_at DATETIME NOT NULL
);
```

处理流程：

```text
收到请求
  ↓
尝试 INSERT 幂等记录
  ├── 成功：当前请求拥有执行权
  └── 唯一键冲突：读取已有记录
          ├── SUCCEEDED：返回之前保存的结果
          ├── PROCESSING：返回处理中或等待
          └── FAILED：根据业务规则决定重试
```

关键点：

- 同一个幂等键必须对应同一组请求参数，可存 `request_hash` 校验。
- 幂等记录和业务数据最好在同一数据库事务中写入。
- 需要设计过期时间和清理策略。
- 不能只在 Redis 中存一个短期 Key，然后完全不设数据库唯一约束。

### 7.3 RAG 文档导入示例

文件内容计算 SHA-256：

```python
import hashlib

def file_sha256(content: bytes) -> str:
    return hashlib.sha256(content).hexdigest()
```

创建导入任务：

```python
async def create_import_task(session, kb_id: int, content: bytes):
    file_hash = file_sha256(content)

    try:
        async with session.begin():
            task = DocumentImport(
                knowledge_base_id=kb_id,
                file_hash=file_hash,
                status="PENDING",
            )
            session.add(task)
            await session.flush()
            return task
    except IntegrityError:
        # 唯一约束表明这份内容已经创建过导入任务
        return await find_by_hash(session, kb_id, file_hash)
```

如果业务允许同一文件重新导入，应显式增加 `revision`、导入批次或“强制重新导入”语义，而不是删除唯一约束后任由重复。

### 7.4 幂等与锁的区别

- **锁**：控制同一时间谁能执行。
- **幂等**：即使执行多次，最终效果也正确。

锁可能过期、服务可能崩溃、网络可能重试，所以关键操作通常仍然需要幂等。

## 8. Redis 分布式锁：控制跨实例互斥

### 8.1 适用场景

- 多实例定时任务
- 同一报表只允许一个实例生成
- 防止热点资源被同时重建
- 某个外部资源不支持事务，但需要短时间互斥

不建议把 Redis 锁作为库存、余额等核心数据的唯一正确性保障。最终不变量仍应由数据库条件、事务或约束守住。

### 8.2 正确的获取方式

```text
SET lock:report:123 <random-token> NX PX 30000
```

- `NX`：Key 不存在时才写入。
- `PX`：设置毫秒级过期时间，避免客户端崩溃后永久死锁。
- `random-token`：标识锁的拥有者。

### 8.3 为什么不能直接 DEL

错误流程：

```text
实例 A 获取锁，锁 30 秒后过期
实例 B 获取同一个锁
实例 A 终于执行完，直接 DEL
实例 A 删除了实例 B 的锁
```

释放锁必须比较 token 后再删除，而且比较和删除必须原子执行：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 8.4 Python 示例

```python
import secrets

RELEASE_LOCK = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

async def generate_report(redis, report_id: int):
    key = f"lock:report:{report_id}"
    token = secrets.token_hex(16)

    acquired = await redis.set(key, token, nx=True, px=30_000)
    if not acquired:
        raise RuntimeError("report is being generated")

    try:
        return await do_generate_report(report_id)
    finally:
        await redis.eval(RELEASE_LOCK, 1, key, token)
```

### 8.5 Node.js 对照

```ts
const key = `lock:report:${reportId}`;
const token = randomUUID();
const acquired = await redis.set(key, token, { NX: true, PX: 30_000 });

if (!acquired) {
  throw new ConflictException('report is being generated');
}

try {
  return await generateReport(reportId);
} finally {
  await redis.eval(RELEASE_LOCK_SCRIPT, {
    keys: [key],
    arguments: [token],
  });
}
```

### 8.6 锁过期但任务没完成怎么办

可选方案：

1. 将过期时间设置为明显大于任务的正常耗时。
2. 使用成熟客户端提供的看门狗续期。
3. 将长任务改成数据库状态机或 MQ 任务，不长时间持有锁。
4. 引入 fencing token，资源端拒绝旧持有者的写入。

如果无法给出“锁过期后旧执行者如何被阻止写入”的答案，就不能宣称该锁保证了严格互斥。

### 8.7 面试回答

> Redis 分布式锁主要解决多实例之间的互斥。我会使用 `SET NX PX` 和随机 token 获取锁，释放时通过 Lua 比较 token 后删除，避免误删其他实例的锁。同时考虑锁过期、续期、服务暂停和网络分区。对于库存或余额，Redis 锁只是协调手段，数据库的事务、条件更新和唯一约束仍是最终正确性保障。

## 9. MQ：削峰与异步，不是并发问题的万能答案

### 9.1 MQ 适合解决什么

- 削峰：把瞬时流量变成可控消费速率。
- 异步：接口快速返回，后台处理耗时任务。
- 解耦：生产者不依赖消费者内部实现。
- 排队：任务按队列调度。
- 重试与失败隔离。
- 最终一致性。

典型场景：

- BI 报表生成
- 文档解析、分块、Embedding
- 设备报文异步处理
- 邮件、通知
- Agent 后台工作流

### 9.2 为什么 MQ 仍然可能重复消费

```text
消费者完成数据库更新
  ↓
消费者还没 ACK 就崩溃
  ↓
Broker 重新投递消息
  ↓
数据库更新再次执行
```

因此常见可靠语义是 **at-least-once**：消息可能重复，但不轻易丢失。消费者必须幂等。

### 9.3 幂等消费者

```sql
CREATE TABLE processed_message (
    consumer_name VARCHAR(64) NOT NULL,
    message_id VARCHAR(128) NOT NULL,
    processed_at DATETIME NOT NULL,
    PRIMARY KEY (consumer_name, message_id)
);
```

处理消息与业务更新放在同一事务中：

```python
async def consume(message):
    try:
        async with session.begin():
            await session.execute(
                text("""
                    INSERT INTO processed_message
                        (consumer_name, message_id, processed_at)
                    VALUES (:consumer, :message_id, NOW())
                """),
                {"consumer": "report-worker", "message_id": message.id},
            )

            await mark_report_finished(session, message.report_id)
    except IntegrityError:
        # message_id 已处理，重复消息直接视为成功
        pass

    await message.ack()
```

ACK 必须在业务事务成功之后。否则先 ACK、后写数据库时进程崩溃，消息已经被 Broker 删除，业务却没有完成。

### 9.4 生产者：数据库提交成功但消息没发出去

错误流程：

```text
提交订单事务成功
  ↓
发布 MQ 消息失败
  ↓
订单存在，但后续流程永远没收到消息
```

### 9.5 Transactional Outbox

在同一个数据库事务中写业务数据和 outbox 事件：

```sql
START TRANSACTION;

INSERT INTO report_task (...);

INSERT INTO outbox_event (
    event_id, event_type, aggregate_id, payload, status
) VALUES (
    :event_id, 'REPORT_CREATED', :report_id, :payload, 'PENDING'
);

COMMIT;
```

独立发布器持续扫描 `PENDING` 事件并发送到 MQ，收到 Broker confirm 后标记为 `PUBLISHED`。即使发布器崩溃，也能再次扫描；消息可能重复，因此消费者仍需幂等。

### 9.6 ACK、Confirm、重试和死信

- **Consumer ACK**：消费者告诉 Broker，消息已成功处理。
- **Publisher Confirm**：Broker 告诉生产者，消息已由 Broker 接管。
- **Retry**：瞬时故障可重试，但必须限制次数并退避。
- **Dead Letter Queue**：多次失败的消息进入死信队列，等待排查或人工处理。
- **Prefetch**：限制消费者同时持有但尚未 ACK 的消息数量，避免消费者内存被压垮。

不要无限 requeue。永久性错误会形成“毒消息”循环。

### 9.7 顺序消费的边界

一个队列只有一个消费者可以提高顺序性，但会牺牲吞吐。更常见的做法是按业务键分区：

```text
partition = hash(order_id) % N
```

同一个订单进入同一分区，不同订单可以并行处理。即便消息有顺序，数据库更新仍应校验当前状态，防止迟到消息把新状态覆盖成旧状态。

### 9.8 面试回答

> MQ 可以把并发请求转化为排队处理，适合削峰、异步和解耦，但它不天然提供 exactly-once 业务语义。生产端需要考虑消息是否真正到达 Broker，可使用 publisher confirm 和 outbox；消费端通常按至少一次投递设计，在数据库事务中记录 message_id，完成业务更新后再 ACK。失败消息需要有限重试、退避和死信队列。

## 10. 缓存并发与数据库一致性

### 10.1 Cache Aside

读取：

```text
读缓存
├── 命中：返回
└── 未命中：读数据库 → 写缓存 → 返回
```

更新通常采用：

```text
更新数据库
  ↓
删除缓存
```

不优先“更新数据库后直接更新缓存”的原因是：缓存结构可能由多表计算得到，且并发更新顺序可能使旧值覆盖新值。

### 10.2 为什么不能先删缓存再更新数据库

```text
请求 A 删除缓存
请求 B 缓存未命中，读取数据库旧值
请求 A 更新数据库新值
请求 B 把旧值写回缓存
```

最终缓存长期保存旧值。

### 10.3 更新数据库后删除缓存也不是绝对一致

数据库提交成功后，删除缓存可能失败。因此实际系统还会考虑：

- 缓存 TTL
- 删除失败重试
- 通过 outbox/MQ 异步失效缓存
- 订阅数据库变更日志
- 对强一致读绕过缓存

### 10.4 缓存击穿与重建锁

热点 Key 过期时，大量请求同时查询数据库：

```python
value = await redis.get(key)
if value is not None:
    return value

if await redis.set(f"lock:{key}", token, nx=True, px=5000):
    try:
        value = await load_from_database()
        await redis.set(key, serialize(value), ex=300)
        return value
    finally:
        await safe_release_lock(...)

# 没拿到锁的请求短暂等待后重读缓存，或返回可接受的旧值
```

还可以使用逻辑过期和后台刷新，避免热点请求同步等待。

### 10.5 三个常见名词

- **缓存穿透**：查询不存在的数据，缓存与数据库都无法命中。可缓存空值、参数校验、使用布隆过滤器。
- **缓存击穿**：单个热点 Key 失效，大量请求打到数据库。可互斥重建、逻辑过期。
- **缓存雪崩**：大量 Key 同时失效或 Redis 故障。可随机化 TTL、多级缓存、限流降级、高可用部署。

## 11. 进程、线程、协程与本地同步工具

### 11.1 什么时候使用本地锁

本地锁适合保护仅存在于当前进程的资源：

- 内存队列
- 非线程安全客户端
- 本地文件
- 单进程内的缓存重建

如果数据位于数据库，通常让数据库处理并发更可靠。

### 11.2 Python `asyncio.Lock`

```python
import asyncio

lock = asyncio.Lock()
cache: dict[str, object] = {}

async def load_config(key: str):
    if key in cache:
        return cache[key]

    async with lock:
        # Double check：等待锁期间可能已有其他协程完成加载
        if key in cache:
            return cache[key]
        cache[key] = await fetch_config(key)
        return cache[key]
```

边界：它不能跨进程或跨机器工作，也不是线程锁。

### 11.3 Semaphore：限制并发量

RAG Embedding、外部 API 和数据库连接都有容量上限：

```python
semaphore = asyncio.Semaphore(8)

async def embed_chunk(chunk: str):
    async with semaphore:
        return await embedding_client.embed(chunk)

results = await asyncio.gather(*(embed_chunk(chunk) for chunk in chunks))
```

Semaphore 不保证业务幂等，只限制同时执行的任务数量。

### 11.4 Node.js 并发限制

```ts
async function mapWithConcurrency<T, R>(
  items: T[],
  limit: number,
  worker: (item: T) => Promise<R>,
): Promise<R[]> {
  const results: R[] = new Array(items.length);
  let cursor = 0;

  async function run() {
    while (true) {
      const index = cursor++;
      if (index >= items.length) return;
      results[index] = await worker(items[index]);
    }
  }

  await Promise.all(Array.from({ length: limit }, run));
  return results;
}
```

实际项目中也可使用成熟并发限制库，但要理解它只限制当前进程中的 Promise 数量。

### 11.5 线程池、连接池与任务队列不要混淆

- **线程池**：复用线程执行阻塞或 CPU 任务。
- **数据库连接池**：复用有限数据库连接。
- **协程并发限制**：限制同时进行的异步操作。
- **MQ**：跨进程、可持久化的任务排队与解耦。

开 1000 个协程不代表数据库能承受 1000 个并发查询。连接池和数据库容量通常才是真实瓶颈。

## 12. 结合你的项目背景怎么设计

### 12.1 BI 报表生成

### 风险

- 用户重复点击生成。
- HTTP 超时后客户端重试。
- 多个 worker 重复生成同一份报表。
- 报表生成耗时，不能一直占用 HTTP 请求。

### 推荐设计

```text
POST /reports
  ↓ Idempotency-Key
事务内创建 report_task + outbox_event
  ↓
Outbox Publisher 发送 MQ
  ↓
Worker 使用 message_id 幂等消费
  ↓
条件更新：PENDING → RUNNING
  ↓
生成文件并上传对象存储
  ↓
条件更新：RUNNING → SUCCEEDED
```

数据库状态和唯一约束保证最终正确性；Redis 锁可以减少重复计算，但不应成为唯一保障。

### 12.2 RAG 文档导入

### 风险

- 同一文件重复上传。
- 文档解析消息重复消费。
- 部分 chunk 已写入，后续失败。
- Embedding API 限流。
- 旧版本导入任务迟到，覆盖新版本状态。

### 推荐设计

- 用 `knowledge_base_id + file_hash` 唯一约束控制重复内容。
- 文档、导入任务和 outbox 在事务中创建。
- 消费者用 message_id 幂等。
- chunk 使用 `document_id + revision + chunk_index` 唯一约束。
- 用 Semaphore 限制 Embedding 并发。
- 状态更新带 `revision` 或 `worker_id` 条件。
- 失败任务记录错误阶段，允许从安全检查点重试。

### 12.3 Text-to-SQL / LangGraph Agent

### 风险

- 同一个 run 被重复启动。
- checkpoint 恢复后重复执行有副作用的节点。
- SQL 执行节点被重放。
- 两个 worker 同时推进同一个任务状态。

### 推荐设计

- `thread_id` 用于会话/状态隔离，不要把它误当业务幂等键。
- 每次业务执行使用唯一 `run_id`。
- 有副作用的节点使用 `run_id + node_name` 建立执行记录或唯一约束。
- 用条件更新抢占任务。
- 查询 SQL 强制只读、超时、行数限制和资源限制。
- checkpoint 解决状态持久化与恢复；数据库幂等解决节点重放的副作用。

一句话区分：

```text
checkpoint 负责“恢复执行状态”；
幂等负责“重复执行也不会产生重复副作用”。
```

### 12.4 设备报文接入

### 风险

- TCP 重连导致设备重发报文。
- 多个消费者同时更新设备最后状态。
- 报文乱序到达。
- 高频写入压垮数据库。

### 推荐设计

- 使用设备 ID + 报文序列号建立唯一约束。
- 原始报文先持久化，再异步解析。
- 状态更新带设备时间戳或序列号条件，拒绝旧报文覆盖新状态。
- 批量写入历史数据，热点当前状态单独维护。
- MQ 按 device_id 分区，兼顾同设备顺序和跨设备并行。

```sql
UPDATE device_state
SET value = :value,
    sequence_no = :sequence_no
WHERE device_id = :device_id
  AND sequence_no < :sequence_no;
```

## 13. 方案选择表

| 方案 | 主要解决的问题 | 不解决什么 | 典型场景 |
|---|---|---|---|
| 原子 UPDATE | 单条数据的条件修改 | 跨多资源事务 | 库存、计数、状态抢占 |
| 数据库事务 | 多条数据库操作共同提交/回滚 | 跨数据库和外部 API | 转账、创建任务与事件 |
| 悲观锁 | 高冲突下串行访问记录 | 长任务、跨系统互斥 | 复杂余额计算、任务领取 |
| 乐观锁 | 检测并拒绝覆盖更新 | 高频冲突下的重试压力 | 编辑、审核、发布 |
| 唯一约束 | “只能有一条”的不变量 | 复杂流程协调 | 订单号、文件指纹、消息 ID |
| 本地锁 | 单进程内互斥 | 多进程、多实例 | 内存缓存、本地资源 |
| Redis 锁 | 多实例短时互斥 | 数据事务、自动幂等 | 定时任务、缓存重建 |
| 幂等键 | 重复请求复用同一业务结果 | 同时执行的性能浪费 | 创建订单、生成报表 |
| MQ | 异步、削峰、排队、解耦 | 自动 exactly-once | 报表、文档解析、通知 |
| Outbox | 数据库提交与事件发布的一致性 | 消费端重复处理 | 可靠领域事件 |
| Semaphore | 限制并发量 | 跨实例全局限流、业务幂等 | Embedding、外部 API |

### 快速决策顺序

```text
1. 能否用一条带条件的 SQL 原子完成？
   能 → 优先原子 UPDATE / INSERT

2. 是否必须让多条数据库操作共同成功？
   是 → 事务

3. 是否是“只能存在一条”的业务规则？
   是 → 唯一约束

4. 冲突是低频还是高频？
   低频 → 乐观锁
   高频且必须串行 → 悲观锁

5. 是否跨多个服务实例互斥？
   是 → Redis 锁或数据库抢占，同时保留幂等/约束

6. 是否耗时、流量突发或需要重试？
   是 → MQ + 幂等消费者 + ACK + DLQ

7. 数据库事务后是否必须可靠发事件？
   是 → Outbox
```

## 14. 并发测试：不要只看代码“感觉没问题”

### 14.1 Python 并发请求脚本

```python
import asyncio
import httpx

URL = "http://localhost:8000/products/1/purchase"

async def purchase(client: httpx.AsyncClient, index: int):
    response = await client.post(
        URL,
        headers={"Idempotency-Key": f"test-{index}"},
    )
    return response.status_code, response.text

async def main():
    async with httpx.AsyncClient(timeout=10) as client:
        results = await asyncio.gather(
            *(purchase(client, index) for index in range(100))
        )

    success = sum(status == 200 for status, _ in results)
    conflict = sum(status == 409 for status, _ in results)
    print({"success": success, "conflict": conflict})

asyncio.run(main())
```

如果初始库存是 10，测试后至少验证：

```text
成功订单数 = 10
库存 = 0
库存从未小于 0
不存在重复业务订单
```

只检查 HTTP 成功率不够，必须检查数据库最终状态和业务不变量。

### 14.2 重复幂等键测试

把 100 个请求全部使用相同 Idempotency-Key，验证：

- 只创建一个任务。
- 后续请求得到相同结果或明确的处理中状态。
- 请求参数不同但幂等键相同会被拒绝。

### 14.3 故障注入

在下面这些位置主动抛出异常：

- 数据库第一步成功、第二步之前
- 业务写入成功、ACK 之前
- outbox 发送成功、标记 PUBLISHED 之前
- Redis 锁过期之后
- 外部 API 成功、本地状态写回之前

并发正确性不是“正常路径能跑”，而是故障与重试发生后不变量仍然成立。

## 15. 高频面试题与回答模板

### 15.1 并发请求访问共享资源怎么办？

> 我会先判断共享资源在哪一层。如果是单进程内存，可以使用互斥锁、Semaphore 或线程安全结构。如果资源在数据库，优先考虑带条件的原子 UPDATE、事务、行锁、乐观锁和唯一约束。如果是多实例系统，需要增加业务幂等，必要时使用 Redis 分布式锁。如果任务耗时或流量高，可以用 MQ 削峰和异步，但消费端仍需依靠事务和幂等保证正确性。

### 15.2 Node.js 是单线程，为什么还会超卖？

> 单线程只表示 JavaScript 通常由一个事件循环线程执行，不表示一个业务请求会连续执行到底。请求在等待数据库或网络时会让出执行权，另一个请求可以读取相同旧值；生产环境还可能有多个进程和实例。因此数据库共享资源仍需使用原子 SQL、事务或锁保护。

### 15.3 乐观锁和悲观锁怎么选？

> 冲突低、读取多、希望减少阻塞时选择乐观锁，通过 version 条件更新并在冲突时返回 409 或有限重试。冲突高、业务必须读取复杂状态后串行修改时使用悲观锁。简单扣减通常先考虑条件原子 UPDATE，它往往比先 SELECT 再加锁更直接。

### 15.4 `SELECT ... FOR UPDATE` 一定只锁一行吗？

> 不一定。InnoDB 对扫描到的索引记录加锁，具体范围与隔离级别、索引和查询条件有关。唯一索引等值查询通常能精确锁定记录；范围查询或缺少索引可能锁定索引范围，严重时扫描并影响大量记录。因此使用行锁必须检查执行计划和索引。

### 15.5 Redis 锁和数据库锁有什么区别？

> 数据库锁直接保护数据库记录，并和事务提交、回滚绑定；Redis 锁主要协调多个服务实例的执行权。Redis 锁存在过期、续期、误释放、网络分区等问题，不能自动保证数据库业务原子性。核心数据仍应依靠数据库事务、条件更新和约束。

### 15.6 MQ 如何保证消息不丢、不重复？

> 严格来说要分别处理生产端、Broker 和消费端。生产端使用持久化消息、publisher confirm，数据库状态与消息发布之间可用 outbox。消费端使用手动 ACK，在业务事务提交后确认。因为故障恢复可能导致重复投递，消费者需要用 message_id 和唯一约束实现幂等。多次失败进入死信队列，不能无限重试。

### 15.7 有 checkpoint 为什么 Agent 节点还要幂等？

> checkpoint 保存和恢复 Graph State，但恢复、重试或 time travel 可能再次执行节点。如果节点会写数据库、发送通知或调用有副作用的外部接口，仅恢复状态不能避免重复副作用。因此需要用 run_id、node_name、业务唯一键或执行记录实现节点级幂等。

## 16. 学习顺序与练习

### 第一阶段：数据库并发

1. 写一个错误的库存“先查再改”接口。
2. 用 100 个并发请求复现超卖或成功数异常。
3. 分别改成原子 UPDATE、悲观锁和乐观锁。
4. 比较成功率、延迟、锁等待和代码复杂度。

### 第二阶段：幂等与状态机

1. 写一个报表任务创建接口。
2. 使用相同幂等键并发请求 100 次。
3. 用唯一约束保证只创建一个任务。
4. 实现 `PENDING → RUNNING → SUCCEEDED/FAILED` 条件更新。

### 第三阶段：Redis 与缓存

1. 启动两个 API 实例。
2. 用 Redis 锁保证同一报表只有一个实例生成。
3. 人为让任务执行时间超过锁 TTL，观察问题。
4. 增加 token 校验、续期或数据库状态保护。

### 第四阶段：MQ 可靠性

1. 创建报表任务并发送 MQ。
2. 在数据库提交后、ACK 前主动终止消费者。
3. 观察消息重新投递。
4. 增加 message_id 去重表。
5. 实现有限重试与死信队列。
6. 使用 outbox 解决数据库提交后消息发送失败。

## 17. 最后记住这十句话

1. 并发问题的本质是多个执行者访问共享状态。
2. `async/await` 不会自动消除竞态条件。
3. 本地锁不能保护多进程、多实例共享资源。
4. 简单扣减优先考虑带条件的原子 UPDATE。
5. 业务“只能有一条”时，唯一约束是最后防线。
6. 事务要短，不要拿着数据库锁调用外部服务。
7. 锁解决互斥，幂等解决重复执行，两者不能互相替代。
8. Redis 分布式锁是协调手段，不是数据库事务的替代品。
9. MQ 通常按至少一次投递设计，所以消费者必须幂等。
10. checkpoint 负责恢复状态，业务幂等负责避免重复副作用。
