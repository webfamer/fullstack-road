把“会用 AI 框架”讲成“能落地 Agent 应用”

这份手册围绕求职面试最常问的三件套：LangChain 负责模型/工具/RAG 组件，LangGraph 负责状态化编排，DeepAgents 负责更上层的工程化 Agent harness。

0. 一张总图：三者到底是什么关系

最稳的表达是：LangChain 是组件层，LangGraph 是编排运行时，DeepAgents 是面向复杂任务的 Agent 工程套件。

用户需求 │ ▼ LangChain：Prompt / Model / Tool / OutputParser / RAG / LCEL │ 适合：单步调用、线性链路、快速 agent ▼ LangGraph：State / Node / Edge / Conditional Edge / Checkpoint / Interrupt │ 适合：多步骤、有分支、可恢复、可人工确认的流程 ▼ DeepAgents：Planning / Todo / Filesystem / Subagents / Memory / Middleware 适合：长任务、复杂项目、需要子代理和上下文管理的应用
框架一句话面试关键词什么时候用
LangChain构建 LLM 应用和 Agent 的基础组件与集成层。PromptTemplate、Model、Tool、OutputParser、Runnable、RAG要快速把模型、工具、检索、结构化输出串起来。
LangGraph低层 Agent orchestration runtime，用状态图控制复杂流程。State、Node、Edge、Conditional Edge、Checkpoint、Interrupt流程需要循环、分支、状态持久化、人工介入、多 Agent。
DeepAgents基于 LangChain/LangGraph 的 agent harness，内置复杂任务常用能力。planning、todo、filesystem、subagents、memory、permissions、middleware要做类似 Claude Code/Cursor 的长任务 Agent。

1. LangChain：你必须讲清楚的核心组件

1.1 PromptTemplate：把 prompt 从字符串变成可维护模板

PromptTemplate 解决的是 prompt 复用、变量注入、输出约束和测试的问题。不要把 prompt 写成随手拼接的字符串，否则后续很难维护。

import { PromptTemplate } from "@langchain/core/prompts";

const sqlPrompt = PromptTemplate.fromTemplate(`
你是一个 BI SQL 助手。
根据用户问题和字段说明生成 SQL。

用户问题：{question}
字段说明：{schema}

要求：
1. 只使用字段说明里出现的表和字段。
2. 只输出 SQL，不要解释。
3. 默认加 LIMIT 100。
`);

const promptValue = await sqlPrompt.invoke({
  question: "最近 7 天各地区销售额是多少？",
  schema: "orders(region, amount, created_at)",
});
面试讲法：PromptTemplate 的价值不是“好看”，而是让 prompt 参数化、可测试、可版本化。复杂 Agent 里每个节点最好有独立 prompt。

1.2 Model：统一不同模型供应商的调用方式

LangChain 的 Model 抽象让你用类似接口接 OpenAI、Anthropic、Gemini 或 OpenAI-compatible 模型。它屏蔽了不同厂商消息格式差异，但不保证模型能力完全一致。

import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: process.env.MODEL_NAME ?? "qwen-plus",
  temperature: 0,
  apiKey: process.env.OPENAI_API_KEY,
  configuration: {
    baseURL: process.env.OPENAI_BASE_URL,
  },
});

const res = await model.invoke("用一句话解释 RAG");
console.log(res.content);

常见追问：temperature 怎么选？结构化输出一般设低温；创意文案可以升高。模型能不能 tool calling、structured output、streaming，要实测。

1.3 OutputParser / Structured Output：让模型输出能被程序消费

模型自然语言输出不稳定，程序需要 JSON、枚举、数组等稳定结构。OutputParser 或 structured output 就是为了把“文本”变成“数据”。

import { z } from "zod";

const QueryPlan = z.object({
  metric: z.string().describe("用户要查询的指标"),
  dimensions: z.array(z.string()).describe("分组维度"),
  timeRange: z.string().describe("时间范围"),
});

const planner = model.withStructuredOutput(QueryPlan, {
  name: "query_plan",
});

const plan = await planner.invoke("按城市统计昨天订单数");
// { metric: "订单数", dimensions: ["城市"], timeRange: "昨天" }
容易混：Structured Output 是让模型按 schema 输出；Tool Calling 是让模型决定调用真实函数。两者底层可能都借助 function/tool schema，但目的不同。

1.4 Tool：让模型能行动

Tool 是带描述和参数 schema 的函数。模型不是直接执行代码，而是生成 tool call；程序执行工具后，把 ToolMessage 结果放回消息列表。

import { tool } from "@langchain/core/tools";
import { z } from "zod";

const getOrderCount = tool(async ({ city, date }) => {
  // 真实项目里这里查数据库或调用后端 API
  return JSON.stringify({ city, date, count: 1280 });
}, {
  name: "get_order_count",
  description: "查询某城市某天的订单数",
  schema: z.object({
    city: z.string(),
    date: z.string(),
  }),
});

const modelWithTools = model.bindTools([getOrderCount]);
const aiMessage = await modelWithTools.invoke("查一下杭州昨天订单数");
console.log(aiMessage.tool_calls);

Tool 设计原则：名字明确、描述具体、参数少而准、返回结构化、失败也结构化、危险操作加确认。

1.5 Runnable / LCEL：把组件串成链

Runnable 是 LangChain 统一调用协议，常见方法是 invoke、batch、stream。LCEL 用管道把 prompt、model、parser、retriever 串起来。

import { StringOutputParser } from "@langchain/core/output_parsers";
import { RunnableSequence } from "@langchain/core/runnables";

const prompt = PromptTemplate.fromTemplate("把下面内容总结成三点：{text}");
const parser = new StringOutputParser();

// 两种写法都常见：pipe 或 RunnableSequence
const chain = prompt.pipe(model).pipe(parser);
// const chain = RunnableSequence.from([prompt, model, parser]);

const summary = await chain.invoke({ text: "这里是一段很长的会议纪要..." });
面试回答：普通 chain 适合线性流程；如果出现循环、条件分支、人工确认、状态恢复，就升级到 LangGraph。

1.6 RAG 基础链路：让模型使用你的私有知识

文档/网页/PDF → Loader 加载成 Document → Splitter 切成 chunk → Embedding 转向量 → VectorStore 存储 → Retriever 根据 query 取 topK → Prompt 注入上下文 → Model 生成答案
import { RecursiveCharacterTextSplitter } from "@langchain/textsplitters";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";
import { RunnablePassthrough } from "@langchain/core/runnables";
import { StringOutputParser } from "@langchain/core/output_parsers";

const docs = [{ pageContent: "LangGraph 用状态图编排复杂 Agent。", metadata: { source: "note" } }];
const splitter = new RecursiveCharacterTextSplitter({ chunkSize: 500, chunkOverlap: 80 });
const chunks = await splitter.splitDocuments(docs);

const vectorstore = await MemoryVectorStore.fromDocuments(
  chunks,
  new OpenAIEmbeddings({ model: process.env.EMBEDDINGS_MODEL_NAME })
);
const retriever = vectorstore.asRetriever({ k: 4 });

const ragPrompt = PromptTemplate.fromTemplate(`
只根据上下文回答问题。\n\n上下文：{context}\n\n问题：{question}
`);

const formatDocs = (docs) => docs.map(d => d.pageContent).join("\n\n");

const ragChain = RunnableSequence.from([
  {
    context: retriever.pipe(formatDocs),
    question: new RunnablePassthrough(),
  },
  ragPrompt,
  model,
  new StringOutputParser(),
]);

const answer = await ragChain.invoke("LangGraph 解决什么问题？");

RAG 面试重点：chunk size、overlap、embedding、topK、rerank、query rewrite、引用来源、幻觉控制、权限过滤。

1.7 怎么把模型调用、结构化输出、工具调用、检索组合成一个链

回答时不要只说“用 LCEL 串起来”，要讲清楚各层职责。

用户问题 → Structured Output 提取意图/参数 → Retriever 检索业务知识/字段说明 → PromptTemplate 组装任务上下文 → Model 生成工具调用或答案 → Tool 查数据库/API → OutputParser 整理最终 JSON/文本 → 前端渲染
// 伪代码：BI 问数链路
const intent = await intentExtractor.invoke(userQuestion);
const context = await retriever.invoke(intent.metric + " " + intent.dimensions.join(" "));
const sql = await sqlGenerator.invoke({ question: userQuestion, context });
const rows = await queryDatabaseTool.invoke({ sql });
const finalAnswer = await answerFormatter.invoke({ question: userQuestion, rows });

2. LangGraph：为什么不用普通 chain，而要用状态图

普通 chain 是流水线；LangGraph 是状态机。只要流程需要“根据上一步结果决定下一步”，LangGraph 就更稳。

2.1 State：流程共享状态

State 是图里每个节点共享的数据。它不只是 messages，也可以包含 question、retrievedDocs、sql、error、retryCount、finalAnswer。

import { StateGraph, StateSchema, START, END } from "@langchain/langgraph";
import { z } from "zod/v4";

const AgentState = new StateSchema({
  question: z.string(),
  docs: z.array(z.string()).default([]),
  sql: z.string().optional(),
  error: z.string().optional(),
  answer: z.string().optional(),
});

2.2 Node：一个明确处理步骤

Node 是函数：接收 state，返回部分 state 更新。好的节点应该职责单一。

const retrieveNode = async (state) => {
  const docs = await retriever.invoke(state.question);
  return { docs: docs.map(d => d.pageContent) };
};

const answerNode = async (state) => {
  const res = await model.invoke(`根据资料回答：${state.docs.join("\n")}\n问题：${state.question}`);
  return { answer: String(res.content) };
};

2.3 Edge / Conditional Edge：固定流转和条件流转

Edge 表示固定下一步；Conditional Edge 表示根据 state 路由。

const graph = new StateGraph(AgentState)
  .addNode("retrieve", retrieveNode)
  .addNode("answer", answerNode)
  .addEdge(START, "retrieve")
  .addEdge("retrieve", "answer")
  .addEdge("answer", END)
  .compile();
const routeAfterSql = (state) => {
  if (state.error && state.retryCount < 2) return "fix_sql";
  if (state.error) return "fail";
  return "summarize";
};

graphBuilder.addConditionalEdges("execute_sql", routeAfterSql, [
  "fix_sql",
  "fail",
  "summarize",
]);

2.4 Checkpoint：流程可恢复

Checkpoint 把每一步 state 存下来。它解决长任务失败恢复、多轮对话线程状态、人工中断后继续执行的问题。

import { MemorySaver } from "@langchain/langgraph";

const checkpointer = new MemorySaver();
const app = graphBuilder.compile({ checkpointer });

await app.invoke(
  { question: "查一下昨天销售额" },
  { configurable: { thread_id: "user-123-session-1" } }
);

面试讲法：Checkpoint 保存的是工作流状态，不等于长期记忆。它更像“流程快照”。

2.5 Interrupt：关键步骤让人确认

Interrupt 用于 Human-in-the-loop。比如执行删除、发邮件、跑 SQL、扣费前暂停，让用户确认。

// 伪代码：执行 SQL 前中断确认
const confirmSqlNode = async (state) => {
  // 实际项目中用 LangGraph interrupt API 暂停并返回给前端
  return { needsApproval: true, sqlPreview: state.sql };
};

2.6 SQL Agent 怎么拆节点

question → extract_intent：抽取指标、维度、时间 → retrieve_schema：检索表结构/字段说明/指标口径 → generate_sql：生成 SQL → validate_sql：检查表名、字段、危险语句 → execute_sql：执行查询 → if error → fix_sql → execute_sql → summarize：把 rows 转成自然语言/图表配置
为什么用 LangGraph：SQL Agent 有错误修正循环、危险 SQL 审批、执行失败重试、结果总结，普通 chain 会变得很绕。

2.7 RAG Agent 怎么拆节点

question → classify：判断是否需要检索 → rewrite_query：改写检索 query → retrieve_vector / retrieve_keyword：向量 + 关键词检索 → rerank：重排 → grade_docs：判断资料是否足够 → if not enough → rewrite_query → generate_answer → verify_citations / answer_check

3. DeepAgents：LangGraph 之上的工程化封装

DeepAgents 不是替代 LangGraph，而是把复杂 Agent 常用能力打包：计划、文件系统、子 Agent、记忆、权限、人类审批、上下文管理。

3.1 最小 DeepAgent

import { createDeepAgent } from "deepagents";
import { tool } from "langchain";
import * as z from "zod";

const searchDocs = tool(async ({ query }) => {
  return "检索结果：LangGraph 适合状态化多步骤 Agent。";
}, {
  name: "search_docs",
  description: "搜索内部 Agent 教程文档",
  schema: z.object({ query: z.string() }),
});

const agent = createDeepAgent({
  tools: [searchDocs],
  systemPrompt: "你是一个 Agent 学习助手，回答要结构化。",
});

const result = await agent.invoke({
  messages: [{ role: "user", content: "帮我复习 LangGraph" }],
});

3.2 DeepAgents 解决什么工程问题

Planning / Todo

把复杂任务拆成步骤，记录进度，避免 Agent 乱跑。

Filesystem

把大上下文、工具结果、草稿写入虚拟文件系统，减少 token 压力。

Subagents

把搜索、分析、编码、审稿等子任务交给不同 Agent，隔离上下文。

Memory

跨线程/跨会话保存长期偏好、事实、项目知识。

Permissions

限制文件读写、shell 执行、敏感工具调用。

Human-in-the-loop

关键动作前中断，让人确认后继续。

3.3 面试讲法

DeepAgents 解决的是“把 LangGraph/Agent Loop 做成可用于真实长任务的工程套件”。它把规划、上下文压缩、文件系统、子 Agent、记忆、权限和流式事件这些能力预置好，适合做类似代码助手、研究助手、运维助手的复杂 Agent。

4. 中间件：作用、接入方式、怎么讲

这里的“中间件”分两类：一类是 LangChain/DeepAgents 执行链路里的 middleware hooks；另一类是 Agent 应用接入的基础设施中间件。

4.1 LangChain Middleware：拦截 Agent 执行过程

Middleware 用来控制 agent loop 的每一步：模型调用前后、工具调用前后、Agent 开始和结束。

import { createAgent, summarizationMiddleware, humanInTheLoopMiddleware } from "langchain";

const agent = createAgent({
  model: "gpt-5.5",
  tools: [searchDocs, sendEmail],
  middleware: [
    summarizationMiddleware({
      model: "gpt-5.4-mini",
      trigger: { tokens: 4000 },
      keep: { messages: 20 },
    }),
    humanInTheLoopMiddleware({
      interruptOn: { send_email: true },
    }),
  ],
});
Hook什么时候运行典型用途
beforeAgentAgent 开始时一次初始化上下文、鉴权、加载用户偏好
beforeModel每次模型调用前动态 prompt、裁剪消息、注入 RAG 资料
afterModel每次模型响应后记录日志、检查输出、统计 token
afterAgentAgent 结束时一次保存结果、写记忆、埋点
wrapModelCall包裹模型调用重试、fallback、缓存、动态换模型
wrapToolCall包裹工具调用工具重试、限流、权限、错误兜底

4.2 自定义 Middleware 示例

import { createMiddleware, AIMessage } from "langchain";

const messageLimitMiddleware = createMiddleware({
  name: "MessageLimitMiddleware",
  beforeModel: {
    canJumpTo: ["end"],
    hook: (state) => {
      if (state.messages.length > 50) {
        return {
          messages: [new AIMessage("对话太长，请开启新话题。")],
          jumpTo: "end",
        };
      }
    },
  },
  afterModel: (state) => {
    console.log("model returned", state.messages.at(-1)?.content);
  },
});

5. 三个高频面试问题怎么回答

5.1 怎么把模型调用、结构化输出、工具调用、检索组合成一个链？

我会把链路拆成四层：先用 structured output 做意图抽取，再用 retriever 检索业务上下文，然后用 prompt + model 生成 SQL 或工具调用，最后执行 tool 并用 parser 格式化结果。如果流程是线性的，用 LCEL；如果涉及失败重试、审批、循环修正，就用 LangGraph。

5.2 为什么不用普通 chain，而要用状态图？

普通 chain 适合固定流水线，但真实 Agent 经常要根据中间结果决定下一步，比如 SQL 执行失败要修正，资料不足要重新检索，危险操作要人工确认。LangGraph 用 State 保存过程数据，用 Node 拆步骤，用 Conditional Edge 路由，用 Checkpoint 恢复流程，所以更适合长任务和可控 Agent。

5.3 DeepAgents 解决什么问题？

DeepAgents 是 LangGraph 之上的 agent harness。它不是只提供一个图，而是内置 planning、todo、文件系统、子 Agent、长期记忆、权限、人类审批和上下文压缩。适合把 Agent 从 demo 做成能处理复杂项目的工程化系统。

6. 你的项目可以怎么包装

你简历里的 BI 智能问数 agent 很适合包装成一个完整例子：

React 聊天界面 → SSE 展示 Agent 步骤 → FastAPI/Nest API → LangGraph 编排：抽意图 / 检索元数据 / 生成 SQL / 校验 SQL / 执行 / 总结 → LangChain 组件：PromptTemplate / Model / Tool / OutputParser / Retriever → MySQL + Qdrant/Milvus + Elasticsearch → LangSmith 日志与 trace → Redis/PostgreSQL 保存会话和状态

2 分钟项目讲法

“这个项目是一个 BI 自然语言问数助手。用户用中文提问，系统先做意图抽取，再检索表结构、字段别名、指标口径和字段取值，然后生成 SQL，执行前做校验和修正，最后查询数据库并通过 SSE 把执行步骤和结果流式返回给前端。LangChain 负责 prompt、模型、结构化输出、工具和 RAG 组件；LangGraph 负责编排多节点流程和错误修正；如果要进一步工程化，可以用 DeepAgents 的 todo、文件系统、子 Agent、权限和长期记忆能力增强长任务可靠性。”
