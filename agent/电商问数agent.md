# 电商问数 Agent 面试准备文档

这份文档按“模拟面试官追问”的方式整理，目标不是背技术名词，而是让你能说出真实做过项目的人才会关心的细节：为什么这么设计、链路怎么跑、遇到什么不稳定点、怎么排查、后续怎么扩展。

## 一句话项目介绍

这是一个面向电商数仓场景的自然语言问数 Agent。用户输入“统计华北地区的销售总额”这类问题后，系统不会直接让大模型生成 SQL，而是先基于元数据知识库召回相关字段、指标和字段取值，再由 LangGraph 编排关键词抽取、三路召回、候选表/指标过滤、上下文补全、SQL 生成、SQL 校验、SQL 修正和 SQL 执行，最后通过 SSE 把每个节点进度和查询结果流式返回前端。

可以这样讲项目亮点：

- 不是简单 NL2SQL，而是“检索增强 + 多阶段推理 + SQL 校验执行”的完整问数链路。
- 字段和指标走 Qdrant 向量检索，字段真实取值走 Elasticsearch 全文检索，权威元数据保存在 MySQL。
- LangGraph 把每个步骤拆成节点，State 传业务中间结果，Context 注入 Repository、Embedding、数据库等外部依赖。
- SQL 不是生成完直接执行，而是先用数据库 `EXPLAIN` 做校验，失败后带着错误信息进入修正节点。
- 后端用 FastAPI + StreamingResponse 返回 SSE，前端可以实时看到“抽取关键词、召回字段、生成 SQL、执行 SQL”等过程。

## 项目真实细节总览

### 1. 业务背景

电商业务问数里，业务人员通常不会写 SQL，数据同学也不可能记住全部表结构、字段含义、指标口径和枚举值。直接把问题丢给大模型会有几个典型问题：

- 表选错：比如把地区字段理解成客户地址字段。
- 字段选错：比如“销售额”到底是 `order_amount` 还是其他金额字段。
- 指标口径错：比如 GMV 应该用订单金额总和，而不是订单数量或平均金额。
- 条件值写错：比如用户说“华北地区”，数据库真实值可能在 `dim_region.region_name` 里。
- SQL 幻觉：编不存在的表、字段、函数。

所以这个项目把自然语言问数拆成两条主线：

- 离线主线：构建元数据知识库。
- 在线主线：基于用户问题召回上下文并生成、校验、执行 SQL。

### 2. 离线元数据知识库构建

入口是 `app/scripts/build_meta_knowledge.py`，核心服务在 `app/services/meta_knowledge_service.py`。

构建过程不是只把表结构写入数据库，而是同时构建三类知识：

- Meta MySQL：保存表、字段、指标、字段和指标的关联关系。
- Qdrant：保存字段和指标的向量索引。
- Elasticsearch：保存字段真实取值全文索引。

具体细节：

- 表和字段来自 `conf/meta_config.yaml`。比如 `dim_region` 是地区维表，包含 `province`、`region_name`、`country`；`fact_order` 是订单事实表，包含 `order_quantity` 和 `order_amount`。
- 字段类型不靠配置猜，而是从真实 DW MySQL 里 `show columns from table_name` 读取。
- 字段示例值通过 `select distinct column from table limit 10` 抽样，用于后续给模型理解字段含义。
- 字段向量不是一个字段一个 point，而是把字段名、字段描述、字段别名都拆成多个向量入口。例如 `order_amount`、`订单金额`、`销售额`、`收入` 都能召回同一个字段。
- 指标也是类似处理，指标名、描述、别名都会进入 Qdrant。比如 GMV 的描述是“所有订单成交金额总和”，依赖字段是 `fact_order.order_amount`。
- 字段真实取值只同步配置中 `sync: true` 的字段，避免把订单 ID、外键、事实度量这种无意义或高基数字段全部写入 ES。
- ES 的 `value` 字段使用 IK 中文分词，适合检索“华北”“会员等级”“商品品类”这类中文值。

面试里可以强调：

“我这里做了一个区分：字段/指标属于语义概念，所以适合向量召回；字段取值更像真实文本匹配，所以走 ES 全文检索。这样比全都塞到向量库里更清晰，也更容易调参和排查。”

### 3. 在线问数链路

在线链路在 `app/agent/graph.py` 里用 LangGraph 编排：

```text
START
  -> extract_keywords
  -> recall_column / recall_value / recall_metric
  -> merge_retrieved_info
  -> filter_table / filter_metric
  -> add_extra_context
  -> generate_sql
  -> validate_sql
  -> run_sql 或 correct_sql
  -> END
```

关键设计：

- `extract_keywords` 用 jieba 的 TF-IDF 关键词抽取，并限制词性，只保留名词、地名、动词、英文、固定短语等更有业务含义的词。
- 抽取结果会额外保留原始问题作为兜底检索词，避免分词错误导致语义丢失。
- `recall_column`、`recall_metric` 会先用 LLM 做查询词扩展，再调用 Embedding，最后去 Qdrant 检索。
- `recall_value` 也会先用 LLM 扩展字段值相关关键词，但检索走 ES。
- 三路召回后通过 `merge_retrieved_info` 合并成 SQL 生成需要的 `table_infos` 和 `metric_infos`。
- `filter_table` 和 `filter_metric` 让模型只做“选择题”，不让模型重写复杂结构。模型返回要保留哪些表、字段、指标，程序再基于原始结构裁剪。
- `add_extra_context` 会补日期信息和数据库方言/版本，支持“本季度”“最近 N 天”这类问题，也让 SQL 生成符合 MySQL 版本。
- `generate_sql` 只输出一条纯 SQL，提示词明确禁止 Markdown 代码块、禁止编造表字段、禁止写操作。
- `validate_sql` 用真实数据库 `EXPLAIN` 提前解析 SQL。
- 如果校验失败，`correct_sql` 会带着原问题、原 SQL、错误信息和完整上下文做最小修正。
- 最终 `run_sql` 执行 SQL，并把结果通过 stream writer 输出给 SSE。

## 模拟面试问答

### Q1：你这个项目是做什么的？解决什么问题？

回答：

这个项目是一个电商场景的自然语言问数 Agent。它解决的是业务人员不会写 SQL，但又需要查询销售额、订单量、地区分布、商品品类表现等数据的问题。

我没有采用“大模型直接生成 SQL”的方式，因为真实问数很依赖表结构、字段语义、指标口径和枚举值。如果上下文不够，大模型很容易编字段、选错表、错用指标。所以我把链路拆成两部分：

第一部分是离线构建元数据知识库，把表、字段、指标、字段取值分别写入 MySQL、Qdrant 和 Elasticsearch。

第二部分是在线问数，用户输入问题后，LangGraph 会先抽关键词，再分别召回字段、指标和字段取值，合并成结构化上下文，再生成 SQL、校验 SQL、必要时修正 SQL，最后执行查询并通过 SSE 流式返回。

如果面试官继续追问“举个例子”：

比如用户问“统计华北地区的销售总额”。系统会抽出“华北”“销售总额”等关键词；字段召回可能命中 `fact_order.order_amount` 和地区维表字段；指标召回会命中 GMV；字段取值召回会在 ES 里找到 `dim_region.region_name = 华北`。最后 SQL 生成时就能基于这些上下文生成事实表和地区维表的 join，并用 `SUM(fact_order.order_amount)` 计算 GMV。

### Q2：为什么不直接让大模型根据表结构生成 SQL？

回答：

主要有三个原因。

第一，真实数仓表和字段很多，不可能把所有表结构都塞进 Prompt。上下文太长会增加成本，也会降低模型关注重点。

第二，业务问题里的表达和数据库字段经常不一致。比如用户说“销售额”，数据库字段可能叫 `order_amount`；用户说“华北”，它不是字段名，而是某个维表字段里的真实取值。

第三，指标口径需要被约束。比如 GMV 应该基于 `fact_order.order_amount` 求和，而不是模型凭经验自己解释。

所以我做的是先召回再生成：先用 Qdrant 召回字段和指标，用 ES 召回字段值，再把少量高相关上下文交给模型。这样模型不是开放式猜 SQL，而是在给定表、字段、指标、样例值范围内生成 SQL。

### Q3：为什么字段和指标用 Qdrant，字段取值用 Elasticsearch？

回答：

字段和指标更偏语义匹配。比如“销售总额”“成交额”“GMV”这些表达不一定字面相同，但语义接近，所以向量检索更适合。

字段取值更偏真实文本匹配。比如“华北”“钻石会员”“手机配件”这些值通常要和数据库里的枚举值对齐。这里用 ES 的全文检索，尤其是配合 IK 中文分词，可以更好地处理中文短语、地区名、品类名。

而且把两类检索分开还有工程上的好处：字段/指标召回看 Qdrant 的相似度阈值，字段值召回看 ES 的 match 和 min_score，排查问题时更容易定位是哪一路召回出了问题。

### Q4：元数据知识库是怎么构建的？

回答：

构建入口读取 `conf/meta_config.yaml`。这个配置里定义了表、字段、字段角色、字段描述、别名、是否同步真实取值，以及指标定义。

构建时我做了几件事：

1. 先把表信息和字段信息写入 Meta MySQL。字段类型不是配置里手写，而是从数仓 MySQL 查询真实类型。
2. 对每个字段抽一些 distinct 示例值，写入字段元数据的 examples，方便模型理解这个字段可能有哪些值。
3. 字段向量入 Qdrant 时，不只是字段名入库，还会把字段描述和别名都作为 embedding text 建 point，但 payload 都指向同一个 `ColumnInfo`。
4. 对配置里 `sync: true` 的字段，把真实取值同步到 ES。像地区、品类、品牌、会员等级这些适合同步；订单 ID、外键、金额字段不适合。
5. 指标信息写入 Meta MySQL，同时写入指标和字段的关联表。比如 GMV 依赖 `fact_order.order_amount`。
6. 指标的名称、描述、别名同样写入 Qdrant，方便用户用“成交额”“销售总额”等说法召回同一个指标。

可以补充一个细节：

为了避免重复构建时产生脏数据，Meta MySQL 写入用了 MySQL 的 `insert ... on duplicate key update`，ES 文档用稳定 id 覆盖，Qdrant 使用 upsert。

### Q5：LangGraph 的 State 和 Context 分别放什么？

回答：

State 放一次图执行过程中会被节点读写的业务中间结果，比如用户 query、关键词、召回到的字段/指标/取值、合并后的表上下文、指标上下文、日期信息、数据库信息、生成的 SQL 和校验错误。

Context 放外部依赖，比如 Qdrant Repository、ES Repository、Meta MySQL Repository、DW MySQL Repository、Embedding Client。这些对象不适合放 State，因为它们不是业务状态，也不需要被图合并或序列化。

这样拆的好处是节点职责比较清楚。节点从 State 读业务输入，从 Context 取工具依赖，然后把新的业务结果写回 State。

### Q6：关键词抽取怎么做？为什么还要保留原始问题？

回答：

关键词抽取节点用的是 jieba 的 `extract_tags`，基于 TF-IDF 抽关键词，并通过 `allowPOS` 限制词性。比如保留名词、地名、机构名、动词、形容词、英文、固定短语等，过滤掉“帮我”“一下”“的”这种低价值词。

但中文分词有不稳定性，比如“销售总额”可能被拆开，或者业务词没有进入词库。所以我额外把原始 query 也放进关键词列表里作为兜底。后续字段、指标、字段值召回都会使用这批关键词。

面试里可以说：

“关键词抽取不是为了完全理解问题，而是为了给召回链路提供多个入口。真正的语义扩展还会在字段召回、指标召回和字段值召回节点里通过 LLM 再做一轮。”

### Q7：字段召回具体怎么做？

回答：

字段召回节点会拿到原始问题和关键词。它先调用一个 Prompt，让 LLM 根据用户问题扩展出更贴近字段语义的关键词。比如用户说“销售总额”，扩展词可能包括“销售额”“订单金额”“成交金额”。

然后把原始关键词和扩展词合并去重。每个关键词调用 Embedding API 得到向量，再去 Qdrant 的字段集合里检索。Qdrant 返回的 payload 会被还原成 `ColumnInfo`。由于同一个字段可能因为字段名、描述、别名多个 point 被命中，所以我用 `column_info.id` 做去重。

最后写回 `retrieved_column_infos`。

### Q8：指标召回和字段召回有什么不同？

回答：

流程类似，都是“关键词扩展 -> Embedding -> Qdrant 检索 -> 去重”。不同点在于召回对象是业务指标。

字段解决的是“SQL 要用哪些数据库字段”，指标解决的是“业务口径应该怎么算”。比如用户问“销售总额”，字段召回能找到 `order_amount`，但指标召回能告诉模型这其实对应 GMV，GMV 的定义是所有订单成交金额总和，依赖 `fact_order.order_amount`。

后续 `merge_retrieved_info` 会根据指标的 `relevant_columns` 把依赖字段补齐，避免指标召回到了，但生成 SQL 时缺少对应字段上下文。

### Q9：字段取值召回解决什么问题？

回答：

字段取值召回解决的是“用户问题里的业务值到底对应哪个字段”的问题。

比如用户问“华北地区销售额”，`华北` 不是字段名，而是 `dim_region.region_name` 里的一个取值。如果没有字段值召回，模型可能知道要按地区过滤，但不知道真实条件应该写在哪个字段上。

实现上，字段取值召回也会先用 LLM 扩展查询词，比如“华北地区”扩展成“华北”。然后用这些词去 ES 的 `value_index` 里查真实值，返回 `ValueInfo`，其中包含 value 和 column_id。后续合并节点会根据 column_id 把这个 value 加到对应字段的 examples 里。

### Q10：三路召回后怎么合并？

回答：

合并节点是整个链路里比较关键的地方。它不是简单把三个列表拼起来，而是按 SQL 生成需要的结构重新组织。

主要做了几步：

1. 先把字段召回结果按 `column_id` 建 map 去重。
2. 遍历召回到的指标，如果指标依赖字段不在字段 map 里，就从 Meta MySQL 里查出来补进去。
3. 遍历字段取值召回结果，如果对应字段没在字段 map 里，也从 Meta MySQL 补字段信息；然后把真实 value 加到字段 examples 里。
4. 按 `table_id` 把字段分组，形成表到字段的映射。
5. 对每张候选表补齐主键和外键字段。因为用户一般不会问“region_id”，但 join 必须依赖这些字段。
6. 查表元数据，把分散字段组装成提示词需要的 `table_infos`。
7. 把召回到的指标整理成 `metric_infos`，保留指标名、描述、依赖字段和别名。

可以强调：

“这里补主外键是一个很实际的细节。很多 NL2SQL 系统只召回用户显式提到的字段，结果生成多表 SQL 时缺 join key。我的做法是只要表进候选集，就兜底补它的 primary_key 和 foreign_key。”

### Q11：为什么还要做表过滤和指标过滤？

回答：

召回阶段为了保证召回率，通常会多召回一些候选信息。如果直接把所有候选表和指标都给 SQL 生成模型，容易带来噪声，导致模型选错表或生成复杂 SQL。

所以我在合并后又做了过滤。过滤节点让 LLM 做的是“选择题”：给它候选表结构和用户问题，让它返回应该保留哪些表、哪些字段；指标过滤也类似，只返回要保留的指标名。

然后程序根据模型返回结果裁剪原始结构。这样有两个好处：

- 模型不用重写复杂嵌套结构，降低格式错误概率。
- 真正进入 SQL 生成节点的上下文更干净。

### Q12：SQL 生成提示词怎么约束模型？

回答：

SQL 生成 Prompt 里有几个关键约束：

- 只能使用上下文里真实存在的表和字段，禁止编造。
- 如果指标信息里有相关定义，必须遵循业务口径。
- 只能生成查询 SQL，不能有 insert、update、delete、create 等写操作。
- 只能生成一条 SQL。
- SQL 语法必须符合数据库环境里的 dialect 和 version。
- 聚合字段和函数表达式必须有稳定别名，避免前端拿到 `SUM(...)` 这种不好展示的列名。
- 输出必须是纯 SQL，不能带 Markdown 代码块。

而且输入上下文是 YAML 格式。因为表结构、字段列表、指标列表是嵌套结构，YAML 比长 JSON 更适合人和模型阅读，也能保留中文描述。

### Q13：SQL 校验和修正怎么做？

回答：

生成 SQL 后不是直接执行，而是先进入 `validate_sql` 节点。这个节点会调用 DW Repository 的 `validate` 方法，内部用 `EXPLAIN <sql>` 让数据库提前解析 SQL。

如果数据库能解析，说明至少语法、表名、字段名大概率没问题，就进入执行节点。

如果 `EXPLAIN` 报错，节点不会直接抛异常终止流程，而是把错误信息写入 `state["error"]`。LangGraph 的条件边会根据 `error` 是否为空决定走 `run_sql` 还是 `correct_sql`。

`correct_sql` 会把原问题、原 SQL、数据库错误、表结构、指标、日期、数据库环境一起给模型，让模型做最小必要修正。修正 Prompt 明确要求保持原始业务语义不变，只修导致 SQL 无法执行的问题。

这个设计的核心是：

“把数据库当成最终裁判，模型只负责生成和修正，SQL 能不能被数据库接受要由真实数据库验证。”

### Q14：SSE 流式返回是怎么实现的？

回答：

API 层是 `POST /api/query`，路由返回 `StreamingResponse`，media type 是 `text/event-stream`。

`QueryService.query` 里调用 `graph.astream(input=state, context=context, stream_mode="custom")`。每个 LangGraph 节点内部通过 `runtime.stream_writer` 写进度事件，比如：

```json
{"type": "progress", "step": "召回字段信息", "status": "running"}
```

节点成功或失败也会写对应状态。最终执行 SQL 后，`run_sql` 会写：

```json
{"type": "result", "data": [...]}
```

`QueryService` 把每个 chunk 包装成 SSE 格式：

```text
data: {...}\n\n
```

如果图执行过程中出异常，由于 SSE 响应可能已经开始返回，HTTP 状态码不适合再改，所以会把异常包装成一条 `type=error` 的 SSE 消息返回。

### Q15：后端依赖和连接资源怎么管理？

回答：

外部客户端在 FastAPI lifespan 里初始化和关闭，包括 Qdrant、Embedding、ES、Meta MySQL、DW MySQL。这些是应用级资源，不应该每个请求重复创建。

请求级的 MySQL session 通过 FastAPI Depends 创建，使用 async context manager 保证请求结束后释放。

Repository 在依赖函数里组装，QueryService 只接收已经创建好的 Repository 和 Client。这样路由层不直接感知底层基础设施，业务服务也不负责创建连接。

可以总结成：

“应用级长生命周期资源放 lifespan，请求级事务资源放 Depends，业务逻辑只依赖抽象后的 Repository。”

### Q16：日志和并发请求怎么区分？

回答：

项目里用了 `ContextVar` 保存 request_id。FastAPI middleware 在每个请求进来时生成 uuid，并写入 `request_id_ctx_var`。

loguru logger 通过 patch 函数在每条日志里注入当前 request_id。因为 ContextVar 是协程上下文隔离的，所以并发请求不会像普通全局变量那样互相覆盖。

这对于排查流式 Agent 链路很有用。因为一次请求会经过很多节点，每个节点都会打日志，有 request_id 后可以把同一次问数链路串起来。

### Q17：如果用户问题是“统计华北地区的销售总额”，链路怎么跑？

回答：

可以按链路一步步说：

1. `extract_keywords` 抽取“华北”“地区”“销售总额”等关键词，同时保留完整问题。
2. `recall_column` 根据问题扩展字段相关词，比如“订单金额”“销售额”，然后通过 Qdrant 召回 `fact_order.order_amount`，也可能召回地区相关字段。
3. `recall_metric` 扩展指标相关词，召回 GMV。GMV 的 relevant_columns 包含 `fact_order.order_amount`。
4. `recall_value` 在 ES 中检索“华北”，命中 `dim_region.region_name` 的取值。
5. `merge_retrieved_info` 把字段、指标依赖字段、字段取值合并起来，并补齐 `fact_order.region_id`、`dim_region.region_id` 这类 join key。
6. `filter_table` 可能保留 `fact_order` 和 `dim_region`，过滤掉无关字段。
7. `filter_metric` 保留 GMV。
8. `add_extra_context` 补当前日期和 MySQL 版本。
9. `generate_sql` 生成类似按地区维表 join 事实表，并对 `order_amount` 求和的 SQL。
10. `validate_sql` 用 `EXPLAIN` 校验。
11. 校验通过就执行，返回结果；失败就进入 `correct_sql` 修正后再执行。

### Q18：这个项目里哪些点最能说明你不是只调 API？

回答：

我会挑四个点讲。

第一，做了离线元数据知识库构建。表字段、指标和取值分别进入不同存储，不是运行时临时拼 Prompt。

第二，在线链路有多路召回。字段和指标用向量，字段值用全文检索，三者合并后再给模型。

第三，SQL 生成后有校验修正闭环。不是模型输出就信，而是让真实数据库用 `EXPLAIN` 验证。

第四，工程结构做了分层。LangGraph 节点只负责流程步骤，Repository 封装 MySQL/Qdrant/ES 访问，Service 组织业务编排，FastAPI 负责接口和依赖注入。

### Q19：这个系统有什么不足？如果继续做你会怎么优化？

回答：

我会比较坦诚地说几个方向。

第一，目前 SQL 校验只用 `EXPLAIN`，能发现语法、表字段、部分执行计划问题，但不能保证业务语义一定正确。后续可以加 SQL AST 检查，比如只允许 select、限制扫描范围、禁止笛卡尔积、检查 join 条件。

第二，当前修正链路只修一次。如果修正后 SQL 仍然失败，当前会在执行阶段报错。后续可以把 validate 和 correct 做成带最大次数的循环，比如最多修正两次，并记录每次错误。

第三，指标口径目前来自配置，适合教学场景。真实企业可以接入指标平台，增加指标版本、口径负责人、时间口径、过滤条件等。

第四，字段和指标召回现在是逐个关键词调用 embedding，量大时会有性能压力。可以做批量 embedding、缓存 query embedding、或者把 LLM 扩展词数量限制得更严格。

第五，安全上还可以加强。比如 SQL 执行前做只读 AST 校验、加查询超时、行数限制、租户权限过滤、字段脱敏。

第六，结果展示现在偏表格，后续可以根据结果维度自动生成图表建议。

### Q20：如果面试官问你做项目时遇到的难点，你怎么回答？

回答：

我会说最大难点不是调用大模型，而是如何把模型限制在可信上下文里。

一开始如果直接把用户问题给模型，它确实能生成看起来像 SQL 的东西，但经常会编不存在的字段，或者把“销售额”理解错。后面我把问题拆成几步：先构建元数据知识库，再用字段、指标、字段值三路召回缩小上下文，最后在 Prompt 里明确只能使用召回结果里的表字段。

另一个难点是多表 join。用户一般不会提主键外键，如果只召回业务字段，模型可能不知道怎么 join。所以我在合并召回结果时，只要某张表进入候选集，就从 Meta MySQL 补齐这张表的主键和外键字段。

还有一个难点是字段值。比如“华北”这种词，它既不是字段名也不是指标，但 SQL 里必须变成 where 条件。这个问题我用 ES 字段值索引解决，检索到 value 后把它合并进对应字段的 examples。

## 深挖追问速记

### 追问：为什么字段向量一个字段要建多个 point？

回答：

因为用户表达可能命中字段名、描述或别名中的任意一个。比如 `order_amount` 这个字段，用户可能说“销售额”“收入”“订单金额”。如果只用字段名做 embedding，中文业务表达和英文字段名的相似度不一定稳定。所以我把字段名、描述、别名都建成独立 point，但 payload 都是同一个字段实体，召回后再按字段 id 去重。

### 追问：为什么字段值不全部同步？

回答：

因为不是所有字段值都有检索价值。像订单 ID、客户 ID、外键、金额这类字段，要么基数太高，要么不会作为自然语言条件出现，同步进 ES 会浪费空间，还会增加噪声。所以配置里用 `sync` 控制，只同步地区、品类、品牌、会员等级、客户名等适合被用户直接提到的字段。

### 追问：为什么过滤节点让模型只返回选择结果？

回答：

因为让模型重写完整表结构风险比较高，容易丢字段属性或者产生格式错误。让它只返回“保留哪些表和字段”，程序再基于原始结构裁剪，是更稳定的做法。模型做语义判断，程序做结构处理。

### 追问：如果 Qdrant 召回了无关字段怎么办？

回答：

第一层靠 score threshold 和 limit 控制基础相关性。

第二层靠后面的 `filter_table` 再过滤候选表字段。

第三层靠 SQL 生成 Prompt 限制只能使用过滤后的上下文。

如果要进一步优化，可以记录线上问题和召回结果，调整字段描述、别名、embedding 文本、阈值，也可以引入 rerank 模型。

### 追问：如果没召回到正确字段怎么办？

回答：

现在有几个兜底：

- 原始问题会作为关键词进入召回。
- LLM 会针对字段、指标、字段值分别扩展关键词。
- 指标召回后会补齐指标依赖字段。
- 字段值召回后会按 column_id 补齐字段。

如果仍然召回不到，后续可以加表级召回、低阈值召回加 rerank、或者在 Prompt 中让模型识别上下文不足并返回澄清问题，而不是强行生成 SQL。

### 追问：SQL 校验为什么用 EXPLAIN，不直接执行？

回答：

直接执行可能产生较重的查询开销，甚至在错误 SQL 上浪费资源。`EXPLAIN` 能让数据库先解析 SQL，检查表名、字段名、语法和执行计划。它不能保证结果语义完全正确，但作为执行前校验成本更低，也更安全。

### 追问：怎样防止模型生成写操作？

回答：

Prompt 里明确要求只能生成查询 SQL，但这只是第一层。更可靠的做法是执行前做 SQL 解析或白名单校验，只允许 `SELECT`，禁止 `INSERT/UPDATE/DELETE/CREATE/DROP`，并限制多语句。当前项目 Prompt 已经约束了只读，后续工程化要加 AST 级校验。

### 追问：为什么要传数据库版本和方言？

回答：

不同数据库的日期函数、limit 写法、字符串函数、时间计算都有差异。比如 MySQL、PostgreSQL、ClickHouse 的语法不一样。把 dialect 和 version 放进上下文，可以减少模型生成不兼容 SQL 的概率。

### 追问：为什么前端要用 SSE？

回答：

Agent 链路不是一次普通请求，可能要经历多个节点和多次外部服务调用。如果只等最终结果，用户会感觉卡住，也不利于排查。SSE 适合服务端持续推送进度：每个节点开始、成功、失败都可以返回，最后再返回 result。

相比 WebSocket，这里主要是服务端单向推送，不需要复杂双向通信，所以 SSE 更轻。

### 追问：Repository 分层有什么实际意义？

回答：

节点和 Service 不直接写 SQL 或调用底层 SDK，而是通过 Repository。这样好处是：

- MySQL、Qdrant、ES 的访问逻辑集中管理。
- 节点逻辑更关注业务步骤，代码更短。
- 后续替换存储或调整查询参数时，不需要改 LangGraph 节点。
- 测试时可以 mock Repository，而不是连真实外部服务。

### 追问：如果线上响应慢，你怎么排查？

回答：

我会按节点拆耗时，因为每个节点都有 progress 事件和日志。

优先看几个外部调用点：

- LLM 关键词扩展是否慢。
- Embedding 是否逐个关键词调用导致慢。
- Qdrant 检索耗时和返回数量。
- ES match 查询耗时。
- SQL `EXPLAIN` 和最终查询耗时。

优化手段包括：

- 对 embedding query 做缓存。
- 批量调用 embedding。
- 控制 LLM 扩展词数量。
- 对常见问题做召回结果缓存。
- 对 SQL 执行加超时和 limit。
- 调整 Qdrant/ES 的阈值和 limit。

### 追问：如果指标口径错了，问题可能出在哪里？

回答：

可能有几个位置：

- `meta_config.yaml` 里的指标定义不完整，比如 relevant_columns 写错。
- 指标别名不足，导致指标没召回到。
- 指标召回到了，但 filter_metric 过滤掉了。
- Prompt 没有强约束模型遵循指标口径。
- SQL 生成时用了字段但没有按照指标定义聚合。

排查时我会先看 LangGraph 流程日志，确认 `retrieved_metric_infos` 和过滤后的 `metric_infos`，再看最终 Prompt 里的指标上下文，最后看生成 SQL 是否遵守 relevant_columns。

## 最后背诵版

这个项目我最想突出的是：它不是一个“Prompt 生成 SQL”的 demo，而是一个围绕电商数仓问数场景做的 Agent 工程链路。

离线阶段，我把表字段、指标和字段取值构造成元数据知识库。结构化元数据放 MySQL，字段和指标放 Qdrant 做语义检索，字段真实取值放 ES 做全文检索。

在线阶段，我用 LangGraph 编排多个节点。先抽关键词，然后字段、指标、字段值三路召回；合并时补齐指标依赖字段、字段真实值和主外键；再过滤表和指标；最后生成 SQL。SQL 会先用数据库 `EXPLAIN` 校验，失败就带错误信息修正，成功才执行。结果和节点进度通过 SSE 流式返回。

这个设计解决了直接 NL2SQL 容易出现的表字段幻觉、指标口径错误、字段值不匹配和 SQL 不可执行问题。它的关键不是把所有事情交给模型，而是让模型在检索到的、可信的、结构化上下文里完成有限任务。
