# 第一部分：Node 运行时与底层模型

## 1. Node.js 到底是什么？

**参考答案：**

Node.js 不是一门语言，而是一个 **JavaScript 运行时**。它把 V8 引擎、libuv、Node 自己的核心模块（如 `fs`、`http`、`stream`、`events`）组合起来，让 JavaScript 能跑在服务端。

**你可以继续展开：**

- 浏览器里 JS 主要处理页面与交互；Node 里 JS 主要处理文件、网络、进程、流、服务端逻辑。
- Node 的强项不是“计算特别快”，而是 **I/O 密集型场景下的高并发处理能力**。
- 真正让 Node 擅长高并发的，不是“单线程”这四个字本身，而是 **事件循环 + 非阻塞 I/O + 合理的异步模型**。

**常见误区：**

- 误区 1：Node 就是单线程。  
  更准确地说：**执行 JS 代码的主线程默认是单线程**，但 Node 运行时并不只有一个线程。
- 误区 2：Node 适合所有后端业务。  
  Node 更适合 I/O 密集场景；纯 CPU 密集型任务不一定占优。

---

## 2. 为什么大家总说 Node “单线程”，但它又能处理高并发？

**参考答案：**

因为 Node 默认只有一个主线程执行 JavaScript，但很多 I/O 操作并不是这条主线程硬做完的。Node 会把底层 I/O 交给操作系统或 libuv 去协调，等结果好了再把回调放回事件循环执行。

**原理拆开讲：**

- **JS 主线程**：执行你的同步代码、回调、Promise 后续逻辑。
- **操作系统 / libuv**：负责监听网络、文件、定时器等异步事件。
- **事件循环**：持续取出“已经准备好”的任务，交给 JS 主线程执行。

所以 Node 的高并发，本质上是：

- 主线程尽量不阻塞；
- 大量等待 I/O 的时间不浪费在“卡住线程”上；
- 一个进程可以高效服务很多连接。

**面试时可加一句：**

> Node 不是靠“同时算很多事”取胜，而是靠“自己少等、少阻塞、把等待交给系统”。

---

## 3. Node 的事件循环（Event Loop）是什么？

**参考答案：**

事件循环可以理解成 Node 的任务调度器。它不断检查当前有哪些异步任务已经完成，然后按阶段把对应回调取出来执行。

**关键原理：**

Node 事件循环常见阶段包括：

- `timers`
- `pending callbacks`
- `poll`
- `check`
- `close callbacks`

其中你面试最常讲的是：

- `setTimeout/setInterval` 在 `timers`
- I/O 回调很多在 `poll`
- `setImmediate` 在 `check`

**重要理解：**

- 定时器不是“到了时间立刻执行”，而是“**最早可以在那个时间之后执行**”。
- 如果前面有耗时同步代码，后面的 timer 一样会被拖延。
- 从 **Node.js 20（libuv 1.45.0）** 开始，timer 的执行时机相对旧版本有调整，这会影响 `setImmediate()` 和 timer 的先后表现。

**常见误区：**

- 误区：背阶段顺序就等于理解事件循环。  
  真正重要的是：**同步代码会阻塞后续调度；异步任务要等主线程空出来才能执行。**

---

## 4. `process.nextTick()`、Promise 微任务、`setImmediate()`、`setTimeout(fn, 0)` 有什么区别？

**参考答案：**

它们都能“延后执行”，但优先级和所处队列不同。

**可以这样理解：**

1. 当前同步代码先执行完
2. `process.nextTick()` 先清空
3. Promise 微任务再清空
4. 事件循环进入后续阶段
5. `setImmediate()` / `setTimeout()` 按各自阶段执行

**面试要点：**

- `process.nextTick()` 不属于普通事件循环阶段，它会在当前操作结束后立即执行，优先级很高。
- 如果递归塞太多 `process.nextTick()`，会导致 **I/O 饥饿**，因为事件循环一直走不到 `poll`。
- `setImmediate()` 适合“本轮 I/O 之后尽快执行”。
- `setTimeout(fn, 0)` 并不是“马上执行”，只是最短等待时间到了以后再参与调度。

**追问答法：为什么有时 `setImmediate` 比 `setTimeout(...,0)` 先？**

因为它们不在一个阶段。特别是在 I/O 回调里注册时，`setImmediate()` 往往会先于 `setTimeout(..., 0)` 执行。

---

## 5. Node 里的异步 I/O 和线程池是什么关系？

**参考答案：**

不是所有异步操作都走线程池。

**要分两类：**

1. **操作系统本身就很擅长异步处理的 I/O**  
   比如很多网络 I/O，更多是事件通知模型，不一定依赖线程池。
2. **底层需要额外线程帮忙做的任务**  
   比如很多文件系统 API、部分 `crypto`、`zlib`、`dns.lookup()` 等，会使用 libuv 的 worker pool。

**这题的意义：**

很多人一说异步就以为“开了个新线程”，这是不准确的。Node 的异步是一个更宽的概念，底层可能是：

- OS 事件通知
- libuv 线程池
- 子进程
- Worker Threads

**一句话总结：**

> 异步不等于多线程；异步是调用方式，多线程只是某些底层实现手段。

---

## 6. `worker_threads` 是干什么的？什么时候该用？

**参考答案：**

`worker_threads` 让 Node 能真正并行执行 JavaScript 代码，适合 CPU 密集型任务，比如图像处理、复杂计算、压缩、AST 分析、大数据转换等。

**原理：**

- 主线程继续处理事件循环和 I/O。
- 重 CPU 的计算交给 Worker Thread。
- 线程之间可以通过消息通信，也可以共享 `SharedArrayBuffer`。

**什么时候适合：**

- 大 JSON 解析/转换
- 图片缩放、加密、压缩
- 服务端模板/代码编译
- CPU 算法任务

**什么时候不适合：**

- 只是普通数据库查询、HTTP 请求、读文件  
  这些本身就更适合 Node 的异步 I/O，不该滥用 worker。

**面试时要讲出的判断：**

> Worker Threads 主要解决“CPU 把主线程卡死”的问题，不是拿来替代普通异步 I/O 的。

---

## 7. `child_process` 和 `worker_threads` 有什么区别？

**参考答案：**

二者都能把任务分出去，但粒度和成本不同。

### `worker_threads`
- 同一个进程内的多个线程
- 更适合 CPU 密集型 JS 逻辑
- 可以共享内存
- 创建成本相对低

### `child_process`
- 直接拉起一个新进程
- 进程隔离更强
- 适合调用系统命令、运行其他语言程序、做进程级隔离
- 成本通常比 worker 更高

**常见追问：`spawn`、`exec`、`execFile`、`fork` 区别？**

- `spawn`：流式拿输出，适合长任务、大输出
- `exec`：通过 shell 执行，输出会缓冲到内存，适合短命令
- `execFile`：直接执行文件，不经过 shell，更安全一些
- `fork`：专门用于拉起新的 Node 进程，并自带 IPC 通道

# 第二部分：模块系统与 package.json

## 8. CommonJS 和 ES Module（ESM）有什么区别？

**参考答案：**

这是 Node 里最常见的模块题。

### CommonJS
- `require()` / `module.exports`
- 加载更偏运行时
- 历史更早，Node 生态里很多老包都基于它

### ESM
- `import` / `export`
- 语言标准模块系统
- 更适合静态分析、tree-shaking、跨运行时一致性

**你要会说出的几个细节：**

- Node 现在同时支持 CommonJS 和 ESM。
- `require()` 走 CommonJS loader；`import` / `import()` 走 ESM loader。
- ESM 里默认没有 `__filename`、`__dirname`、`require`、`module.exports` 这些 CommonJS 变量。

**面试建议：**

如果是新项目，尤其是偏现代工具链、跨端、库开发，通常更倾向 ESM；但做老项目维护时 CommonJS 仍很常见。

---

## 9. `package.json` 里的 `type` 字段有什么作用？

**参考答案：**

`type` 决定 Node 如何解释当前包内的 `.js` 文件。

- `"type": "module"`：`.js` 按 ESM 解析
- `"type": "commonjs"`：`.js` 按 CommonJS 解析
- 没写时：默认更偏 CommonJS

**为什么这题重要：**

因为很多“为什么 import 不能用”“为什么 require 报错”“为什么这个文件被当成 ESM”问题，本质都跟模块类型判定有关。

**面试时一句话讲透：**

> Node 不是只看语法，还会结合文件扩展名和最近的 `package.json` 来决定模块系统。

---

## 10. `package.json` 里的 `exports` 有什么用？为什么现在越来越重要？

**参考答案：**

`exports` 是现代包的对外入口声明。它可以：

- 明确暴露哪些入口可以被外部 import / require
- 隐藏内部实现文件
- 支持条件导出（比如区分 `import` 与 `require`）

**为什么重要：**

以前很多包默认可以 `require('pkg/lib/a.js')` 深层引用内部文件，这会让包作者难以重构。加上 `exports` 后，可以把“公共 API 边界”收紧。

**常见追问：**

- `exports` 和 `main` 的关系？  
  新版本 Node 更推荐 `exports`，它比 `main` 更强；在支持的版本里 `exports` 优先级更高。
- 风险是什么？  
  老项目若直接加 `exports`，可能把用户原来能访问的内部路径全部拦掉，属于潜在 breaking change。

---

## 11. ESM 和 CommonJS 能互相引用吗？

**参考答案：**

可以，但有边界，不能想当然混用。

**常见规律：**

- ESM 可以 `import` CommonJS，通常会拿到一个默认导出（本质上是 `module.exports`）。
- CommonJS 想加载 ESM，通常要用 `import()`，而不是简单 `require()`。
- 混用时最容易踩坑的是：默认导出、命名导出、执行时机、top-level await。

**面试表达建议：**

> Node 支持互操作，但互操作不代表零成本。项目里最好尽量统一模块风格，尤其在基础库与构建配置上。

# 第三部分：异步编程、错误处理与 process

## 12. Node 里为什么早期大量使用 error-first callback？

**参考答案：**

经典形式是：

```js
fs.readFile('a.txt', (err, data) => {
  if (err) return handle(err)
  console.log(data)
})
```

这种设计的核心价值是：

- 异步结果统一出口
- 错误和成功路径都明确
- 避免同步 `try/catch` 误以为能接住异步错误

**原理层面的理解：**

在异步回调模型里，函数调用和结果返回已经不是同一个时间点了，所以不能再用普通返回值表达结果，必须通过回调、Promise 或事件把结果“送回来”。

**今天怎么评价它？**

- 它不够优雅，容易回调嵌套
- 但它把 Node 的异步心智模型打得很清楚
- 现代项目更常用 Promise / `async-await`，但你应该知道这个历史包袱怎么来的

---

## 13. `Promise` 和 `async/await` 在 Node 里解决了什么问题？

**参考答案：**

它们主要解决的是：**异步流程组合与错误传播更可维护**。

### Promise 解决了什么
- 回调地狱
- 多个异步步骤串联难读
- 成功/失败状态难统一管理

### `async/await` 又带来了什么
- 让异步代码更像同步代码
- 更适合写复杂业务流程
- 可以直接用 `try/catch` 管理一个 await 链路上的异常

**但要注意：**

`await` 只是语法糖，不会让阻塞变成非阻塞。你 `await` 一个很重的同步计算也没意义，CPU 还是会被占住。

---

## 14. `async/await` 有哪些常见误区？

**参考答案：**

### 误区 1：写了 `await` 就一定更快
不一定。很多场景反而被你串行化了。

```js
// 慢：串行
const a = await fetchA()
const b = await fetchB()
```

如果互相没依赖，应该考虑：

```js
const [a, b] = await Promise.all([fetchA(), fetchB()])
```

### 误区 2：`try/catch` 能兜住所有异步错误
不是。你要看错误发生在哪个异步边界里。

### 误区 3：`await` 不会阻塞
更准确地说：它**不会阻塞整个线程去等 I/O**，但当前函数的后续逻辑确实要等结果回来。若 `await` 的是 CPU 重活，主线程照样可能被卡住。

**面试加分点：**

讲出“并发控制”和“错误边界”，通常就不只是会写语法了。

---

## 15. `unhandledRejection` 和 `uncaughtException` 是什么？应该怎么用？

**参考答案：**

### `unhandledRejection`
Promise 被 reject 了，但在一个事件循环轮次内没有被处理。

### `uncaughtException`
同步异常一路冒泡，最终没人接住。

**关键原则：**

它们可以用于 **记录日志、告警、兜底清理**，但不应该被当成“继续正常运行”的常规手段。

**为什么？**

因为一旦走到 `uncaughtException`，程序状态往往已经不可信了。Node 官方文档明确不建议把它当成正常恢复机制。

**更工程化的答法：**

- 局部错误在业务边界内处理
- 进程级错误做日志和资源清理
- 然后让进程退出，由外部守护或平台拉起

**一句话总结：**

> 它们是最后一道警报，不是业务层异常处理主通道。

---

## 16. `process` 对象平时最常用来做什么？

**参考答案：**

`process` 是 Node 当前进程的全局信息入口，常见用途包括：

- 读取环境变量：`process.env`
- 获取命令行参数：`process.argv`
- 监听进程事件：`exit`、`SIGINT`、`SIGTERM`
- 获取当前工作目录：`process.cwd()`
- 设置退出码：`process.exitCode`

**面试时可以讲实际用法：**

- 用环境变量区分开发、测试、生产
- 优雅关闭服务时监听 `SIGTERM`
- CLI 工具通过 `argv` 读参数

**注意：**

不要随手 `process.exit()`，尤其在服务端应用里，粗暴退出可能导致正在处理的请求、日志、连接来不及收尾。

# 第四部分：EventEmitter、Buffer 与 Stream

## 17. `EventEmitter` 的核心机制是什么？

**参考答案：**

Node 很多核心模块都建立在事件驱动模型上，`EventEmitter` 是这个模型的基础设施。

它的核心就是：

- `on` 注册监听器
- `emit` 触发事件
- 监听器被依次调用

**非常重要的一点：**

Node 文档里明确说明：**`emit()` 触发时，相关监听器是同步调用的。**

这意味着：

- 监听器里写重 CPU 代码，会直接拖慢当前流程
- “事件驱动”不等于“自动异步”

**常见追问：**

为什么很多人以为它是异步的？  
因为很多事件本身来源于异步 I/O，但“事件产生是异步的”不等于“监听器执行是异步的”。

---

## 18. `Buffer` 是什么？为什么 Node 里一定要理解它？

**参考答案：**

`Buffer` 用来表示一段二进制数据，本质上是字节序列。在 Node 里，文件、TCP、HTTP、流、加密、压缩，都绕不开二进制。

**为什么服务端必须理解它：**

- 网络传输的数据底层都是字节
- 文件不是天然字符串
- 不同编码之间要明确转换
- 大文件、流式处理、协议解析都离不开 Buffer

**关键点：**

- `Buffer` 继承自 `Uint8Array`
- `Buffer.from(str)` 是把字符串编码成字节
- `buf.toString()` 是把字节按指定编码解码

**常见误区：**

- 误区：拿到数据就默认是字符串。  
  真实情况往往是“先是字节，再根据编码解释成字符串”。

---

## 19. 什么是 Stream？为什么它比一次性读完整个文件更重要？

**参考答案：**

Stream 是 Node 处理流式数据的抽象。它适合处理：

- 大文件
- 网络传输
- 压缩与解压
- 视频、日志、代理转发

**四类流：**

- `Readable`
- `Writable`
- `Duplex`
- `Transform`

**为什么它重要：**

如果你 `fs.readFile()` 读一个超大文件，会把整块内容尽量放进内存；而 stream 是边读边处理、边处理边输出，更省内存，也更贴合真实 I/O 场景。

**一句话总结：**

> Stream 的核心价值不是“API 花哨”，而是“用持续流动的数据替代一次性全量加载”。

---

## 20. 什么是背压（backpressure）？

**参考答案：**

背压就是：**上游生产数据太快，下游消费不过来**，于是中间缓冲区越堆越多。

这在下面场景里很常见：

- 大文件读取 → 网络发送
- 上传流 → 压缩流 → 落盘
- 数据库导出 → HTTP 下载

**为什么背压重要：**

如果没有流控，内存会不断上涨，严重时会把进程拖垮。

**Node 怎么处理：**

Stream 体系内置了流控机制，比如写入方的 `write()` 返回值、`drain` 事件、`pipeline()` 等。

**面试高频表达：**

> 背压的本质是“生产速度”和“消费速度”不匹配，Stream 的设计就是为了让这种速度差有地方被感知和调节。

---

## 21. 为什么推荐 `stream.pipeline()`，而不是自己手搓多个 `pipe()`？

**参考答案：**

因为 `pipeline()` 更适合生产环境，它会更系统地处理：

- 错误传递
- 资源关闭
- 流结束
- 中间环节失败时的清理

**为什么手写 `a.pipe(b).pipe(c)` 不够？**

因为一旦中间某个流出错，错误与关闭逻辑容易遗漏，最后造成句柄泄漏、连接不释放、响应挂住。

**面试答法：**

> Demo 里直接 `pipe()` 很常见，但线上链路更推荐 `pipeline()`，因为它更强调整条流管道的生命周期管理。

---

# 第五部分：文件、路径与 HTTP 基础

## 22. `fs.readFile()` 和 `fs.createReadStream()` 应该怎么选？

**参考答案：**

看数据规模和处理方式。

### 用 `readFile()` 的场景
- 文件小
- 就是一次性读取配置、模板、小文本
- 逻辑简单

### 用 `createReadStream()` 的场景
- 文件大
- 要边读边传
- 要和压缩、网络、加密流串起来
- 内存敏感

**面试的本质不是背 API，而是讲出权衡：**

- `readFile()` 开发更快
- Stream 更适合长期运行的服务端程序

---

## 23. 为什么说 Node 很适合做 BFF、网关、中间层、SSR 接入层？

**参考答案：**

因为这些场景通常都不是超重 CPU，而是：

- 聚合多个接口
- 做协议转换
- 转发请求
- 拼装数据
- 控制缓存
- 管理登录态和鉴权

这些都偏 **I/O 密集、编排型逻辑**，正好符合 Node 的优势。

**一个成熟一点的说法：**

> Node 适合做“离用户近、离 I/O 近、离前端协议近”的一层，尤其适合前端团队向服务端延伸时做 BFF。

---

## 24. 一个最基础的 HTTP 服务，在 Node 里本质上发生了什么？

**参考答案：**

当你 `http.createServer()` 时，本质是在创建一个监听 socket 的服务。收到请求后，Node 会把请求包装成 `IncomingMessage`，把响应包装成 `ServerResponse`，然后把回调交给你处理。

**你可以这样展开：**

- 请求本质是字节流，不是天然 JSON
- 请求头、方法、URL、body 都要分别处理
- 响应必须显式结束，通常要 `res.end()`
- 如果请求体很大，本质上也应该按流处理，而不是默认一次性拼到内存

**加分点：**

把 HTTP 理解成“协议 + 流 + 连接管理”，而不是“框架帮我拿个 req/res”。

---

## 25. 为什么上传文件、代理转发、下载大文件时，流式处理几乎是必选项？

**参考答案：**

因为这类场景的数据量大、持续时间长，如果你总想“先全部收齐，再一次性处理”，很容易带来：

- 内存暴涨
- 首字节延迟变长
- 错误恢复差
- 进程不稳定

**更底层的理解：**

网络和磁盘本来就是分段到达、分段写入的，所以流式处理更符合底层设备工作的方式。

**面试时一句话总结：**

> 处理大数据量时，正确姿势通常不是“存满再说”，而是“边到边处理”。

---

# 第六部分：性能、稳定性与工程意识

## 26. 为什么不能在 Node 主线程里做重 CPU 计算？

**参考答案：**

因为 Node 主线程既要跑你的业务逻辑，也要继续推进事件循环。一旦你在主线程做很重的同步计算，其他请求就拿不到执行机会。

**后果包括：**

- 所有请求一起变慢
- timer 延迟
- I/O 回调迟迟执行不到
- 高并发下吞吐明显下降

**面试时要讲出的本质：**

> Node 最怕的不是“忙”，而是“主线程一直不让别人轮到”。

---

## 27. 为什么同步 API（如 `readFileSync`、`execSync`）在线上服务里通常要慎用？

**参考答案：**

因为同步 API 会直接阻塞当前线程。在服务端程序里，这通常意味着阻塞整个事件循环。

**典型问题：**

- `readFileSync`：遇到慢磁盘、网络文件系统、超大文件时风险更大
- `execSync`：外部命令慢时，整个进程跟着卡住
- `JSON.parse` / `JSON.stringify` 超大对象：虽然不是 API 名字带 Sync，但本质也是同步 CPU 开销

**正确回答姿势：**

不是说同步 API 永远不能用，而是：

- CLI 脚本、小工具、启动阶段配置读取可以接受
- 长期运行的服务端请求链路里要慎用

---

## 28. Node 服务的内存问题通常从哪里来？

**参考答案：**

高频来源通常有这几类：

1. **意外的全局引用**  
   某些对象一直被引用，GC 回收不了。
2. **缓存没有边界**  
   只存不清，没有 TTL 或容量限制。
3. **事件监听器泄漏**  
   重复 `on`，不 `off`。
4. **流/连接没正确关闭**
5. **大对象长期滞留**  
   比如一次性读超大文件、超大 JSON 常驻内存。

**面试时别只说“内存泄漏”。**  
最好讲成：

> JS 有 GC，不代表不会泄漏。只要对象还可达，就不会被回收。

这句话通常很加分。

---

## 29. 如何理解 Node 的“不要阻塞事件循环”这句话？

**参考答案：**

它不是一句口号，而是 Node 服务设计的根约束。

**你可以从两层理解：**

1. **不要写超长同步回调**  
   比如复杂循环、大正则、超大 JSON 处理。
2. **不要滥用会卡线程的 API**  
   比如同步文件、同步子进程、超重加密/压缩直接在主线程跑。

**再进一层：为什么它甚至和安全有关？**

因为如果某类输入能稳定把你的主线程拖很久，恶意请求就可能造成 DoS。Node 官方就专门提醒过要避免阻塞事件循环与 worker pool。

---

## 30. 线上 Node 服务做错误处理时，为什么强调“分层处理”？

**参考答案：**

因为不是所有错误都该在一个地方兜。

### 更合理的分层方式
- **参数层**：输入校验错误，尽早返回
- **业务层**：可预期异常，返回明确业务码
- **基础设施层**：数据库、缓存、RPC、文件系统异常，做重试/熔断/降级
- **进程层**：兜底日志、告警、优雅退出

**为什么分层重要：**

如果你把所有错误都扔到最外层统一 try/catch，短期看简单，长期看会丢语义，最终很难定位、很难恢复。

---

# 第七部分：安全与依赖意识

## 31. Node 服务常见的安全问题有哪些？

**参考答案：**

基础面试里，能说出以下几类就不错：

- 输入校验不足
- 命令注入（尤其乱用 `exec`）
- 路径穿越
- 不安全反序列化 / 原型污染
- 敏感信息泄露（环境变量、日志、堆栈）
- npm 供应链风险

**为什么前端转全栈的人容易忽略这个？**

因为前端更常关注 XSS、CSRF；而 Node 服务端还要直面文件系统、进程、网络边界、依赖执行权限。

---

## 32. 为什么 `exec()` 比 `spawn()` / `execFile()` 更需要小心？

**参考答案：**

因为 `exec()` 会通过 shell 执行命令。只要用户输入有机会拼进命令字符串，就有命令注入风险。

**更稳妥的实践：**

- 能不用 shell 就别用
- 优先考虑 `execFile()` 或 `spawn()` 传参数数组
- 对输入做严格白名单校验

**面试时一句话：**

> 只要 shell 介入，转义、拼接、注入面都会更复杂。

---

# 第八部分：综合场景题（非常适合面试追问）

## 33. 如果让你用 Node 做一个大文件上传服务，你会重点考虑什么？

**参考答案：**

这题考的是你是否真的理解 Node 的优势和边界。

**至少要答到这几个点：**

1. **流式处理**：不要整文件读入内存
2. **背压控制**：上传速度和写盘速度不一致时要能兜住
3. **异常处理**：断连、超时、部分上传、清理临时文件
4. **安全性**：文件类型、大小、路径、恶意内容校验
5. **可扩展性**：分片上传、断点续传、对象存储对接

**如果再加分一点：**

可以补一句：

> 这类场景里 Node 的价值主要在接入、编排、流处理，不在于做复杂文件计算本身；如果要做重转码，可能要下沉到 worker 或独立服务。

---

## 34. 如果某个 Node 接口响应越来越慢，你会怎么排查？

**参考答案：**

一个比较像样的排查思路是：

1. **先分层**：是网络慢、下游慢、主线程卡、还是内存/GC 抖动
2. **看事件循环是否被阻塞**
3. **看是否有慢 SQL / 慢 RPC / 慢文件 I/O**
4. **看是否误用了同步 API**
5. **看是否有大对象序列化、超大 JSON、重 CPU 逻辑**
6. **看日志、metrics、profiling、heap/CPU 快照**

**面试时重点不是工具名背多少，而是：**

> 你有没有“先判断瓶颈属于哪一层”的意识。

---

## 35. 如果前端团队想把 BFF 层交给 Node，你觉得团队最容易踩什么坑？

**参考答案：**

我认为最常见的坑有四个：

1. **只会写功能，不懂服务稳定性**  
   例如超时、重试、限流、日志、错误边界全没想。
2. **把 Node 当浏览器 JS 延伸**  
   结果忽视流、进程、内存、I/O 模型。
3. **过度依赖框架，忽略底层原理**
4. **CPU/内存意识不够**  
   比如请求里做大循环、大 JSON 操作、同步文件。

**这题其实很适合体现你的成长视角：**

> 前端转全栈最大的提升，不是学会 Express/Nest，而是开始用“服务”的角度思考响应时间、并发、资源、故障和边界。

# 最后总结

如果你的目标是“前端转全栈时，Node 面试不至于没话说”，那真正该掌握的不是框架 API，而是这几个底层问题：

- Node 为什么能扛高并发
- 它什么时候会变慢
- 它怎么处理异步与错误
- 它为什么重视流和背压
- 它适合什么，不适合什么

把这些讲清楚，Node 面试就不再只是背题，而是你真的理解这门运行时。
