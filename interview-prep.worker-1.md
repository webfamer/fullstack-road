## 一、最可能被深挖的项目题（按概率排序）

## 1. BI 报表分析平台（最高概率）

### 题 1：你在 BI 报表平台里做的最核心工作是什么？
**60 秒回答**
我在这个项目里主要负责复杂查看链路的需求迭代、线上问题处理，以及图表渲染体系的建设。相对更核心的一块，是基于 Storybook 搭建图表渲染器组件体系，并把数据处理到图表渲染这条链路做成管道式解耦，最后抽离成 npm 包提升复用性。除此之外，我也做了复杂报表场景下的加载和渲染性能优化、移动端适配、埋点建设、国际化，以及基于 LangChain.js 的自然语言生成报表探索。

**面试官为什么会问**
- 这是最近一段经历，时效性最强
- 同时覆盖组件化、架构抽象、性能优化、国际化、AI 探索，信息密度最高

**高频追问点**
- 为什么要做图表渲染器组件体系？
- 管道模式具体怎么拆？
- npm 包抽离后怎么管理版本和兼容性？
- 复杂报表性能问题主要卡在哪？
- LangChain 方案最后做到什么程度？

**追问参考回答**
- **为什么做组件体系**：报表图表类型多、配置项多，如果每个业务页面各自拼装，后期维护成本会非常高，所以我更倾向先统一图表渲染入口，再把数据处理、配置转换、图表渲染拆开。
- **管道模式怎么拆**：我会按“原始数据 -> 业务数据结构化 -> 图表配置生成 -> 渲染器输出”去分层，每层只关心自己的输入输出，这样新增图表类型或者调整数据映射时，不需要改整条链路。
- **为什么抽 npm 包**：因为同类图表渲染逻辑会在多个报表场景复用，抽包后可以统一升级、统一修 bug，也更利于 Monorepo 内协作。
- **性能优化重点**：主要是首屏加载、复杂报表渲染耗时、以及高交互场景下的重复计算。我一般会先定位瓶颈，再从减少无效渲染、降低计算量、按需加载、缓存中间结果这些方向处理。
- **LangChain 探索做到什么程度**：主要是做自然语言到图表配置的基础映射，属于探索型能力验证，不会把它说成成熟生产化系统。

---

### 题 2：你说做了报表性能优化，具体怎么定位和解决？
**60 秒回答**
我一般先区分是数据获取慢、数据处理慢，还是渲染慢。对 BI 报表这类系统，常见瓶颈是字段多、图表多、联动复杂，导致首屏加载慢和渲染卡顿。我会先通过性能面板、埋点和实际场景复现去定位瓶颈，再分别处理，比如减少不必要的渲染、避免重复的数据转换、拆分重计算链路、按需渲染非首屏内容，以及优化图表组件本身的更新时机。核心思路不是盲目上优化手段，而是先定位最重的那一段。

**追问参考回答**
- **怎么判断是 React 渲染问题还是图表库问题**：我会先看组件更新次数和耗时，再看图表实例更新是否过于频繁。如果 React 层更新不多，但图表 setOption 很重，那重点就在图表层；如果组件层频繁重渲染，就先从状态拆分和依赖控制入手。
- **做过哪些具体动作**：比如拆分大组件、稳定 props 引用、减少全局状态联动范围、把高成本计算做缓存、把非关键内容延后加载。
- **优化结果怎么表达更真实**：如果没有特别准确的数据，我会说“明显改善首屏响应和复杂报表操作流畅度”，不会编造百分比。

---

### 题 3：你在报表平台里做的 AI 能力探索是什么？
**60 秒回答**
我做的是基于 LangChain.js 的自然语言生成报表能力探索，目标是把用户的自然语言问题先映射成字段、筛选条件和图表类型，再生成基础配置。这个能力我会定义成探索型方案，不会夸大成完整 AI 产品。它更像是在验证“自然语言 -> 报表配置”这条链路是不是能跑通，以及在哪些场景下有实际价值。

**追问参考回答**
- **LangChain 为什么适合这个场景**：因为它更适合做多步骤编排，比如提示词组织、字段映射、上下文拼接和结果结构化输出。
- **最大难点是什么**：不是生成一句答案，而是要把自然语言稳定地映射到受控的数据字段和图表配置，这里对约束和结构化要求很高。
- **有没有上线**：如果只是探索，就直接说是探索和验证，不要包装成大规模生产落地。

## 2. 电力设备监测云平台（高概率）

### 题 4：这个项目为什么能体现你的全栈能力？
**60 秒回答**
这个项目我比较能体现前后端一体化能力，因为我是独立完成平台前后端系统设计与开发的，前端覆盖设备台账、报文详情、告警规则、统计分析和驾驶舱，后端负责 JWT 认证、RBAC 权限控制、TCP 报文接入和解析链路，以及 MySQL、Redis 等基础能力接入。这个项目的特点不是单点页面开发，而是把设备接入、协议解析、业务管理和可视化串成完整链路。

**高频追问点**
- TCP 报文链路怎么设计？
- 怎么处理拆包粘包？
- 为什么要把原始报文落库？
- RBAC 权限怎么做？
- 慢 SQL 是怎么排查和优化的？

**追问参考回答**
- **TCP 链路**：接入层先负责连接和原始报文接收，再做协议识别、拆包粘包处理、字段解析、结果持久化，最后再进入告警和展示链路。
- **拆包粘包处理**：核心是按协议边界做缓存和切分，确保一条完整报文再进入解析逻辑。
- **原始报文为什么要落库**：便于追溯、排错和后续补解析，尤其现场设备协议经常会调整。
- **RBAC 怎么做**：我会按账号、角色、菜单/接口权限来控制，同时让前后端各自做一层约束，前端负责可见性和路由控制，后端负责最终鉴权。
- **慢 SQL 优化**：先定位慢查询来源，再看是不是索引缺失、条件不合理、关联过重或者分页方式有问题，然后针对性调整。

---

### 题 5：这个项目里最能体现你解决复杂问题能力的点是什么？
**60 秒回答**
我认为是 TCP 报文接入和解析链路这块，因为它不是纯前端页面开发，而是要理解设备上报协议、处理网络传输层常见问题、做好解析结果持久化，还要让前端页面和告警链路能真正消费这些数据。这个过程中我既要考虑协议兼容，也要考虑后续排错和扩展，所以会保留原始报文，并把解析流程拆成相对清晰的步骤。

## 3. 智慧靶场驾驶舱（高概率）

### 题 6：你在高频实时数据可视化场景里做了什么？
**60 秒回答**
这个项目是典型的大屏 + WebSocket 实时数据场景，我主要负责核心模块开发，包括训练过程、人员轨迹、命中情况、现场视频和训练结果的可视化展示。重点难点在于高频数据流处理和长时间运行稳定性，所以我做了增量更新、性能优化，以及对白屏、卡顿、内存占用问题的排查处理。

**高频追问点**
- WebSocket 高频更新怎么避免页面卡死？
- Unity WebGL 和前端怎么配合？
- 白屏和内存问题怎么排查？

**追问参考回答**
- **高频更新处理**：不会每来一条数据就整页重算，而是尽量做增量更新、分模块更新，避免把所有状态都绑在一起。
- **Unity WebGL 配合**：前端负责业务控制、数据接入和 UI 展示，Unity WebGL 负责三维表现，两边通过约定的数据结构和事件进行交互。
- **白屏/内存排查**：先复现场景，再看是视频流、WebGL 资源、还是页面对象没释放干净；核心是把问题拆开，不要一上来就判断是某一层的问题。

---

### 题 7：长时间运行的大屏系统你最关注什么？
**60 秒回答**
我最关注的是稳定性，而不只是某个瞬时页面性能。因为这类系统经常要长时间运行，所以除了渲染流畅，还要关注内存是否持续增长、WebSocket 是否稳定、视频和 WebGL 资源是否正确释放、异常场景下页面能不能恢复。我的思路一般是先保证系统不崩，再去追求更极致的视觉效果或交互体验。

## 六、前端 + 全栈高频八股（先记问题，再记答题骨架）

> 这一部分先解决“你知道面试官会问什么”；下一部分再给标准回答。
> 优先级：React/TypeScript/性能 > 浏览器/工程化 > Node/NestJS/数据库/缓存 > AI 工程化。

### A. React / TypeScript（最高频）

1. React 组件为什么会重新渲染？
2. React.memo、useMemo、useCallback 分别解决什么问题？
3. useEffect 和 useLayoutEffect 的区别是什么？
4. Hooks 闭包陷阱怎么出现，怎么避免？
5. key 在 Diff 里起什么作用，为什么不建议乱用 index？
6. 受控组件和非受控组件的区别是什么？
7. React 状态为什么要尽量最小化、局部化？
8. TypeScript 的 any、unknown、never 有什么区别？
9. 泛型的价值是什么，项目里一般怎么用？
10. interface 和 type 在项目里怎么选？

**记忆骨架**
- React：先讲“状态变化驱动 UI”，再讲“render ≠ DOM 重建”，最后讲“优化重点是减少无效更新”。
- Hooks：先讲依赖数组，再讲闭包引用旧值，再讲 cleanup。
- TS：先讲类型安全，再讲可复用抽象，再讲边界约束。

### B. 前端性能 / 可视化（高频）

1. 前端性能优化通常怎么分层？
2. BI / 图表 / 大屏场景最容易卡在哪里？
3. 高频更新场景怎么避免页面卡死？
4. 长时间运行页面为什么容易出稳定性问题？
5. 怎么判断问题在 React、图表库还是数据处理链路？
6. 懒加载、缓存、虚拟列表、按需渲染分别适合什么场景？

**记忆骨架**
- 先定位：采集指标 / Performance / 埋点 / 复现场景。
- 再拆层：请求、计算、渲染、资源、更新频率。
- 最后落手段：减少重算、拆组件、缓存中间结果、分批更新、延后非首屏。

### C. 浏览器 / 网络（高频）

1. 浏览器从输入 URL 到页面渲染发生了什么？
2. HTTP 缓存强缓存和协商缓存怎么区分？
3. 浏览器事件循环（Event Loop）怎么理解？
4. 宏任务和微任务的区别是什么？
5. WebSocket 和 HTTP 轮询的区别是什么？
6. 前端内存泄漏常见场景有哪些？

**记忆骨架**
- URL：解析 -> DNS -> TCP/HTTPS -> 请求响应 -> HTML 解析 -> CSSOM/DOM -> Render Tree -> Layout/Paint。
- 缓存：先看 Cache-Control/Expires，再看 ETag/Last-Modified。

## 七、问题 + 标准回答（贴近真实面试）

### 1. React 组件为什么会重新渲染？
**标准回答**
React 本质上是状态变化驱动 UI 重新计算。组件重新渲染常见有 4 个原因：props 变化、state 变化、父组件重新渲染、context 变化。重新渲染不等于真实 DOM 一定重建，React 会先做虚拟 DOM 对比，再把最小变更提交到真实 DOM。实际优化时，我关注的不是“完全不 render”，而是避免无效计算和高成本子树重复更新。

### 2. useEffect 和 useLayoutEffect 的区别是什么？
**标准回答**
useEffect 是在浏览器绘制后异步执行，适合请求、订阅、日志上报这类不阻塞首屏展示的副作用；useLayoutEffect 是在 DOM 更新后、浏览器绘制前同步执行，适合读取布局、测量尺寸、修正闪动。我的默认选择一定是 useEffect，只有确实依赖同步布局结果时才会用 useLayoutEffect。

### 3. Hooks 为什么容易有闭包问题？
**标准回答**
因为函数组件每次渲染都会生成新的作用域，如果异步回调、定时器或者事件监听持有的是旧作用域里的值，就会出现“拿到旧 state”的问题。常见处理方式有三类：第一，正确写依赖数组；第二，用函数式更新拿最新状态；第三，用 ref 保存不需要触发渲染但需要保持最新的值。

### 4. key 的作用是什么？为什么不能随便用 index？
**标准回答**
key 的作用是帮助 React 在 Diff 时识别节点身份，判断节点是复用、移动还是销毁重建。如果列表是稳定且不会重排，index 还能勉强用；但只要列表存在插入、删除、排序，用 index 就会让节点身份错位，可能带来状态错乱和额外渲染。所以更合理的做法是使用稳定且业务唯一的 id。

### 5. TypeScript 泛型的价值是什么？
**标准回答**
泛型的价值是让类型在“复用”和“约束”之间取得平衡。它不是把类型写复杂，而是在不知道具体类型时，先把类型参数抽出来，等调用时再确定。项目里比较常见的用法包括接口响应封装、通用表格/表单组件、工具函数，以及数据处理管道里的输入输出约束。

### 6. 说一下你理解的前端性能优化。
**标准回答**
我会先把性能问题拆成四层：资源加载、数据处理、渲染更新和长期运行稳定性。第一步不是直接上手段，而是先定位瓶颈，比如通过 Performance、埋点、日志和复现场景确认到底慢在哪。确认之后，再分别用按需加载、缓存中间结果、减少无效渲染、拆分重计算链路、延后非关键内容这些方式处理。核心不是“把所有优化都做一遍”，而是找到最重的那一段。

### 7. 高频图表 / 大屏场景怎么做性能优化？
**标准回答**
这类场景最怕的是数据量大、更新频率高、模块联动多，所以我一般会先避免“每来一条数据就整页重算”。具体会做三件事：第一，按模块拆更新边界；第二，对高成本计算做缓存或增量更新；第三，把非首屏和非关键内容延后处理。对于图表层，还会重点看 setOption 时机、数据裁剪、实例复用和销毁清理。

### 8. WebSocket 和 HTTP 轮询有什么区别？
**标准回答**
HTTP 轮询是客户端定期发请求问服务端有没有新数据，实现简单，但有额外请求开销和实时性损耗；WebSocket 是建立持久连接后双向通信，更适合实时性要求高的场景，比如大屏、IM、设备状态流。实际项目里我不会只说“用了 WebSocket”，还会补充断线重连、心跳保活、异常恢复和消息去重这些稳定性处理。

### 9. 什么是前端内存泄漏？常见场景有哪些？
**标准回答**
前端内存泄漏本质上是一些本该被回收的对象仍然被引用，导致页面运行越久占用越高。常见场景包括：定时器没清理、事件监听没解绑、闭包持有大对象、缓存无限增长、图表/WebGL/视频资源没有销毁。像大屏这种长时间运行页面，我会特别关注资源释放，而不只是首屏速度。

### 10. 浏览器从输入 URL 到页面渲染的过程是什么？
**标准回答**
可以按链路去答：先解析 URL，查 DNS，建立 TCP 或 TLS 连接；然后发送 HTTP 请求、服务端返回响应；浏览器拿到 HTML 后开始解析 DOM，同时加载 CSS 和 JS，生成 CSSOM，组合成渲染树，接着做布局和绘制，最后页面呈现出来。面试里我会重点强调：JS、CSS、网络和渲染不是割裂的，它们会共同影响首屏性能。

### 11. HTTP 缓存怎么理解？
**标准回答**
HTTP 缓存可以分成强缓存和协商缓存。强缓存命中时浏览器直接用本地副本，不发请求，常见头是 Cache-Control 和 Expires；协商缓存会向服务端确认资源有没有变，常见头是 ETag / If-None-Match 和 Last-Modified / If-Modified-Since。实际项目里静态资源通常更适合长缓存 + 文件 hash，接口数据缓存则要更谨慎，避免脏数据问题。

### 12. Event Loop 怎么理解？
**标准回答**
JavaScript 主线程一次只做一件事，所以异步任务需要通过事件循环来调度。大致顺序可以理解成：先执行同步代码；当前宏任务结束后，立即清空微任务队列；如果条件允许，浏览器再进行渲染；然后进入下一个宏任务。面试里说清“Promise.then 是微任务，setTimeout 是宏任务”，再补一句“不要在微任务里堆太重逻辑影响渲染”就比较完整。

### 13. Monorepo 有什么优缺点？
**标准回答**
Monorepo 的优点是跨包协作方便、依赖版本统一、公共能力复用更顺畅，尤其适合组件库、工具包和业务项目要一起演进的场景。缺点是仓库规模变大后，构建、权限、发布和依赖治理复杂度会上升。所以我更看重的是有没有配套的边界管理、构建缓存、版本策略和文档规范，而不是只把代码放在一个仓库里。

### 14. JWT 和 Session 的区别是什么？
**标准回答**
Session 模式一般是服务端保存会话状态，客户端通过 Cookie 携带会话标识；JWT 则是把认证后的声明信息编码成 token，由客户端携带，服务端按规则校验。JWT 更适合前后端分离和多服务场景，但它并不天然解决所有安全问题，比如过期、刷新、注销和敏感信息控制都要额外设计。我在项目里更关注的是“认证链路是否清晰”和“鉴权边界是否统一”。

### 15. RBAC 权限模型怎么设计？
**标准回答**
RBAC 的核心是用户、角色、权限三层关系。用户不直接绑定很多细粒度权限，而是先绑定角色，角色再关联菜单、按钮、接口等权限点。前端主要负责菜单可见性、路由守卫和按钮控制，后端负责最终鉴权。这样设计的好处是权限管理更可维护，也更适合中后台平台类系统。

### 16. TCP 粘包 / 拆包是什么？怎么处理？
**标准回答**
TCP 是字节流协议，不保证应用层消息边界，所以发送方的多条消息在接收方可能连在一起，也可能一条消息被拆成多段，这就是粘包和拆包。处理关键不是“收到就解析”，而是先基于协议约定做缓存和切分，比如按固定长度、分隔符或消息头长度字段组装成完整报文，再进入业务解析流程。

### 17. Redis 常见使用场景有哪些？
**标准回答**
Redis 常见场景包括热点数据缓存、登录态或短期状态存储、计数器、排行榜、限流，以及做一些消息分发或分布式协调辅助能力。我的理解是 Redis 不是单纯为了“快”，而是放在真正高频读取、短生命周期、对延迟敏感的那部分数据上，同时要考虑过期策略和一致性边界。

### 18. 慢 SQL 通常怎么排查和优化？
**标准回答**
慢 SQL 我一般按“先定位、再分析、后优化”的顺序做。先从慢日志、监控或接口耗时里找到具体 SQL，再看执行计划，判断是不是索引缺失、过滤条件不合理、关联过多、排序分页代价大，或者回表严重。优化时优先做最小改动，比如补索引、改条件、拆查询、控制返回字段，而不是上来就重构整库。

### 19. LangChain 适合做什么？
**标准回答**
LangChain 更适合做多步骤编排和应用层串联，比如提示词组织、上下文拼接、工具调用、结构化输出，而不是替代模型本身。放在我的项目里，它的价值主要是帮助我把“自然语言 -> 字段理解 -> 图表配置生成”这条链路搭起来。但我会明确说这是工程探索，不会夸大成成熟 AI 平台能力。
