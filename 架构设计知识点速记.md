# 架构设计知识点速记

## 一、架构范式

### OO vs SOA
| 维度 | 面向对象(OO) | 面向服务(SOA) |
|------|-------------|-------------|
| 核心 | 对象 | 服务 |
| 层级 | 代码层面 | 系统层面 |
| 集成方式 | 工作流 + 对象调用 | 服务编排 / 服务组合 |

### REST 架构特征（口诀：抽唯统标状）
1. **资源抽象** - 一切皆资源
2. **唯一标识** - URL表示资源
3. **统一接口** - 通用接口操作
4. **标识不变** - 操作不改变资源ID
5. **无状态** - 客户端与服务器无状态交互

### 事件驱动架构
```
事件源 → 事件管理器 → 事件处理器
 (触发)    (分发调度)    (执行业务)
```

### 前后端分离
- **定义** - 解耦+API通信
- **优势** - 并行开发、技术栈独立、SPA体验好
- **挑战** - 跨域、接口同步、权限控制

## 二、框架对比

### MVC vs MVP vs MVVM
| 模式 | 关键特征 | 数据更新 | 适用场景 |
|-----|--------|--------|--------|
| MVC | View直连Model | 手动更新 | 传统Web |
| MVP | View只通过Presenter访问Model | 手动/接口 | 易测试的系统 |
| MVVM | View-ViewModel双向绑定 | 自动响应 | 现代前端框架 |

### Spring MVC vs WebX（口诀：多基扩）
| 维度 | Spring MVC | WebX |
|-----|-----------|------|
| 多应用 | 自行规划 | 天生支持 |
| 基础服务 | 需第三方 | 内置丰富 |
| 扩展性 | 中等 | 强大 |

### Hibernate vs MyBatis
| 维度 | Hibernate | MyBatis |
|-----|-----------|---------|
| 特点 | 全自动ORM | 半自动ORM |
| SQL生成 | 自动 | 手写 |
| 优势 | 开发快、无关性强 | 可控、易调优 |
| 前提 | 表设计合理 | 无要求 |

### SOAP vs REST（口诀：协状事场）
| 维度 | SOAP | REST |
|-----|------|------|
| 基础 | XML+WSDL | HTTP+URI |
| 状态 | 有状态 | 无状态 |
| 安全事务 | 完整支持 | 需额外处理 |
| 场景 | 企业级复杂系统 | 互联网应用 |
| 特点 | 安全可靠复杂 | 轻量简单易扩展 |

### HTTP vs WebSocket
| 维度 | HTTP | WebSocket |
|-----|------|-----------|
| 通信 | 请求-响应 | 全双工双向 |
| 连接 | 短连接 | 长连接 |
| 场景 | 资源加载 | 实时推送/聊天 |

**口诀** - HTTP短连接加资源，WebSocket长连接实时推。

### .NET vs Java
| 维度 | .NET | Java |
|-----|------|------|
| 生态 | 微软 | 多厂商 |
| 特点 | 集成度高、好部署 | 跨平台、可定制 |
| 应用 | Windows系统 | 企业级 |

## 三、系统设计

### 单体 vs 微服务
| 维度 | 单体 | 微服务 |
|-----|------|--------|
| 复杂度 | 整体复杂 | 拆分降低 |
| 可维护性 | 一处改全局受影响 | 独立维护 |
| 灵活性 | 不灵活 | 灵活调整 |
| 扩展性 | 整体扩容低效 | 按需独立扩展 |

**微服务挑战** - 运维复杂、网络开销、数据一致性难

### 设计原则
1. **扩展性** - 水平vs垂直扩展
2. **系统保护** - 降级/限流/熔断
3. **解耦原则** - 读写分离的本质是解耦
4. **性能优化** - 异步化提升吞吐

## 四、事务与一致性

### ACID特性
- **A** - 原子性（全部成功或全部失败）
- **C** - 一致性（数据完整性约束）
- **I** - 隔离性（并发事务相互隔离）
- **D** - 持久性（一旦提交永久保存）

### EJB事务控制
| 方式 | CMT(容器管理) | BMT(开发者管理) |
|-----|--------------|----------------|
| 特点 | 自动管理 | 手动控制 |
| 范围 | 方法级 | 可跨方法 |
| 优点 | 简单常用 | 灵活 |

### 2PC(两阶段提交)
```
准备阶段：协调者问"能提交吗?" → 参与者执行但不提交 → 回复YES/NO
提交阶段：全YES→提交 | 有NO或超时→回滚
```

**微服务中不适用原因**
- 2PC追求强一致，微服务强调高可用扩展
- NoSQL不支持XA协议
- 与微服务自治理念冲突

### 本地事务发布事件
```
第一步：订单+EVENT表 → 一个事务提交（保证原子性）
第二步：后台扫描EVENT表 → 发MQ → 更新状态
```
**优点** - 解决消息可靠性问题

## 五、消息与流处理

### 消息队列模式
- **点对点** - 一条消息只被一个消费者消费
- **发布/订阅** - 一条消息被多个订阅者消费

### 消息中间件作用
- **核心** - 削峰填谷（流量暴增先排队）
- **优势** - 异步、解耦、缓冲

### 软件功能识别
| 软件 | 作用 | 用途 |
|-----|------|------|
| Kafka | 消息中间件/缓冲 | 日志流、数据管道 |
| Flink | 实时流处理/批处理 | 实时指标、离线统计 |
| HBase | 分布式NoSQL存储 | 明细数据、用户画像 |
| Redis | 内存缓存 | 热点数据、会话存储 |

## 六、数据库

### 数据库选择
| 业务特征 | 数据库架构 |
|--------|----------|
| 读多写少 | 一主多从（读写分离） |
| 写多 | 多主/分片 |
| 高并发读 | 从库扩展 |

### NoSQL分类
- **键值型** - Redis（速度快）
- **文档型** - MongoDB（灵活）
- **列族型** - HBase（海量）
- **图数据库** - Neo4j（关系）

**NoSQL特点** - 易水平扩展、高性能、弱事务

---

## 七、数据安全

### 数据脱敏五法（口诀：替混两变删）
1. **替换** - 用随机/字典值替换
2. **混排** - 字段间交换打乱对应
3. **时间平移** - 日期前后偏移保趋势
4. **数值变异** - 加随机扰动保统计规律
5. **删除** - 直接置空敏感字段

### 负载均衡与代理
- **HAProxy** - 负载均衡
- **Squid** - 缓存代理

---

## 八、分布式与区块链

### 分布式文件
- **FastDFS** - 通用分布式文件系统
- **HDFS** - Hadoop生态文件系统

### 区块链核心特征
1. **去中心化** - 多节点共同记账
2. **开放性** - 数据公开透明可监督

**核心技术**
- 分布式账本 - 保证"改不了"
- 公私钥签名 - 保证"赖不掉"

## 九、云计算架构

### 混合云优势
1. **安全合规** - 敏感数据放私有云
2. **弹性扩展** - 高峰用公有云补充
3. **成本优化** - 渐进式上云，保留资产
4. **高可用** - 容灾切换，业务连续性
