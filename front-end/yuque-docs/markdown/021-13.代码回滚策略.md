# 13.代码回滚策略

> 来源：https://www.yuque.com/xiumubai/doc/hf2w7egn72gnmpgt

问题：我们现在上线的时候是每次都先把分支代码打一个tag，然后通过tag进行上线。上线成功后代码合并到master。如果线上出现问题，需要进行回滚，master合并的代码撤销。那么这个回滚策略如何来做？
基于Tag的代码回滚策略
回滚策略概述
在您的工作流中，回滚需要解决两个主要问题：
1将线上环境恢复到之前稳定的版本
2撤销已经合并到master的代码变更
详细回滚方案
方案一：基于Tag的直接回滚（推荐）
这是最直接有效的方案，直接使用上一个稳定的tag进行回滚。
实现步骤：
1确定回滚目标版本
​912# 查看最近的tag列表git tag --sort=-creatordate | head2部署回滚
​9123# 假设要回滚到tag v1.2.3git checkout v1.2.3# 触发部署流程，将v1.2.3版本重新部署到生产环境3处理master分支
​9123456# 在master分支上创建回滚提交git checkout mastergit revert <问题提交的hash>..HEAD# 或者创建一个回滚分支git checkout -b hotfix/rollback-from-v1.3.0git reset --hard v1.2.34代码审核与合并
​9123# 提交回滚变更到远程仓库git push origin hotfix/rollback-from-v1.3.0# 创建Pull Request，进行代码审核后合并到master优点：
●操作简单直接，回滚到已知稳定状态
●保留完整的版本历史
●不需要强制推送，避免团队协作问题
方案二：Master分支重置
如果问题非常严重且刚刚合并，可以考虑重置master分支。
实现步骤：
1确定回滚点
​9123# 查看master分支的提交历史git log master --pretty=oneline# 找到合并问题代码之前的提交2重置master分支
​9123git checkout mastergit reset --hard <合并前的提交hash>git push --force origin master3重新部署稳定版本
​9123# 使用最后一个稳定的taggit checkout v1.2.3# 触发部署流程注意事项：
●这种方法会改写历史，团队所有成员需要重新同步
●只适用于问题代码刚合并且团队成员尚未基于新master开发的情况
●需要团队协调，避免代码丢失
方案三：Cherry-Pick方式重建Master
这种方法适合需要保留某些功能但回滚特定问题的情况。
实现步骤：
1创建临时分支

为提高回滚效率，建议实现自动化回滚流程：
1创建回滚脚本

# 1. 检出目标taggit checkout $TARGET_TAG
# 2. 触发部署流程./deploy.sh

# 4. 推送回滚分支git push origin hotfix/rollback-to-$TARGET_TAG

在CI/CD系统中添加回滚选项，允许一键触发回滚流程：
​912345678# 在Jenkins或GitLab CI中的配置示例rollback-job:  stage: rollback  when: manual  script:    - ./rollback.sh ${TARGET_TAG}  variables:    TARGET_TAG: "v1.2.3"预防措施与最佳实践
1Tag命名规范
○使用语义化版本号（如v1.2.3）
○考虑在tag中包含日期信息（如v1.2.3-20230615）
2部署前验证
○在合并到master前，确保tag版本在预发环境充分测试
○实施自动化测试，确保代码质量
3记录变更日志
○每个tag详细记录变更内容
○标注关键变更点，便于回滚决策
4分阶段发布
○考虑实施金丝雀发布或蓝绿部署
○先向部分用户发布，验证无问题后全量发布
5回滚演练
○定期进行回滚演练，确保团队熟悉流程
○测量回滚所需时间，持续优化流程
回滚后的处理
1问题分析
○彻底分析导致回滚的问题
○在隔离环境中重现并修复问题
2重新发布
○修复问题后，创建新的功能分支
○完成测试后打新tag并重新发布
3流程优化
○根据每次回滚经验，持续改进发布流程
○更新自动化测试，防止类似问题再次发生
通过以上策略，您可以在现有的基于tag的发布流程中，实现高效、可靠的代码回滚机制，最大限度地减少线上问题对用户的影响。
