1vite现在使用的底层打包工具是啥？
Vite 使用的是 esbuild 作为底层的打包工具。esbuild 是一个非常快速的 JavaScript 打包工具，它利用 Go 语言编写，能够实现比传统的 JavaScript 打包工具（如 webpack）更高效的构建性能。
具体来说，Vite 的开发模式使用 esbuild 进行快速的模块解析和编译，这使得热更新 (HMR) 非常迅速。而在生产模式中，Vite 则使用 esbuild 来进行 JavaScript 的预构建和优化，最终使用 Rollup 来做最终的打包和代码优化。通过这种方式，Vite 实现了在开发时的快速响应和在生产时的高效打包。
总的来说，Vite 将 esbuild 用于开发构建过程中的大部分任务，同时结合 Rollup 来完成最终的打包和优化。
2vite为什么不用esbuild来来做最终的打包和代码优化。
Vite 不直接使用 esbuild 来做最终的打包和代码优化，主要是因为 esbuild 和 Rollup 各自有不同的优势和适用场景，而 Rollup 更适合进行最终的生产打包和优化。具体来说，原因包括以下几点：
2.11. Rollup 的优化功能更强大
●树摇 (Tree Shaking)：Rollup 对代码的树摇优化比 esbuild 更加精细和成熟。虽然 esbuild 也有树摇功能，但 Rollup 的优化算法更强大，尤其是在处理多个模块和复杂依赖时，能够更准确地去除未使用的代码，从而减少最终包的体积。
●代码拆分 (Code Splitting)：Rollup 在代码拆分（尤其是对于多个入口点的拆分）方面比 esbuild 更为成熟。它可以生成高效的代码拆分，尤其适合复杂的应用场景，尤其是库开发和多入口的应用。
2.22. 兼容性和生态支持
●插件生态：Rollup 拥有丰富且成熟的插件生态系统，很多针对生产环境的优化（如静态资源优化、压缩、模块处理等）都已经有现成的插件支持。而 esbuild 的插件生态相对较新和简陋，虽然在不断发展，但在某些特定场景下不如 Rollup 强大。
●兼容性：Rollup 更注重对标准 ES 模块的支持，适合用于库的打包，且对浏览器支持更加全面。很多生产环境的优化需求，比如针对不同模块格式（如 CommonJS、ESM、IIFE）和构建目标的定制化支持，在 Rollup 中实现得更为完善。
2.33. Rollup 在大项目中的表现
●在处理大型应用时，Rollup 能够更好地管理和优化输出，尤其是在输出多文件或者拆分大型应用时，Rollup 的精细控制和高效输出能力更胜一筹。对于应用级别的打包，Rollup 可以进行更多细节的优化，例如按需加载（lazy loading）和动态导入等。
2.44. 构建的速度与开发模式的平衡
●Vite 在开发模式中使用 esbuild，因为它的构建速度非常快，能够提供出色的开发体验。而生产模式下，尽管 esbuild 本身也非常快速，但 Rollup 在最终打包时的细致优化和处理能力更强。因此，Vite 选择在生产模式下使用 Rollup 来实现最佳的打包效果。
2.5总结
Vite 采用 esbuild 在开发模式中加速构建速度，而在生产模式中使用 Rollup 来做最终的打包和代码优化，是为了平衡构建速度和优化效果。esbuild 在开发时提供了足够的性能，而 Rollup 在最终的打包阶段提供了更强的优化能力，尤其是在处理大型应用和精细的代码拆分方面。
3那在开发模式下使用esbuild打包不用对代码做优化吗？
在开发模式下，虽然 esbuild 主要关注快速构建和提供流畅的开发体验，但它也会对代码做一些基础的优化，只是这些优化相比生产模式下的优化要少一些。具体来说，开发模式下的优化重点和生产模式下有显著区别，以下是一些主要差异和原因：
3.11. 开发模式的目标：快速反馈
在开发模式下，Vite 的主要目标是提供尽可能快的构建速度和热更新（HMR）。此时，开发者希望看到快速的构建和即时反馈，而不太关心最终产物的体积或性能。因此，esbuild 在开发时会尽量减少一些耗时的优化步骤。
3.22. 基础优化：
尽管如此，esbuild 仍然会对代码进行一些基础的优化：
●转译和压缩：esbuild 会将现代 JavaScript（如 ES6+）转译成浏览器兼容的代码，并做基本的压缩，但这通常不如生产模式下那样进行深度压缩和优化。
●模块解析：esbuild 会处理和解析代码中的模块导入，优化模块的构建过程，确保依赖关系正确，减少不必要的重新构建。
●代码分割：esbuild 支持基础的代码分割（Code Splitting），但它不像 Rollup 那样精细地进行多入口拆分和按需加载的优化。
3.33. 未启用高级优化：
在开发模式下，一些性能和体积优化（如树摇优化、代码压缩、详细的模块拆分等）不会开启，因为这些步骤会增加构建时间，且在开发过程中并不带来显著的好处。开发模式的核心目标是快速的模块热更新（HMR），而不是最终代码的高效打包。
●树摇优化 (Tree Shaking)：开发模式下 esbuild 并不做过多的树摇优化，因为开发时的重点是尽快反映更改而不是最终的代码体积。生产模式下则会对所有的未使用代码进行深入分析并剔除。
●代码压缩和优化：esbuild 的代码压缩在开发模式下是非常基础的，它不会进行像生产模式那样深入的压缩和合并，确保代码更易调试。
3.44. HMR 和实时构建
●在开发模式下，Vite 通过 HMR（Hot Module Replacement 热模块替换） 实现非常快速的代码更新。为了保证开发过程中尽可能快的反馈，esbuild 只进行必要的构建和更新，避免复杂的优化步骤以减少构建时间。
●由于开发过程中不需要生成最终的优化包，esbuild 的目标是尽量加速构建和热更新过程，而不是进行深入的性能优化。
3.5总结
在 开发模式 下，esbuild 主要关注快速的模块解析、转译、热更新和基础优化。它虽然会进行一些代码处理和简单的压缩，但不会像生产模式那样执行深入的优化（如树摇、复杂的代码拆分等），因为这些优化会增加构建时间，且在开发过程中并不直接带来好处。
而在 生产模式 下，Vite 则会使用 Rollup 进行精细的代码优化和打包，专注于减少代码体积、提高加载性能和执行效率。因此，开发模式和生产模式的优化策略是有所不同的，分别针对开发速度和最终产物的质量做出不同的权衡。
