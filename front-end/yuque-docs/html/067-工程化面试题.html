08：webpack 中什么是 HMR，原理是什么
HMR（Hot Module Replacement），热更新又称热替换，基于 webpack-dev-server。
第一步：webpack 对文件系统进行 watch 打包到内存中
第二步：devServer 通知浏览器端文件发生改变，在这一阶段，sockjs 是服务端和浏览器端之间的桥梁，在启动 devServer 的时候，sockjs 在服务端和浏览器端建立了一个 webSocket 长连接，以便将 webpack 编译和打包的各个阶段状态告知浏览器
第三步：webpack-dev-server/client 接收到服务端消息做出响应
第四步：webpack 接收到最新 hash 值验证并请求模块代码
第五步：HotModuleReplacement.runtime 对模块进行热更新
第六步：调用 accept 方法，及时将新后的内容插入到页面中

一、对于方法的处理
通过 tree shaking 设置后，webpack 里面会将没有使用的方法标记为： unused harmony export xxx，但代码仍然保留。（webpack 编译后的源码里面仍然包含没有使用的方法） 随后使用UglifyJSPlugin进行第二步，将已经标记的没有使用的方法进行删除。
二、对于类的处理
与标记方法不同，webpack 打包时会将整个类进行标记，也就是说，即使类里面的方法没有被使用也会进行打包编译。 这表明 webpack tree shaking 只处理顶层内容，例如类和对象内部都不会再被分别处理。 综上所述，可以得出 “对于已经 import 但未实际使用的模块使用 webpack 还会对它打包” 。 当然，想要精简代码是可以实现的

14：webpack 的 runtime 做了什么事情
webpack 的 runtime，也就是 webpack 最后生成的代码，做了以下三件事:
1__webpack_modules__: 维护一个所有模块的数组。将入口模块解析为 AST，根据 AST 深度优先搜索所有的模块，并构建出这个模块数组。每个模块都由一个包裹函数 (module, module.exports, __webpack_require__) 对模块进行包裹构成。
2__webpack_require__(moduleId): 手动实现加载一个模块。对已加载过的模块进行缓存，对未加载过的模块，执行 id 定位到 __webpack_modules__ 中的包裹函数，执行并返回 module.exports，并缓存
3__webpack_require__(0): 运行第一个模块，即运行入口模块
另外，当涉及到多个 chunk 的打包方式中，比如 code spliting，webpack 中会有 jsonp 加载 chunk 的运行时代码。

持久化缓存: cache
webpack5 内置了关于缓存的插件，可通过 cache 字段(opens in a new tab)配置开启。
它将 Module、Chunk、ModuleChunk 等信息序列化到磁盘中，二次构建避免重复编译计算，编译速度得到很大提升。

如对一个 JS 文件配置了 eslint、typescript、babel 等 loader，他将有可能执行五次编译，被五次解析为 AST
1acorn: 用以依赖分析，解析为 acorn 的 AST
2eslint-parser: 用以 lint，解析为 espree 的 AST
3typescript: 用以 ts，解析为 typescript 的 AST
4babel: 用以转化为低版本，解析为 @babel/parser 的 AST
5terser: 用以压缩混淆，解析为 acorn 的 AST
而当开启了持久化缓存功能，最耗时的 AST 解析将能够从磁盘的缓存中获取，再次编译时无需再次进行解析 AST。
得益于持久化缓存，二次编译甚至可得到与 Unbundle 的 vite 等相近的开发体验
在 webpack4 中，可使用 cache-loader(opens in a new tab) 仅仅对 loader 进行缓存。需要注意的是该 loader 目前已是 @deprecated 状态。

多进程: thread-loader
thread-loader(opens in a new tab) 为官方推荐的开启多进程的 loader，可对 babel 解析 AST 时开启多线程处理，提升编译的性能。

30：semver 指什么，试图解释一下
Semantic Versioning（语义化版本控制，简称 SemVer）是一种用于标识软件版本的规范和约定。它的主要目的是在软件开发中，通过版本号来清晰地传达有关软件变化和向后兼容性的信息，以便开发者和用户可以更好地理解和管理软件的更新。
SemVer 使用三个数字（Major.Minor.Patch）来表示版本号：
1Major Version（主版本号）：当你进行了不向后兼容的 API 修改，或者进行了重大功能改进时，应该增加主版本号。这表示可能需要对现有代码进行调整才能适应新版本。
2Minor Version（次版本号）：当你进行了向后兼容的功能性新增，或者进行了某种程度的重构，应该增加次版本号。这表示引入了新的功能，但现有功能仍然保持向后兼容。
3Patch Version（修订版本号）：当你进行了向后兼容的问题修复或小幅变化，应该增加修订版本号。这表示没有引入新功能，只是修复了问题。

SemVer 还允许在版本号后添加预发布版本信息和构建元数据，以更详细地说明版本。例如：
●1.0.0-alpha：预发布版本，可能包含未完成的功能或正在测试的内容。
●2.3.1+build42：包含构建信息，用于标识特定的构建版本。
通过遵循 SemVer 规范，开发者可以更好地理解软件的版本变化和向后兼容性，从而更轻松地进行升级和管理依赖项。这有助于减少因版本不一致而导致的问题，同时提供了一种清晰的方式来描述软件的变化。
31：dependencies、devDependencies和peerDependencies区别
在 Node.js 项目中，dependencies、devDependencies 和 peerDependencies 是三个不同的字段，用于指定项目所依赖的包（或模块）。它们在项目开发、构建和运行过程中扮演不同的角色。下面是它们的区别：
1dependencies（依赖项）: 这是项目在运行时需要的包（或模块）。这些包将在生产环境中使用，用于项目的正常运行。比如，如果你开发了一个 Web 应用程序，可能会依赖于一些用于处理服务器端逻辑或前端界面的包。这些依赖项会随着应用程序部署到生产环境中而被安装。
2devDependencies（开发依赖项）: 这是仅在开发过程中需要的包（或模块）。这些包通常包括测试框架、构建工具、代码检查工具等，用于开发、测试和构建项目。在项目部署到生产环境之前，这些依赖项通常不会被包含在其中。这样可以减小生产环境的包大小。
3peerDependencies（同依赖项）: peerDependencies 是一种指定包之间版本关系的机制。它指定了项目所依赖的另一个包的最低版本，以确保依赖的包能够在正确的版本下工作。这在创建可重用的库时很有用，可以确保使用该库的项目在依赖的包版本上保持一致。

●dependencies 是项目在运行时所需要的包。
●devDependencies 是在开发过程中需要的包，不会包含在生产环境中。
●peerDependencies 是用于指定包之间版本关系的机制，确保依赖的包版本一致。

package-lock.json：
1npm 生成的文件：package-lock.json 是由 npm（Node.js 包管理工具）生成的，它在 npm v5.0.0 及更高版本中引入。
2版本锁定：package-lock.json 的主要目的是锁定项目依赖项的精确版本，以确保在不同环境中安装相同版本的包。它记录了每个包的确切版本、依赖关系树和依赖解析算法。
3快速安装：package-lock.json 还包含了一些优化，使得在安装依赖时更加快速，因为它不必像以前那样重新解析依赖关系。

yarn.lock：
1Yarn 生成的文件：yarn.lock 是由 Yarn（另一个流行的包管理工具）生成的，Yarn 是由 Facebook 开发的，旨在提供更快的依赖项安装速度。

虽然 package-lock.json 和 yarn.lock 在功能上有一些差异，但它们的核心目标都是确保项目的包依赖版本一致性和可靠性。在选择使用哪个文件时，通常取决于你使用的包管理工具（npm 还是 Yarn）。
33：什么是浏览器的关键渲染路径
浏览器的关键渲染路径（Critical Rendering Path）是指浏览器在将网页内容呈现给用户之前所执行的一系列步骤。这个过程包括从服务器获取 HTML、CSS 和 JavaScript 文件，解析和构建 DOM（文档对象模型）、CSSOM（CSS 对象模型）以及执行 JavaScript，并最终将网页内容绘制到屏幕上。优化关键渲染路径可以提高页面加载速度和用户体验。
以下是关键渲染路径的主要步骤：
1获取 HTML： 浏览器通过网络请求获取 HTML 文件。
2构建 DOM 和 CSSOM： 浏览器解析 HTML 和外部 CSS 文件，生成 DOM（表示网页结构）和 CSSOM（表示样式信息）。
3生成 Render 树： 浏览器将 DOM 和 CSSOM 合并成一棵 Render 树，Render 树包含了需要在页面上显示的可见元素。
4Layout（布局）： 浏览器计算出每个元素在屏幕上的位置和大小，以确定页面的布局。
5Paint（绘制）： 浏览器绘制页面上的元素，将它们渲染成位图。
6Composite（合成）： 浏览器将位图按照正确的顺序组合在一起，以生成最终的页面图像。

关键渲染路径的优化涉及以下几个方面：
●减少网络请求： 使用技术如合并文件、压缩资源、使用浏览器缓存等来减少页面所需的网络请求。
●优化 CSS 和 JavaScript： 缩短 CSSOM 和 JavaScript 文件的加载、解析和执行时间，可以通过延迟非关键代码的加载或异步加载脚本来实现。
●避免布局抖动： 避免频繁的 DOM 元素尺寸变化，以减少不必要的页面布局。
●使用 CSS 动画： 使用 CSS 动画而不是 JavaScript 动画，因为 CSS 动画可以由浏览器优化，不会阻塞主线程。
●利用浏览器缓存： 使用适当的缓存策略，确保浏览器可以重用之前加载过的资源。
●最小化重绘和合成： 减少页面的重绘和合成操作，可以提高渲染性能。

优化关键渲染路径有助于加速页面加载速度，改善用户体验，并减少页面的加载时间。

35：什么是安全的正则表达式
安全的正则表达式（Safe Regular Expressions）是指在使用正则表达式时，能够避免正则表达式本身引发的安全漏洞或性能问题的一种做法。正则表达式在处理文本匹配和转换时非常强大，但如果不小心编写，可能会导致一些安全问题，例如正则表达式拒绝服务攻击（Regular Expression Denial of Service，简称 ReDoS）。
以下是一些编写安全的正则表达式的注意事项：
1避免复杂的回溯： 复杂的回溯（backtracking）正则表达式可能会导致性能问题和 ReDoS 攻击。确保正则表达式不会在某些输入上产生极端的回溯情况。
2限制量词： 量词（例如 *、+、{n,m} 等）可以导致回溯，尤其是当应用于大量重复字符时。在使用量词时，要确保它们不会导致不必要的回溯。
3避免嵌套重复： 嵌套的重复（例如 (a*)*）也可能导致回溯问题。尽量避免使用这种形式的正则表达式。
4明确的字符集： 在字符集（例如 [a-z]）中尽量避免使用范围的组合，因为它们可能导致回溯。明确列出需要的字符更安全。
5使用限制条件： 对于可能有不同可能性的情况，使用限制条件或非贪婪匹配，以避免匹配过多的文本。
6测试正则表达式： 在真实数据集上测试正则表达式，以确保它们不会导致性能问题或不安全的行为。
7使用正则表达式库的安全选项： 一些正则表达式库提供了安全选项，用于限制正则表达式的使用，从而减少安全风险。
8对用户输入进行验证和转义： 在将用户输入用作正则表达式模式之前，务必对其进行验证和适当的转义，以防止注入攻击。

总之，编写安全的正则表达式需要谨慎和注意，避免不必要的复杂性和回溯，以确保性能和安全性。如果在正则表达式中使用不当，可能会导致应用程序的安全漏洞或性能问题。

39：什么是服务器渲染 (SSR)
服务器渲染（Server-Side Rendering，缩写为 SSR）是一种前端渲染的技术，用于在服务器端生成页面的 HTML 内容，然后将其发送到客户端进行显示。相比于传统的客户端渲染（Client-Side Rendering，缩写为 CSR），在服务器渲染中，页面的初始内容是在服务器上生成的，而不是在浏览器中通过 JavaScript 渲染。
在服务器渲染中，当用户请求访问某个页面时，服务器会执行以下步骤：
1接收请求： 服务器接收来自客户端的请求。
2获取数据： 服务器根据请求获取数据，可能包括从数据库、API 或其他数据源中获取数据。
3生成 HTML： 服务器使用获取到的数据，结合页面模板和组件，生成包含数据的 HTML 页面。
4发送响应： 服务器将生成的 HTML 页面作为响应发送给客户端。
5客户端交互： 浏览器接收到 HTML 后，会显示页面的初始内容。如果页面需要交互，浏览器会下载相应的 JavaScript 代码，并在页面上执行，以实现动态交互和更新。

服务器渲染的优势包括：
1SEO 优化： 由于搜索引擎爬虫可以直接获取渲染后的 HTML 内容，服务器渲染有助于提升搜索引擎优化（SEO）。
2首屏加载速度： 服务器渲染可以提供更快的首次加载速度，因为初始页面内容在服务器上就已经生成，减少了客户端渲染所需的额外请求和处理时间。
3性能和用户体验： 服务器渲染可以减少客户端渲染中 JavaScript 的加载和执行时间，从而改善页面的整体性能和用户体验。

40：Core Web Vitals 是什么，它有哪些指标
Core Web Vitals 是一组由 Google 提出的重要的用户体验指标，用于衡量网页加载性能和交互性。这些指标旨在帮助网站开发者和拥有者评估和改进其网站的用户体验，特别是在移动设备上。
Core Web Vitals 包括以下三个指标：
1Largest Contentful Paint (LCP)： 最大内容渲染时间，衡量页面加载速度。它测量从页面开始加载到最大元素（通常是图片或文本块）完全渲染的时间，以毫秒为单位。较低的 LCP 时间表示页面加载速度更快。
2First Input Delay (FID)： 首次输入延迟，衡量页面的交互性。它测量用户首次与页面交互（例如点击按钮、链接等）的时间延迟，以毫秒为单位。较低的 FID 时间表示页面对用户的交互响应更快。
3Cumulative Layout Shift (CLS)： 累积布局偏移，衡量页面的视觉稳定性。它测量页面上元素在加载过程中发生的突然移动或变化的总量。较低的 CLS 分数表示页面内容更稳定，减少了意外的视觉变化。

Google 计划将 Core Web Vitals 作为搜索排名算法的一部分，这意味着网站的性能和用户体验可能会影响其在搜索结果中的排名。通过优化这些指标，网站开发者可以提升用户体验，减少页面加载时间，增加用户的满意度和留存率。
要评估网站的 Core Web Vitals 指标，可以使用 Google 的工具，如 PageSpeed Insights、Chrome DevTools 和 Web Vitals JavaScript 库。这些工具可以帮助开发者了解网站的性能表现，并提供改进建议。

45：package.json 中 main/module/browser/exports 字段有何区别
在 package.json 文件中，main、module、browser 和 exports 字段都是用于指定模块的入口文件或导出方式的字段，但它们在不同的环境或用途下有不同的作用。
1main 字段：
○main 字段指定了模块的入口文件，通常是 CommonJS 格式的文件路径。这是 Node.js 等环境中默认使用的入口文件。
○当使用 require() 导入模块时，会默认加载 main 字段指定的入口文件。
○示例："main": "dist/main.js"
1module 字段：
○module 字段指定了模块的 ES 模块格式的入口文件路径。这主要用于现代构建工具和浏览器原生支持的 ES 模块加载。
○当使用 import 导入模块时，会默认加载 module 字段指定的入口文件。
○示例："module": "dist/module.js"
1browser 字段：
○browser 字段允许在浏览器环境中重定向模块的导入路径。它可以是一个字符串，也可以是一个对象，用于指定在不同环境下的导入路径。
○用于处理在浏览器中某些模块可能存在的不兼容性问题，或者用于指向已经适配为浏览器环境的模块。
○示例："browser": "dist/browser.js"
1exports 字段：
○exports 字段是 ECMAScript 模块（ESM）的导出映射配置，允许你精确地指定模块的导出方式。
○它可以包含一个对象，其中的键是导出名，值是相对于包的入口文件的路径。这样可以指定不同的导出路径，适配不同的环境。
○示例："exports": { "imported-module": "./src/imported-module.mjs" }
总结：
●main 字段用于 CommonJS 环境的入口文件。
●module 字段用于 ES 模块环境的入口文件。
●browser 字段用于重定向浏览器环境中的导入路径。
●exports 字段用于精确指定模块的导出方式，支持更细粒度的配置。

52：简述 npm cache
npm cache 是 npm 包管理工具内部用来缓存已下载的包和资源的机制。它的存在旨在提高包的获取速度，避免重复下载相同的包，以及减少网络请求，从而加快项目的构建和依赖安装过程。
npm cache 有以下几个主要方面的作用：
1包缓存： 当你使用 npm install 命令安装依赖时，npm 会首先检查本地缓存中是否已经存在这些包。如果存在，就直接从缓存中获取，避免了网络请求和下载时间。这可以提高依赖安装的速度。
2源缓存： npm 缓存还包括了从 registry（包注册中心） 下载的元数据和 tarball 文件（压缩的包文件）。这些元数据包括了包的描述信息、版本号等。当你下载某个包的不同版本时，它们的元数据可以被共享，减少重复的下载。
3自定义缓存： 你可以通过 npm config 命令来配置缓存的路径或使用不同的 registry 来自定义缓存策略，以适应不同的项目需求。

虽然 npm cache 提供了很多好处，但有时也可能导致一些问题，例如：
●过期的缓存： 缓存中可能存有过期或无效的资源，导致不一致性问题。你可以通过 npm cache clean 命令来清除缓存。
●依赖更新： 如果某个包的依赖发生了更新，但缓存中的元数据仍然是旧的版本，可能会导致依赖解析错误。这时可以尝试清除缓存或使用 npm update 命令来更新依赖。

总之，npm cache 是 npm 包管理工具的一个重要组成部分，通过缓存已下载的包和资源，可以提高项目的构建速度和依赖安装过程，同时也需要注意处理一些可能的缓存相关问题。

54：简述 browserslist 的意义
browserslist(opens in a new tab) 用特定的语句来查询浏览器列表，如 last 2 Chrome versions。

细说起来，它是现代前端工程化不可或缺的工具，无论是处理 JS 的 babel，还是处理 CSS 的 postcss，凡是与垫片相关的，他们背后都有 browserslist 的身影。
●babel，在 @babel/preset-env 中使用 core-js 作为垫片
●postcss 使用 autoprefixer 作为垫片
关于前端打包体积与垫片关系，我们有以下几点共识:
1由于低浏览器版本的存在，垫片是必不可少的
2垫片越少，则打包体积越小
3浏览器版本越新，则垫片越少
那在前端工程化实践中，当我们确认了浏览器版本号，那么它的垫片体积就会确认。

56：如何对 npm package 进行发包
要将你的 npm 包发布到 npm 注册表，你可以按照以下步骤进行操作：
1注册 npm 账号： 如果你还没有 npm 账号，首先需要在 npm 注册表上注册一个账号。你可以在命令行中运行 npm adduser 来登录并创建账号。
2创建项目： 在你的项目目录中，确保有一个符合 npm 包要求的项目。项目中需要包含一个有效的 package.json 文件，其中包含了必要的元数据、依赖项和其他配置信息。
3版本控制： 确保你的项目代码使用版本控制系统（如 Git）进行管理。这将有助于追踪代码的变更历史，并在发布时指定正确的版本号。
4更新版本号： 在每次准备发布新版本时，你需要更新 package.json 文件中的版本号。你可以使用以下命令来更新版本号：
○npm version patch：递增修订版本号（例如，从 1.0.0 到 1.0.1）。
○npm version minor：递增次版本号（例如，从 1.0.0 到 1.1.0）。
○npm version major：递增主版本号（例如，从 1.0.0 到 2.0.0）。
1构建和测试： 在发布之前，确保项目可以正常构建并通过测试。运行适当的构建命令和测试套件，确保你的包在发布后的使用中没有明显的问题。
2登录 npm： 在命令行中运行 npm login，使用你在第一步注册的账号进行登录。这将将你的登录凭证存储在本地，以便后续的发布操作。
3发布包： 运行 npm publish 命令来将你的包发布到 npm 注册表。这将会将你的包上传到 npm，并使其可供他人安装和使用。
4验证发布： 完成发布后，可以在 npm 注册表上查看你的包是否已成功发布。可以访问 https://www.npmjs.com/package/你的包名 来查看你的包页面。

58：请问什么是 CICD
●CI，Continuous Integration，持续集成。
●CD，Continuous Deployment，持续部署。
CICD 一般合称，无需特意区分二者区别。从开发、测试到上线的过程中，借助于 CICD 进行一些自动化处理，保障项目质量。
CICD 与 git 集成在一起，可理解为服务器端的 git hooks: 当代码 push 到远程仓库后，借助 WebHooks 对当前代码在构建服务器(即 CI 服务器，也称作 Runner)中进行自动构建、测试及部署等。


它有若干好处:
1功能分支提交后，通过 CICD 进行自动化测试、语法检查等，如未通过 CICD，则无法 CodeReview，更无法合并到生产环境分支进行上线
2功能分支提交后，通过 CICD 检查 npm 库的风险、检查构建镜像容器的风险等
3功能分支提交后，通过 CICD 对当前分支代码构建独立镜像并生成独立的分支环境地址进行测试，如对每一个功能分支生成一个可供测试的地址，一般是 <branch>.dev.shanyue.tech 此种地址
4功能分支测试通过后，合并到主分支，自动构建镜像并部署到生成环境 (一般生成环境需要手动触发、自动部署)

59：在项目中，如何平滑升级 npm 包
npm 的版本号为 semver 规范，由 [major, minor, patch] 三部分组成，其中
●major: 当你发了一个含有 Breaking Change 的 API
●minor: 当你新增了一个向后兼容的功能时
●patch: 当你修复了一个向后兼容的 Bug 时
假设 react 当前版本号为 17.0.1，我们要升级到 17.0.2 应该如何操作？

自动发现更新
升级版本号，最不建议的事情就是手动在 package.json 中进行修改。

毕竟，你无法手动发现所有需要更新的 package。
此时可借助于 npm outdated，发现有待更新的 package。


使用 npm outdated，还可以列出其待更新 package 的文档。

自动更新版本号
使用 npm outdated 虽能发现需要升级版本号的 package，但仍然需要手动在 package.json 更改版本号进行升级。
此时推荐一个功能更强大的工具 npm-check-updates，比 npm outdated 强大百倍。
npm-check-updates -u，可自动将 package.json 中待更新版本号进行重写。

升级 [minor] 小版本号，有可能引起 Break Change，可仅仅升级到最新的 patch 版本。
一点小建议
1当一个库的 major 版本号更新后，不要第一时间去更新，容易踩坑，可再度过几个 patch 版本号再更新尝试新功能
2当遇到 major 版本号更新时，多看文档中的 ChangeLog，多看升级指导并多测试及审计
